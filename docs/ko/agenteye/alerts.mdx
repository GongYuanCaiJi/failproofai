고객으로부터 듣기 전에, 팀이 이미 사용하는 채널로 기준을 넘는 순간 바로 알림을 받으세요. 규칙을 한 번 설정해 두면 Failproof AI Observability가 주기적으로 확인한 뒤 이메일, Slack, 웹훅, 또는 대시보드를 통해 알려줍니다.

![알림 페이지: 각 알림 규칙 카드에 트리거, 평가 윈도우, 채널, info/warning/critical 심각도 배지가 표시된 그리드](/agenteye/images/alerts.png)
*모든 알림 규칙을 한눈에: 무엇을 감시하는지, 얼마나 자주, 어디에 알리는지, 얼마나 긴급한지.*

## 사용자보다 먼저 문제를 파악하세요

회귀가 발생했는지 확인하려고 대시보드를 새로고침하는 일은 그만하세요. 아무도 보고 있지 않을 때도 알고 싶은 신호가 있다면 알림을 설정하고, 이미 사용 중인 곳으로 전달받으세요:

- **이메일**: 알아야 할 담당자에게.
- **Slack**: 인시던트로 바로 이동하는 버튼이 포함된 리치 메시지.
- **웹훅**: PagerDuty, Opsgenie, 또는 자체 엔드포인트를 위한 JSON POST. 수신 측이 신뢰할 수 있도록 선택적 서명 포함.
- **대시보드 내**: 규칙을 조정 중일 때 불필요한 알림 없이 조용히 확인하기 위한 옵션.

하나의 규칙에 여러 채널을 조합할 수 있으며, 심각도(info, warning, critical)도 함께 전달되어 긴급한 알림은 긴급하게 보입니다.

## 규칙은 폼으로 작성하세요, JSON이 아니라

"고장났다"는 의미를 폼으로 설명하면 Failproof AI Observability가 내부 규칙을 대신 작성해 줍니다. JSON 스펙은 해당 폼이 내부적으로 생성하는 결과물일 뿐이므로, 규칙을 이해하기 위해 읽을 수는 있지만 직접 입력할 일은 거의 없습니다.

![새 알림 폼: 이름과 설명, 활성화 토글, 그리고 메트릭 임계값, 커스텀 SQL, 평가 점수, 복합 평가, 이벤트별 조건을 제공하는 트리거 선택기](/agenteye/images/alert-new.png)
*트리거를 선택하면 폼이 해당 필드로 전환됩니다. 저장하면 규칙이 작성됩니다.*

기본 흐름은 간단합니다: 이름을 입력하고, **트리거**(무엇을 감시할지)를 선택하고, **임계값과 윈도우**(얼마나 심각한지, 어느 기간 동안)를 설정하고, 하나 이상의 **채널**을 연결한 후 **저장**하고 **테스트**를 눌러 가상 알림을 발송해 모든 대상이 제대로 연결됐는지 확인합니다. 내부적으로는 다음과 같은 간단한 스펙이 생성됩니다:

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

한 가지 신호 유형에 제한되지 않습니다. 장애를 어떻게 인식하는지에 맞는 트리거를 선택하세요:

| 트리거 | 발동 조건 |
|---|---|
| **메트릭 임계값** | 사전 정의된 메트릭(에러율, p95/p99 지연 시간, 이벤트 또는 에러 횟수, 토큰 사용량)이 지정한 기간 동안 기준을 초과할 때 |
| **커스텀 SQL** | 직접 작성한 읽기 전용 쿼리가 행을 반환하거나, 쿼리가 계산한 값이 임계값을 넘을 때 |
| **평가 점수** | 특정 평가자 점수의 평균(예: 환각 점수)이 임계값을 넘을 때 |
| **복합 평가** | 여러 점수 조건을 any, all, at-least-N 논리로 결합해, 여러 점수에 걸쳐서만 나타나는 회귀를 감지할 때 |
| **이벤트별** | 특정 에이전트, 특정 에러 유형, 또는 메시지 부분 문자열과 일치하는 단일 이벤트가 발생할 때 |

[에러 페이지](/ko/agenteye/error-tracking)에서 이미 장애를 보고 있다면? 각 행에 **+ alert** 버튼이 있어 해당 장애를 정확히 포착하도록 미리 채워진 폼이 열립니다. 방금 트리아지한 인시던트가 다음번에 알림을 보내는 규칙이 됩니다.

**위치:** 알림은 `/<org-slug>/alerts`에 있습니다. 규칙 생성, 편집, 삭제, 테스트에는 **`alerts:write`** 권한이 필요하며, 조회에는 `alerts:read`로 충분합니다. 수신자 선택기에는 조직 멤버가 이름으로 표시되므로, 폼을 벗어나지 않고 특정 담당자에게 알림을 보낼 수 있습니다.

## 실제 문제일 때만 알림 받기

측정값 하나가 잘못됐다고 잠에서 깨어나선 안 됩니다. **M of N** 노이즈 필터는 최근 몇 번의 확인 중 몇 번이 실패해야 실제로 알림이 발송될지 제어합니다. **5번 중 3번**으로 설정하면 최근 5번의 확인 중 3번 이상 기준을 초과한 경우에만 규칙이 발동되므로, 불안정한 신호가 계속 잘못된 경고를 보내는 것을 방지합니다. 기본값인 **1 of 1**로 두면 첫 번째 초과 시 바로 발동됩니다. 신호가 얼마나 빠르게 변동하는지에 맞게 1m, 5m, 15m, 1h 프리셋 중 하나를 선택해 규칙 실행 주기를 설정할 수도 있습니다.

## 알림이 발동되면 무슨 일이 일어나나요

기준 초과 시 **인시던트**가 생성되고 채널에 한 번 알림이 전송됩니다. 이후 팀이 확인하고, 담당자를 지정하고, 논의하며 해결합니다. 이 모든 과정은 명확하고 책임이 명시된 기록으로 남습니다. 이 트리아지 워크플로우는 별도의 공간에 있습니다: [인시던트](/ko/agenteye/incidents)를 참고하세요.

## 관련 항목

- [인시던트](/ko/agenteye/incidents): 발동된 알림을 열림에서 확인됨, 해결됨까지 추적합니다.
- [에러 추적](/ko/agenteye/error-tracking): 에이전트 장애를 그룹화하고 클릭 한 번으로 알림으로 전환합니다.
- [대시보드](/ko/agenteye/dashboards): 알림 임계값의 기반이 되는 공유 보드를 확인합니다.
- [CLI 및 에이전트](/ko/agenteye/cli-and-agents): 터미널에서 알림을 생성하고 인시던트를 확인하거나, CI에 스크립트로 통합합니다.
