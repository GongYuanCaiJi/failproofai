문제가 발생하는 순간, 팀이 이미 사용 중인 채널에서 직접 알림을 받아보세요. 고객으로부터 먼저 듣는 일이 없도록 하세요. 규칙을 한 번 설정하면 FailproofAI Observability가 주기적으로 확인하고, 이메일, Slack, 웹훅, 또는 대시보드를 통해 알림을 전송합니다.

![알림 페이지: 각 알림 규칙 카드에 트리거, 평가 기간, 채널, 정보·경고·심각 수준 배지가 표시된 그리드](/agenteye/images/alerts.png)
*모든 알림 규칙을 한눈에: 감시 대상, 평가 주기, 알림 채널, 긴급도.*

## 사용자보다 먼저 문제를 파악하세요

회귀 오류를 잡으려고 대시보드를 계속 새로고침하지 마세요. 아무도 보지 않을 때도 알아야 할 신호가 있다면 알림을 설정하고, 이미 사용 중인 곳으로 전달받으세요:

- **이메일**, 알아야 할 담당자에게 직접.
- **Slack**, 인시던트로 바로 이동하는 버튼이 포함된 풍부한 메시지.
- **웹훅**, PagerDuty, Opsgenie 또는 직접 구축한 엔드포인트로 전송되는 JSON POST. 수신 측 검증을 위한 서명 옵션도 지원합니다.
- **대시보드 내**, 규칙을 조정하면서 아직 아무에게도 알리고 싶지 않을 때 활용할 수 있는 조용한 알림.

하나의 규칙에 여러 채널을 조합해서 연결할 수 있으며, 심각도(정보, 경고, 심각)도 함께 전달되어 긴급한 알림은 긴급하게 보입니다.

## JSON이 아닌 폼으로 규칙을 만드세요

무엇이 "문제"인지를 폼에 입력하면 FailproofAI Observability가 내부적으로 규칙을 작성합니다. JSON 명세는 폼이 내부적으로 생성하는 결과물이므로, 규칙을 이해하기 위해 읽을 수는 있지만 직접 입력할 일은 거의 없습니다.

![새 알림 폼: 이름과 설명, 활성화 토글, 그리고 메트릭 임계값, 사용자 정의 SQL, 평가 점수, 복합 평가, 이벤트별 조건을 제공하는 트리거 선택기](/agenteye/images/alert-new.png)
*트리거를 선택하면 폼이 해당 필드로 전환됩니다. 저장하면 규칙이 생성됩니다.*

기본 흐름은 간단합니다: 이름을 지정하고, **트리거**(감시 대상)를 선택한 다음, **임계값과 기간**(얼마나 나쁜 상태가 얼마나 오래 지속될 때)을 설정하고, 최소 하나의 **채널**을 연결한 뒤 **저장**합니다. 이후 **테스트**를 눌러 테스트 알림을 발송하고 모든 채널이 정상적으로 연결되었는지 확인하세요. 내부적으로는 다음과 같은 소규모 명세가 생성됩니다:

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

하나의 신호 유형에만 국한되지 않습니다. 장애를 어떻게 인식하느냐에 맞는 트리거를 선택하세요:

| 트리거 | 발동 조건 |
|---|---|
| **메트릭 임계값** | 오류율, p95 또는 p99 지연 시간, 이벤트 또는 오류 횟수, 토큰 사용량 등 사전 정의된 메트릭이 일정 기간 동안 설정한 기준을 초과할 때 |
| **사용자 정의 SQL** | 직접 작성한 읽기 전용 쿼리가 행을 반환하거나, 계산된 값이 임계값을 초과할 때 |
| **평가 점수** | 환각 탐지 등 평가 점수의 평균이 임계값을 초과할 때 |
| **복합 평가** | 여러 점수 조건을 any, all, 또는 N개 이상 로직으로 조합하여, 여러 점수에 걸쳐 나타나는 회귀를 감지할 때 |
| **이벤트별** | 특정 에이전트, 특정 오류 유형, 또는 메시지 하위 문자열과 일치하는 단일 이벤트가 발생할 때 |

[오류 페이지](/ko/agenteye/error-tracking)에서 이미 장애를 확인하고 있다면, 각 행에 있는 **+ 알림** 버튼으로 동일한 폼을 열 수 있습니다. 해당 장애를 다시 감지하도록 미리 채워져 있어, 방금 처리한 인시던트가 다음번에 바로 알림으로 연결됩니다.

**위치:** 알림은 `/<org-slug>/alerts`에 있습니다. 규칙 생성, 편집, 삭제, 테스트에는 **`alerts:write`** 권한이 필요하며, 조회에는 `alerts:read`로 충분합니다. 수신자 선택기에는 조직 구성원 이름이 표시되므로, 폼을 벗어나지 않고도 특정 담당자에게 알림을 보낼 수 있습니다.

## 실제 문제일 때만 알림을 받으세요

잘못된 측정값 하나로 잠을 깨워서는 안 됩니다. **M of N** 노이즈 필터는 최근 몇 번의 확인 중 몇 번이 실패해야 실제 알림을 발송할지 제어합니다. **5번 중 3번**으로 설정하면 최근 다섯 번의 확인 중 세 번이 기준을 초과한 경우에만 규칙이 발동하므로, 불안정한 신호가 지속적으로 오탐을 유발하는 것을 방지합니다. 기본값인 **1 of 1**을 유지하면 첫 번째 기준 초과 시 바로 발동합니다. 신호가 실제로 변동하는 속도에 맞게 1m, 5m, 15m, 1h 프리셋 중 하나를 선택하여 규칙 실행 주기를 설정할 수도 있습니다.

## 알림이 발동되면 어떻게 되나요

기준 초과가 발생하면 **인시던트**가 열리고 채널에 한 번 알림이 전송됩니다. 이후 팀이 인시던트를 확인하고, 담당자를 지정하고, 논의하고, 해결하는 과정이 명확하게 기록됩니다. 이 트리아지 워크플로에는 별도의 공간이 있습니다: [인시던트](/ko/agenteye/incidents)를 참조하세요.

## 관련 문서

- [인시던트](/ko/agenteye/incidents): 발동된 알림을 오픈부터 확인, 해결까지 추적합니다.
- [오류 추적](/ko/agenteye/error-tracking): 에이전트 장애를 그룹화하고 클릭 한 번으로 알림으로 전환합니다.
- [대시보드](/ko/agenteye/dashboards): 알림 임계값의 기준이 되는 공유 보드를 확인합니다.
- [CLI 및 에이전트](/ko/agenteye/cli-and-agents): 터미널에서 알림을 생성하고 인시던트를 확인하거나, CI에 스크립트로 통합합니다.
