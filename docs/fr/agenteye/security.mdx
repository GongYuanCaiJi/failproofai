Failproof AI Observability est conçu pour fonctionner au plus près de vos agents en production, ce qui signifie qu'il voit vos prompts, les entrées des outils et les sorties. Cette page explique comment vos données sont isolées, contrôlées et restent entre vos mains. Si vous évaluez Failproof AI Observability dans le cadre d'un audit de sécurité, commencez ici.

---

## Vos données restent dans votre environnement

Failproof AI Observability est auto-hébergé. Les événements, prompts, réponses des modèles et données analytiques sont stockés dans vos propres bases de données, dans votre propre environnement. Rien n'est envoyé vers un SaaS tiers pour y être stocké, et vos données demeurent dans votre propre compte cloud.

---

## Isolation des tenants

Une instance de Failproof AI Observability peut héberger plusieurs organisations, chacune étant isolée au niveau de la couche de stockage — appliqué par la base de données elle-même, et pas seulement par l'interface :

- Les données opérationnelles d'une organisation (utilisateurs, clés, tableaux de bord, requêtes sauvegardées) sont limitées à cette organisation, et les lectures inter-organisations sont bloquées par la base de données elle-même.
- Chaque événement ingéré est marqué avec l'organisation propriétaire, de sorte que les événements d'une organisation ne peuvent jamais être lus par une autre.

Chaque route de tableau de bord est délimitée par un slug d'organisation (`/<org-slug>/…`).

---

## Connexion

Failproof AI Observability utilise une connexion sans mot de passe, basée sur l'e-mail. Il n'y a pas de mot de passe à hameçonner ou à divulguer. Un utilisateur demande un code à usage unique (ou un lien magique en un clic), qui lui est envoyé par e-mail et expire rapidement. La connexion est conditionnée par une **liste d'autorisation** : seules les adresses e-mail (ou domaines) que vous autorisez peuvent s'authentifier.

## Accès limité avec des clés API

Chaque client s'authentifie avec une clé API dotée de permissions granulaires selon le principe du moindre privilège. Un collecteur n'a besoin que de `events:add` ; une clé de tableau de bord ou d'assistant peut être en lecture seule ; les actions destructives (suppression, régénération) sont des autorisations distinctes que vous choisissez d'inclure.

![La page des clés API : les autorisations de chaque clé, avec un code couleur selon la portée lecture, écriture et destructive](/agenteye/images/api-keys.png)

Conservez la clé d'amorçage administrateur pour la configuration, et émettez des clés restreintes pour tout le reste. Voir [Clés API](/fr/agenteye/api-keys).

## Un assistant en lecture seule soumis à approbation

L'[assistant IA](/fr/agenteye/assistant) intégré au tableau de bord répond à des questions sur vos données, mais il est limité par conception :

- Il est **en lecture seule par défaut** : son SQL passe par un garde-fou qui n'autorise que les requêtes `SELECT`/`WITH`, à instruction unique, avec une limite de lignes.
- Tout ce qu'il crée (une requête sauvegardée, un tableau de bord) est **soumis à approbation** : vous vérifiez et approuvez chaque écriture avant qu'elle n'ait lieu.
- Il **ne peut jamais supprimer**.

Ainsi, un membre de l'équipe peut demander « quels agents ont le plus planté cette semaine ? » et agir en conséquence, sans que l'assistant puisse modifier ou supprimer vos données par lui-même.

---

## En transit

Tout le trafic transite via HTTPS. Vous terminez le TLS avec vos propres certificats, de sorte que le trafic entre le collecteur et le serveur, ainsi qu'entre le navigateur et le serveur, est chiffré en transit.
