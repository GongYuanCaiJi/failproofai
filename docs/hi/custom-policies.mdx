कस्टम नीतियाँ आपको किसी भी एजेंट व्यवहार के लिए नियम लिखने देती हैं: प्रोजेक्ट परंपराओं को लागू करें, drift को रोकें, विनाशकारी ऑपरेशन को गेट करें, stuck एजेंट का पता लगाएं, या Slack, approval वर्कफ़्लो, और अधिक के साथ एकीकृत करें। वे built-in नीतियों के समान ही hook event सिस्टम और `allow`, `deny`, `instruct` निर्णयों का उपयोग करते हैं।

---

## त्वरित उदाहरण

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

इसे install करें:

```bash
failproofai policies --install --custom ./my-policies.js
```

## कस्टम नीतियों को लोड करने के दो तरीके

### विकल्प 1: परंपरा-आधारित (अनुशंसित)

`.failproofai/policies/` में `*policies.{js,mjs,ts}` फ़ाइलें डालें और वे स्वचालित रूप से लोड हो जाती हैं — कोई flags या config परिवर्तन की आवश्यकता नहीं। यह git hooks की तरह काम करता है: एक फ़ाइल डालें, यह बस काम करता है।

```
# प्रोजेक्ट स्तर — git में commit किया गया, टीम के साथ साझा किया गया
.failproofai/policies/security-policies.mjs
.failproofai/policies/workflow-policies.mjs

# यूजर स्तर — व्यक्तिगत, सभी प्रोजेक्ट्स पर लागू होता है
~/.failproofai/policies/my-policies.mjs
```

**यह कैसे काम करता है:**
- प्रोजेक्ट और यूजर दोनों निर्देशिकाओं को स्कैन किया जाता है (union — पहली-scope-wins नहीं)
- फ़ाइलें प्रत्येक निर्देशिका में वर्णक्रम के अनुसार लोड की जाती हैं। क्रम को नियंत्रित करने के लिए `01-`, `02-` से prefix करें
- केवल `*policies.{js,mjs,ts}` से मेल खाने वाली फ़ाइलें लोड की जाती हैं; अन्य फ़ाइलें ignored की जाती हैं
- प्रत्येक फ़ाइल independently लोड की जाती है (fail-open per file)
- explicit `--custom` और built-in नीतियों के साथ काम करता है

<Tip>
परंपरा नीतियाँ आपके org के लिए एक quality standard बनाने का सबसे आसान तरीका हैं। `.failproofai/policies/` को git में commit करें और हर टीम सदस्य को स्वचालित रूप से एक जैसे नियम मिलते हैं — कोई per-developer setup की आवश्यकता नहीं। जैसे-जैसे आपकी टीम नई failure modes खोजती है, एक नीति जोड़ें और push करें। समय के साथ ये एक living quality standard बन जाती हैं जो हर contribution के साथ सुधरती रहती हैं।
</Tip>

### विकल्प 2: स्पष्ट फ़ाइल पथ

```bash
# कस्टम नीतियों की फ़ाइल के साथ install करें
failproofai policies --install --custom ./my-policies.js

# नीतियों की फ़ाइल पथ को replace करें
failproofai policies --install --custom ./new-policies.js

# config से कस्टम नीतियों का पथ हटाएं
failproofai policies --uninstall --custom
```

Resolved absolute पथ `policies-config.json` में `customPoliciesPath` के रूप में stored है। फ़ाइल को हर hook event पर fresh लोड किया जाता है - events के बीच कोई caching नहीं है।

### दोनों को एक साथ उपयोग करना

परंपरा नीतियाँ और explicit `--custom` फ़ाइल एक साथ मौजूद हो सकते हैं। लोड क्रम:

1. Explicit `customPoliciesPath` फ़ाइल (यदि configured हो)
2. प्रोजेक्ट परंपरा फ़ाइलें (`{cwd}/.failproofai/policies/`, वर्णानुक्रमीय)
3. यूजर परंपरा फ़ाइलें (`~/.failproofai/policies/`, वर्णानुक्रमीय)

## API

### Import

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

एक नीति को register करता है। एक ही फ़ाइल में multiple नीतियों के लिए जितनी बार आवश्यकता हो इसे call करें।

```ts
customPolicies.add({
  name: string;                         // आवश्यक - unique identifier
  description?: string;                 // `failproofai policies` output में दिखाया गया
  match?: { events?: HookEventType[] }; // event type के आधार पर filter करें; सभी से मेल खाने के लिए omit करें
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### निर्णय सहायक

| फ़ंक्शन | प्रभाव | यह उपयोग करें |
|----------|--------|----------|
| `allow()` | ऑपरेशन को चुप रहकर permit करें | कार्य सुरक्षित है, कोई संदेश की आवश्यकता नहीं है |
| `deny(message)` | ऑपरेशन को block करें | एजेंट को यह कार्य नहीं करना चाहिए |
| `instruct(message)` | बिना blocking के context जोड़ें | एजेंट को रास्ते पर रहने में मदद करने के लिए अतिरिक्त context दें |

`deny(message)` - संदेश Claude को `"Blocked by failproofai:"` prefix के साथ दिखाई देता है। एक ही `deny` सभी आगे के evaluation को short-circuit करता है।

`instruct(message)` - संदेश Claude के context में current tool call के लिए जोड़ा जाता है। सभी `instruct` संदेश accumulated किए जाते हैं और एक साथ deliver किए जाते हैं।

<Tip>
आप किसी भी `deny` या `instruct` संदेश में `policyParams` में `hint` field जोड़कर अतिरिक्त guidance जोड़ सकते हैं — कोई code परिवर्तन की आवश्यकता नहीं। यह custom (`custom/`), प्रोजेक्ट परंपरा (`.failproofai-project/`), और यूजर परंपरा (`.failproofai-user/`) नीतियों के लिए भी काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

### Informational allow संदेश

`allow(message)` ऑपरेशन को permit करता है **और** Claude को एक informational संदेश भेजता है। संदेश को hook handler के stdout response में `additionalContext` के रूप में deliver किया जाता है — `instruct` द्वारा उपयोग किया जाने वाला ही mechanism, लेकिन semantically अलग: यह एक status update है, warning नहीं।

| फ़ंक्शन | प्रभाव | यह उपयोग करें |
|----------|--------|----------|
| `allow(message)` | Permit करें और Claude को context भेजें | एक check passed confirm करें, या explain करें कि check क्यों skip किया गया |

उपयोग के मामले:
- **Status confirmations:** `allow("All CI checks passed.")` — Claude को बताएं सब कुछ green है
- **Fail-open explanations:** `allow("GitHub CLI not installed, skipping CI check.")` — Claude को बताएं कि check क्यों skip किया गया ताकि इसे पूरा context मिले
- **Multiple संदेश accumulate:** यदि कई नीतियाँ प्रत्येक `allow(message)` return करती हैं, सभी संदेश newlines से जुड़े हैं और एक साथ deliver किए जाते हैं

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### `PolicyContext` fields

| Field | Type | Description |
|-------|------|-------------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | जो tool call किया जा रहा है (जैसे `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Tool के input parameters |
| `payload` | `Record<string, unknown>` | Claude Code से पूरी raw event payload |
| `session` | `SessionMetadata \| undefined` | Session context (नीचे देखें) |

### `SessionMetadata` fields

| Field | Type | Description |
|-------|------|-------------|
| `sessionId` | `string` | Claude Code session identifier |
| `cwd` | `string` | Claude Code session की working directory |
| `transcriptPath` | `string` | Session की JSONL transcript फ़ाइल का पथ |

### Event types

| Event | यह कब fire होता है | `toolInput` contents |
|-------|--------------|----------------------|
| `PreToolUse` | Claude एक tool चलाने से पहले | Tool का input (जैसे Bash के लिए `{ command: "..." }`) |
| `PostToolUse` | एक tool पूरा होने के बाद | Tool का input + `tool_result` (output) |
| `Notification` | जब Claude एक notification भेजता है | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - hooks को हमेशा `allow()` return करना चाहिए, वे notifications को block नहीं कर सकते |
| `Stop` | जब Claude session समाप्त होता है | Empty |

## Evaluation क्रम

नीतियों का मूल्यांकन इस क्रम में किया जाता है:

1. Built-in नीतियाँ (definition क्रम में)
2. `customPoliciesPath` से explicit custom नीतियाँ (`.add()` क्रम में)
3. प्रोजेक्ट `.failproofai/policies/` से परंपरा नीतियाँ (फ़ाइलें वर्णानुक्रमीय, `.add()` क्रम में)
4. यूजर `~/.failproofai/policies/` से परंपरा नीतियाँ (फ़ाइलें वर्णानुक्रमीय, `.add()` क्रम में)

<Note>
पहली `deny` सभी subsequent नीतियों को short-circuit करती है। सभी `instruct` संदेश accumulated किए जाते हैं और एक साथ deliver किए जाते हैं।
</Note>

## Transitive imports

कस्टम policy फ़ाइलें relative paths का उपयोग करके local modules को import कर सकती हैं:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Entry फ़ाइल से पहुंचने योग्य सभी relative imports resolve किए जाते हैं। यह `from "failproofai"` imports को actual dist path पर rewrite करके और ESM compatibility सुनिश्चित करने के लिए temporary `.mjs` फ़ाइलें बनाकर लागू किया जाता है।

## Event type filtering

`match.events` का उपयोग करके limit करें कि कोई नीति कब fire होती है:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // केवल तब fire होता है जब session समाप्त होता है
    // ctx.session.transcriptPath में पूरा session log है
    return allow();
  },
});
```

हर event type पर fire करने के लिए `match` को पूरी तरह omit करें।

## Error handling और failure modes

कस्टम नीतियाँ **fail-open** हैं: errors कभी built-in नीतियों को block नहीं करते या hook handler को crash नहीं करते।

| विफलता | व्यवहार |
|---------|----------|
| `customPoliciesPath` set नहीं है | कोई explicit कस्टम नीतियाँ नहीं चलती; परंपरा नीतियाँ और built-ins सामान्य रूप से जारी रहती हैं |
| फ़ाइल नहीं मिली | `~/.failproofai/hook.log` में warning logged; built-ins जारी रहते हैं |
| Syntax/import error (explicit) | `~/.failproofai/hook.log` में error logged; explicit कस्टम नीतियाँ skipped |
| Syntax/import error (परंपरा) | Error logged; वह फ़ाइल skipped, अन्य परंपरा फ़ाइलें अभी भी load होती हैं |
| `fn` runtime पर throw करता है | Error logged; वह hook `allow` के रूप में treated; अन्य hooks जारी रहते हैं |
| `fn` 10s से अधिक समय लेता है | Timeout logged; `allow` के रूप में treated |
| परंपरा निर्देशिका missing है | कोई परंपरा नीतियाँ नहीं चलती; कोई error नहीं |

<Tip>
कस्टम policy errors को debug करने के लिए, log फ़ाइल देखें:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## पूरा उदाहरण: multiple नीतियाँ

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// एजेंट को secrets/ निर्देशिका में लिखने से रोकें
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// एजेंट को रास्ते पर रखें: commit करने से पहले tests verify करें
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Freeze अवधि के दौरान unplanned dependency परिवर्तन रोकें
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```

## उदाहरण

`examples/` निर्देशिका में ready-to-run policy फ़ाइलें हैं:

| फ़ाइल | Contents |
|------|----------|
| `examples/policies-basic.js` | पाँच starter नीतियाँ जो common agent failure modes को cover करती हैं |
| `examples/policies-advanced/index.js` | Advanced patterns: transitive imports, async calls, output scrubbing, और session-end hooks |
| `examples/convention-policies/security-policies.mjs` | परंपरा-आधारित security नीतियाँ (.env writes को block करें, git history rewriting को रोकें) |
| `examples/convention-policies/workflow-policies.mjs` | परंपरा-आधारित workflow नीतियाँ (test reminders, audit file writes) |

### Explicit फ़ाइल उदाहरणों का उपयोग करना

```bash
failproofai policies --install --custom ./examples/policies-basic.js
```

### परंपरा-आधारित उदाहरणों का उपयोग करना

```bash
# प्रोजेक्ट स्तर पर copy करें
mkdir -p .failproofai/policies
cp examples/convention-policies/*.mjs .failproofai/policies/

# या यूजर स्तर पर copy करें
mkdir -p ~/.failproofai/policies
cp examples/convention-policies/*.mjs ~/.failproofai/policies/
```

कोई install command की आवश्यकता नहीं — फ़ाइलें अगली hook event पर स्वचालित रूप से pick up की जाती हैं।
