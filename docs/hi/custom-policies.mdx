कस्टम नीतियां आपको किसी भी एजेंट व्यवहार के लिए नियम लिखने देती हैं: प्रोजेक्ट सम्मेलनों को लागू करें, drift को रोकें, विनाशकारी ऑपरेशनों को gate करें, stuck एजेंटों का पता लगाएं, या Slack, approval workflows, और बहुत कुछ के साथ एकीकृत करें। वे built-in नीतियों के समान hook event system और `allow`, `deny`, `instruct` निर्णयों का उपयोग करते हैं।

---

## त्वरित उदाहरण

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

इसे इंस्टॉल करें:

```bash
failproofai policies --install --custom ./my-policies.js
```

## कस्टम नीतियों को लोड करने के दो तरीके

### विकल्प 1: सम्मेलन-आधारित (अनुशंसित)

`.failproofai/policies/` में `*policies.{js,mjs,ts}` फाइलें ड्रॉप करें और वे स्वचालित रूप से लोड हो जाती हैं — कोई फ्लैग या कॉन्फ़िग परिवर्तन की आवश्यकता नहीं है। यह git hooks की तरह काम करता है: एक फाइल ड्रॉप करें, यह सिर्फ काम करता है।

```
# प्रोजेक्ट स्तर — git में committed, टीम के साथ साझा
.failproofai/policies/security-policies.mjs
.failproofai/policies/workflow-policies.mjs

# उपयोगकर्ता स्तर — व्यक्तिगत, सभी प्रोजेक्ट पर लागू होता है
~/.failproofai/policies/my-policies.mjs
```

**यह कैसे काम करता है:**
- प्रोजेक्ट और उपयोगकर्ता दोनों निर्देशिकाओं को स्कैन किया जाता है (union — first-scope-wins नहीं)
- फाइलें प्रत्येक निर्देशिका के भीतर वर्णक्रमानुसार लोड होती हैं। क्रम को नियंत्रित करने के लिए `01-`, `02-` से प्रीफिक्स करें
- केवल `*policies.{js,mjs,ts}` से मेल खाने वाली फाइलें लोड होती हैं; अन्य फाइलें अनदेखी की जाती हैं
- प्रत्येक फाइल स्वतंत्र रूप से लोड की जाती है (प्रति फाइल fail-open)
- explicit `--custom` और built-in नीतियों के साथ काम करता है

<Tip>
सम्मेलन नीतियां आपके संगठन के लिए एक गुणवत्ता मानदंड बनाने का सबसे आसान तरीका हैं। `.failproofai/policies/` को git में commit करें और हर टीम सदस्य को स्वचालित रूप से वही नियम मिल जाते हैं — कोई per-developer सेटअप आवश्यक नहीं है। जैसे-जैसे आपकी टीम नई विफलता के तरीके खोजती है, एक नीति जोड़ें और push करें। समय के साथ ये एक जीवंत गुणवत्ता मानदंड बन जाते हैं जो हर योगदान के साथ सुधरते रहते हैं।
</Tip>

### विकल्प 2: Explicit फाइल पथ

```bash
# कस्टम नीतियों की फाइल के साथ इंस्टॉल करें
failproofai policies --install --custom ./my-policies.js

# नीतियों की फाइल पथ को बदलें
failproofai policies --install --custom ./new-policies.js

# कॉन्फ़िग से कस्टम नीतियों के पथ को हटाएं
failproofai policies --uninstall --custom
```

resolved absolute पथ `policies-config.json` में `customPoliciesPath` के रूप में संग्रहीत होता है। फाइल हर hook event पर fresh लोड होती है - events के बीच कोई caching नहीं है।

### दोनों को एक साथ उपयोग करना

सम्मेलन नीतियां और explicit `--custom` फाइल एक साथ मौजूद हो सकते हैं। लोड क्रम:

1. Explicit `customPoliciesPath` फाइल (यदि configured है)
2. प्रोजेक्ट सम्मेलन फाइलें (`{cwd}/.failproofai/policies/`, वर्णक्रमानुसार)
3. उपयोगकर्ता सम्मेलन फाइलें (`~/.failproofai/policies/`, वर्णक्रमानुसार)

## API

### Import

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

एक नीति को पंजीकृत करता है। एक ही फाइल में कई नीतियों के लिए इसे जितनी बार चाहें उतनी बार कॉल करें।

```ts
customPolicies.add({
  name: string;                         // required - unique identifier
  description?: string;                 // shown in `failproofai policies` output
  match?: { events?: HookEventType[] }; // filter by event type; omit to match all
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### निर्णय सहायक

| Function | Effect | उपयोग करें जब |
|----------|--------|----------|
| `allow()` | ऑपरेशन को चुपचाप अनुमति दें | कार्रवाई सुरक्षित है, कोई संदेश आवश्यक नहीं |
| `deny(message)` | ऑपरेशन को block करें | एजेंट को यह कार्रवाई नहीं करनी चाहिए |
| `instruct(message)` | blocking के बिना संदर्भ जोड़ें | एजेंट को track पर रहने के लिए extra संदर्भ दें |

`deny(message)` - संदेश Claude को `"Blocked by failproofai:"` से prefixed दिखाई देता है। एक single `deny` सभी आगे के मूल्यांकन को short-circuit करता है।

`instruct(message)` - संदेश Claude के current tool call के लिए संदर्भ में appended होता है। सभी `instruct` संदेश जमा होते हैं और एक साथ दिए जाते हैं।

<Tip>
आप किसी भी `deny` या `instruct` संदेश में `policyParams` में `hint` फाइल जोड़कर extra guidance append कर सकते हैं — कोड परिवर्तन की आवश्यकता नहीं है। यह custom (`custom/`), project convention (`.failproofai-project/`), और user convention (`.failproofai-user/`) नीतियों के लिए भी काम करता है। विवरण के लिए [Configuration → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

### सूचनात्मक allow संदेश

`allow(message)` ऑपरेशन को **और** Claude को एक सूचनात्मक संदेश भेजता है। संदेश hook handler के stdout response में `additionalContext` के रूप में दिया जाता है — `instruct` द्वारा उपयोग किया जाने वाला ही तंत्र, लेकिन अर्थ में भिन्न: यह एक warning नहीं है, एक स्थिति अपडेट है।

| Function | Effect | उपयोग करें जब |
|----------|--------|----------|
| `allow(message)` | अनुमति दें और Claude को संदर्भ भेजें | एक check पास होने की पुष्टि करें, या समझाएं कि check को क्यों skip किया गया था |

उपयोग के मामले:
- **स्थिति पुष्टियां:** `allow("All CI checks passed.")` — Claude को बताता है कि सब कुछ ठीक है
- **Fail-open व्याख्याएं:** `allow("GitHub CLI not installed, skipping CI check.")` — Claude को बताता है कि check को क्यों skip किया गया ताकि इसके पास पूर्ण संदर्भ हो
- **एकाधिक संदेश जमा होते हैं:** यदि कई नीतियां प्रत्येक `allow(message)` लौटाती हैं, तो सभी संदेश newlines के साथ जुड़े हुए होते हैं और एक साथ दिए जाते हैं

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### `PolicyContext` fields

| Field | Type | Description |
|-------|------|-------------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | कॉल किया जा रहा tool (जैसे `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | tool के input parameters |
| `payload` | `Record<string, unknown>` | Claude Code से पूर्ण raw event payload |
| `session` | `SessionMetadata \| undefined` | Session context (नीचे देखें) |

### `SessionMetadata` fields

| Field | Type | Description |
|-------|------|-------------|
| `sessionId` | `string` | Claude Code session identifier |
| `cwd` | `string` | Claude Code session की working directory |
| `transcriptPath` | `string` | Session की JSONL transcript फाइल का पथ |

### Event types

| Event | कब यह होता है | `toolInput` contents |
|-------|--------------|----------------------|
| `PreToolUse` | Claude एक tool चलाने से पहले | Tool का input (जैसे Bash के लिए `{ command: "..." }`) |
| `PostToolUse` | Tool पूरा होने के बाद | Tool का input + `tool_result` (output) |
| `Notification` | जब Claude एक notification भेजता है | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - hooks को हमेशा `allow()` लौटना चाहिए, वे notifications को block नहीं कर सकते |
| `Stop` | जब Claude session समाप्त होता है | Empty |

## मूल्यांकन क्रम

नीतियों का मूल्यांकन इस क्रम में होता है:

1. Built-in नीतियां (definition क्रम में)
2. `customPoliciesPath` से explicit कस्टम नीतियां (`.add()` क्रम में)
3. प्रोजेक्ट `.failproofai/policies/` से सम्मेलन नीतियां (फाइलें वर्णक्रमानुसार, के भीतर `.add()` क्रम)
4. उपयोगकर्ता `~/.failproofai/policies/` से सम्मेलन नीतियां (फाइलें वर्णक्रमानुसार, के भीतर `.add()` क्रम)

<Note>
पहली `deny` सभी subsequent नीतियों को short-circuit करती है। सभी `instruct` संदेश जमा होते हैं और एक साथ दिए जाते हैं।
</Note>

## Transitive imports

कस्टम नीति फाइलें relative paths का उपयोग करके स्थानीय modules को import कर सकती हैं:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

entry फाइल से सभी reachable relative imports resolved होते हैं। यह `from "failproofai"` imports को actual dist path में rewrite करके और ESM compatibility सुनिश्चित करने के लिए temporary `.mjs` फाइलें बनाकर लागू किया जाता है।

## Event type filtering

एक नीति कब चलती है इसे सीमित करने के लिए `match.events` का उपयोग करें:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // केवल तब होता है जब session समाप्त होता है
    // ctx.session.transcriptPath में पूर्ण session log होता है
    return allow();
  },
});
```

हर event type पर होने के लिए `match` को पूरी तरह से omit करें।

## Error handling और failure modes

कस्टम नीतियां **fail-open** हैं: errors कभी भी built-in नीतियों को block नहीं करते या hook handler को crash नहीं करते।

| Failure | Behavior |
|---------|----------|
| `customPoliciesPath` set नहीं है | कोई explicit कस्टम नीतियां नहीं चलती; सम्मेलन नीतियां और built-ins normally जारी रहते हैं |
| फाइल नहीं मिली | Warning `~/.failproofai/hook.log` में logged होती है; built-ins जारी रहते हैं |
| Syntax/import error (explicit) | Error `~/.failproofai/hook.log` में logged होती है; explicit कस्टम नीतियां skipped होती हैं |
| Syntax/import error (convention) | Error logged होती है; वह फाइल skipped होती है, अन्य सम्मेलन फाइलें अभी भी लोड होती हैं |
| `fn` runtime में throw होता है | Error logged होती है; वह hook `allow` के रूप में treated होता है; अन्य hooks जारी रहते हैं |
| `fn` 10s से अधिक समय लेता है | Timeout logged होती है; `allow` के रूप में treated होता है |
| सम्मेलन निर्देशिका missing है | कोई सम्मेलन नीतियां नहीं चलती; कोई error नहीं |

<Tip>
कस्टम नीति errors को debug करने के लिए, log फाइल को देखें:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## पूर्ण उदाहरण: कई नीतियां

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// एजेंट को secrets/ निर्देशिका में लिखने से रोकें
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// एजेंट को track पर रखें: committing से पहले tests को verify करें
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// freeze अवधि के दौरान अनियोजित dependency changes को रोकें
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```

## उदाहरण

`examples/` निर्देशिका में ready-to-run नीति फाइलें हैं:

| File | Contents |
|------|----------|
| `examples/policies-basic.js` | पांच starter नीतियां जो common agent failure modes को cover करती हैं |
| `examples/policies-advanced/index.js` | Advanced patterns: transitive imports, async calls, output scrubbing, और session-end hooks |
| `examples/convention-policies/security-policies.mjs` | सम्मेलन-आधारित security नीतियां (.env writes को block करें, git history rewriting को रोकें) |
| `examples/convention-policies/workflow-policies.mjs` | सम्मेलन-आधारित workflow नीतियां (test reminders, audit file writes) |

### Explicit file उदाहरण का उपयोग करना

```bash
failproofai policies --install --custom ./examples/policies-basic.js
```

### सम्मेलन-आधारित उदाहरण का उपयोग करना

```bash
# प्रोजेक्ट स्तर पर कॉपी करें
mkdir -p .failproofai/policies
cp examples/convention-policies/*.mjs .failproofai/policies/

# या उपयोगकर्ता स्तर पर कॉपी करें
mkdir -p ~/.failproofai/policies
cp examples/convention-policies/*.mjs ~/.failproofai/policies/
```

कोई install command आवश्यक नहीं — फाइलें अगले hook event पर स्वचालित रूप से pick up होती हैं।
