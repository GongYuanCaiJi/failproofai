failproofai 39 निर्मित नीतियों के साथ आता है जो सामान्य एजेंट विफलता मोड को पकड़ते हैं। प्रत्येक नीति एक विशिष्ट हुक इवेंट प्रकार और टूल नाम पर काम करती है। उन्नीस नीतियां पैरामीटर स्वीकार करती हैं जो आपको कोड लिखे बिना उनके व्यवहार को ट्यून करने देती हैं। पांच वर्कफ़्लो नीतियां commit → push → PR → CI पाइपलाइन को लागू करती हैं जिससे पहले Claude रुक जाता है।

---

## अवलोकन

नीतियों को श्रेणियों में समूहीकृत किया गया है:

| श्रेणी | नीतियां | हुक प्रकार |
|----------|----------|-----------|
| [खतरनाक कमांड](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [इंफ्रा कमांड](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [गोपनीयता (सैनिटाइज़र)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [पर्यावरण](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [फाइल एक्सेस](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [डेटाबेस](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [चेतावनियां](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [पैकेज प्रबंधक](#package-managers) | prefer-package-manager | PreToolUse |
| [वर्कफ़्लो](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — एजेंट को आगे बढ़ने से रोकें।
- **`warn-`** — एजेंट को अतिरिक्त संदर्भ दें ताकि वह स्व-सुधार कर सके।
- **`sanitize-`** — एजेंट को देखने से पहले टूल आउटपुट से संवेदनशील डेटा को साफ करें।

### नामस्पेस

प्रत्येक नीति एक `<namespace>/<name>` स्लॉट में रहती है। निर्मित नीतियां **`exospherehost/`** नामस्पेस से संबंधित हैं — उदाहरण के लिए, `exospherehost/sanitize-jwt`। नामस्पेस तब टकराव को रोकता है जब आप समान छोटे नामों वाली कस्टम या तृतीय-पक्ष नीतियों को भी लोड करते हैं।

आपके कॉन्फ़िग में आप निर्मित को या तो इसके छोटे नाम या योग्य नाम से संदर्भित कर सकते हैं; दोनों रूप समान नीति को हल करते हैं:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

यदि किसी नाम में `/` नहीं है, तो failproofai इसे डिफ़ॉल्ट नामस्पेस `exospherehost` से संबंधित मानता है। ऐसे नाम जिनमें पहले से `/` है (जैसे `myorg/foo`, `custom/my-hook`) जैसे-जैसे रखे जाते हैं।
- **`require-`** — स्टॉप इवेंट को ब्लॉक करें जब तक शर्तें पूरी न हों।

---

<Tip>
प्रत्येक नीति `policyParams` में एक वैकल्पिक `hint` फील्ड का समर्थन करती है। हिंट को deny या instruct संदेश में जोड़ा जाता है जो Claude देखता है, नीति कोड को संशोधित किए बिना कार्रवाई योग्य मार्गदर्शन देता है। निर्मित, कस्टम और कन्वेंशन नीतियों के साथ काम करता है। विवरण के लिए [कॉन्फ़िगरेशन → hint](/hi/configuration#hint-cross-cutting) देखें।
</Tip>

## खतरनाक कमांड

एजेंटों को ऐसे ऑपरेशन चलाने से रोकें जो पूर्ववत करने में मुश्किल हैं या जो होस्ट सिस्टम को नुकसान पहुंचा सकते हैं।

### `block-sudo`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `sudo` कमांड को नकारता है।

`sudo` कीवर्ड को शामिल करने वाली आह्वान को ब्लॉक करता है। पैटर्न मिलान कच्चे स्ट्रिंग पर नहीं, बल्कि पार्स की गई कमांड टोकन पर किया जाता है, जिससे शेल ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोका जा सके।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | सटीक कमांड उपसर्ग जो अनुमति हैं। प्रत्येक प्रविष्टि को पार्स किए गए argv टोकन के विरुद्ध मिलाया जाता है। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `sudo systemctl status nginx` की अनुमति है, लेकिन `sudo rm /etc/hosts` को नकार दिया जाता है।

<Note>
पैटर्न को कच्चे कमांड स्ट्रिंग के विरुद्ध नहीं, बल्कि पार्स किए गए टोकन के विरुद्ध मिलाया जाता है। यह अनुलग्न शेल ऑपरेटर के माध्यम से बाईपास को रोकता है (जैसे `sudo systemctl status x; rm -rf /` `sudo systemctl status *` से मेल नहीं खाता)।
</Note>

### `block-rm-rf`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `rm -rf`, `rm -fr`, और समान पुनरावर्ती विलोपन रूपों को नकारता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | पथ जो पुनरावर्ती रूप से हटाने के लिए सुरक्षित हैं (जैसे `/tmp`)। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

**डिफ़ॉल्ट:** `curl <url> | bash`, `curl <url> | sh`, `wget <url> | bash`, और समान पैटर्न को नकारता है।

कोई पैरामीटर नहीं।

कोई पैरामीटर नहीं।

## इंफ्रा कमांड

कोडिंग एजेंटों को इंफ्रास्ट्रक्चर CLIs चलाने या CI/CD पाइपलाइन को ट्रिगर करने से रोकें। इस श्रेणी में सभी नीतियां **ऑप्ट-इन** हैं (`defaultEnabled: false`) — ऐसे एजेंट जिन्हें वास्तव में `kubectl`, `terraform`, आदि को कॉल करने की आवश्यकता है, नीति सक्षम न होने तक व्यवहार में नहीं आएंगे। जब सक्षम किया जाता है, तो मिलान किए गए CLI की प्रत्येक आह्वान को नकार दिया जाता है जब तक कि कमांड `allowPatterns` में एक प्रविष्टि से मेल न खाए।

पैटर्न व्याकरण [`block-sudo`](#block-sudo) के समान है: टोकन को पार्स किए गए argv के विरुद्ध मिलाया जाता है, `*` एक टोकन के लिए जंगली कार्ड है, और कोई भी कमांड जिसमें एक स्टैंडअलोन शेल ऑपरेटर (`&&`, `||`, `|`, `;`) या एम्बेडेड शेल मेटाचार्स वाला टोकन है, इंजेक्शन बाईपास को रोकने के लिए allowlist मिलान से पहले अस्वीकार किया जाता है।

### `block-kubectl`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** किसी भी `kubectl` आह्वान को नकारता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | kubectl कमांड उपसर्ग जो अनुमति हैं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `kubectl get pods` की अनुमति है लेकिन `kubectl apply -f deploy.yaml` को नकार दिया जाता है।

### `block-gh-pipeline`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** निम्नलिखित `gh` CLI उप-आदेशों को नकारता है जो स्थिति को बदलते हैं या पाइपलाइन को ट्रिगर करते हैं:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

केवल-पढ़ने के लिए `gh` उप-आदेश जैसे `gh pr view`, `gh pr list`, `gh run list`, `gh release view`, और `gh api repos/.../...` इस नीति से मेल **नहीं** खाते हैं — वे वर्कफ़्लो जांच के लिए नियमित रूप से आवश्यक हैं (failproofai की अपनी `require-ci-green-before-stop` सहित)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | विशिष्ट स्क्रिप्ट की गई आह्वान जो अनुमति हैं भले ही वे अन्यथा नकार दिए जाएं। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## गोपनीयता (सैनिटाइज़र)

एजेंटों को अपने संदर्भ या आउटपुट में क्रेडेंशियल लीक करने से रोकें। सैनिटाइज़र नीतियां **PostToolUse** इवेंट पर काम करती हैं। जब Claude एक Bash कमांड चलाता है, एक फाइल पढ़ता है, या किसी भी टूल को कॉल करता है, तो ये नीतियां आउटपुट को Claude को वापस करने से पहले निरीक्षण करती हैं। यदि एक गोपनीय पैटर्न का पता चलता है, तो नीति एक नकार निर्णय लौटाती है जो आउटपुट को वापस पास करने से रोकता है।

### `sanitize-jwt`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** JWT टोकन को रिडैक्ट करता है (तीन base64url खंड `.` से अलग)।

कोई पैरामीटर नहीं।

---

### `sanitize-api-keys`

**इवेंट:** PostToolUse (सभी टूल)  
**डिफ़ॉल्ट:** सामान्य API कुंजी प्रारूपों को रिडैक्ट करता है: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS एक्सेस कुंजियां (`AKIA`), Stripe कुंजियां (`sk_live_`, `sk_test_`), और Google API कुंजियां (`AIza`)।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | गोपनीय के रूप में माने जाने के लिए अतिरिक्त regex पैटर्न। |

**उदाहरण:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

## फाइल एक्सेस

एजेंटों को प्रोजेक्ट सीमाओं के अंदर काम करने और संवेदनशील फाइलों से दूर रखें।

### `block-read-outside-cwd`

**इवेंट:** PreToolUse (Read, Bash)  
**डिफ़ॉल्ट:** प्रोजेक्ट रूट के बाहर फाइलों को पढ़ने को नकारता है। सीमा `CLAUDE_PROJECT_DIR` है (Claude Code द्वारा सत्र प्रति एक बार सेट), जब वह चर अनसेट हो तो सत्र की वर्तमान कार्यशील निर्देशिका का फॉलबैक। लाइव `cwd` के बजाय प्रोजेक्ट रूट का उपयोग करने का मतलब है कि सीमा स्थिर रहती है भले ही Claude `cd` एक सबडायरेक्टरी में जाता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | निरपेक्ष पथ उपसर्ग जो अनुमति हैं भले ही प्रोजेक्ट रूट के बाहर हों। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

**इवेंट:** PreToolUse (Write, Edit)

## Git

आकस्मिक pushes, force-pushes, और ब्रांच गलतियों को रोकें जो पूर्ववत करने में मुश्किल हैं।

### `block-push-master`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push origin main` और `git push origin master` को नकारता है।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जो सीधे push नहीं कर सकते। |

**उदाहरण:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
सभी ब्रांच को push करने की अनुमति देने के लिए (प्रभावी रूप से इस नीति को `enabledPolicies` से हटाए बिना अक्षम करने के लिए), `protectedBranches: []` सेट करें।
</Tip>

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | ब्रांच नाम जिन पर commit/merge/rebase/cherry-pick को नकार दिया जाता है। |

### `block-force-push`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** `git push --force` और `git push -f` को नकारता है।

कोई नीति-विशिष्ट पैरामीटर नहीं। विकल्पों का सुझाव देने के लिए क्रॉस-कटिंग [`hint`](/hi/configuration#hint-cross-cutting) का उपयोग करें:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "अपने वर्तमान HEAD से एक नई ब्रांच बनाएं (जैसे `git checkout -b <new-branch>`) और इसके बजाय उसे push करें।"
    }

}
```

## पैकेज प्रबंधक

एजेंट को उपयोग करने की अनुमति देने वाले पैकेज प्रबंधक को लागू करें।

### `prefer-package-manager`

**इवेंट:** PreToolUse (Bash)  
**डिफ़ॉल्ट:** अक्षम। जब सक्षम किया जाता है, तो `allowed` सूची में न होने वाली किसी भी पैकेज प्रबंधक कमांड को ब्लॉक करता है और Claude को अनुमत प्रबंधक का उपयोग करके कमांड को फिर से लिखने का निर्देश देता है।

पहचानता है: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo।

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | अनुमत पैकेज प्रबंधक नाम। इस सूची में न होने वाले किसी भी पहचाने गए प्रबंधक को ब्लॉक किया जाता है। जब खाली हो, तो नीति एक नो-ऑप है। |
| `blocked` | string[] | `[]` | निर्मित सूची के अतिरिक्त ब्लॉक करने के लिए अतिरिक्त प्रबंधक नाम (जैसे `['pdm', 'pipx']`)। |

निर्मित ब्लॉक सूची में शामिल हैं: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo। इस सूची में न होने वाले प्रबंधकों को जोड़ने के लिए `blocked` का उपयोग करें।

**उदाहरण कॉन्फ़िगरेशन:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

इस कॉन्फ़िग के साथ, `pip install flask` और `pdm install flask` दोनों को Claude को `uv` या `bun` का उपयोग करने का निर्देश देते हुए एक संदेश के साथ नकार दिया जाता है। `uv pip install flask` जैसी कमांडें अनुमत हैं क्योंकि `uv` allowlist में है और पहले जांचा जाता है।

## वर्कफ़्लो

एक अनुशासित सत्र-समाप्ति वर्कफ़्लो को लागू करें। ये नीतियां **Stop** इवेंट पर काम करती हैं और Claude को तब तक रोकने से नकारती हैं जब तक प्रत्येक शर्त पूरी न हो जाए। वे एक प्राकृतिक निर्भरता श्रृंखला का पालन करते हैं: commit → push → PR → CI। यदि कोई नीति नकारती है, तो श्रृंखला में बाद की नीतियों को छोड़ दिया जाता है (नकार शॉर्ट-सर्किट)।

सभी वर्कफ़्लो नीतियां **fail-open** हैं: यदि आवश्यक टूल उपलब्ध नहीं है (जैसे `gh` स्थापित नहीं है, कोई git रिमोट नहीं), तो नीति एक सूचनात्मक संदेश के साथ अनुमति देती है जो समझाता है कि जांच को क्यों छोड़ा गया था।

### `require-commit-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब अपरिवर्तित परिवर्तन (संशोधित, मंचित, या अनट्रैक्ड फाइलें) हों तो रोकने को नकारता है। कार्यशील निर्देशिका स्वच्छ होने पर एक सूचनात्मक संदेश लौटाता है।

कोई पैरामीटर नहीं।

---

### `require-push-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब अपुष्ट commits हों या वर्तमान ब्रांच के पास कोई रिमोट ट्रैकिंग ब्रांच नहीं हो तो रोकने को नकारता है। यदि आवश्यक हो तो `git push -u` का सुझाव देता है ट्रैकिंग ब्रांच बनाने के लिए। यदि कोई रिमोट कॉन्फ़िगर नहीं है तो fail-open।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | push करने के लिए रिमोट नाम। |

**उदाहरण:**

```json
{
  "policyParams": {
    "require-push-before-stop": {
      "remote": "upstream"
    }
  }
}
```

---

### `require-pr-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान ब्रांच के लिए कोई pull request मौजूद न हो, या जब मौजूदा PR बिना मर्ज किए बंद हो गया हो तो रोकने को नकारता है। Claude को `gh pr create` के साथ एक PR बनाने का निर्देश देता है। जब PR **merged** हो जाता है, तो नीति अनुमति देती है (काम ही शिप हो गया है) और संदेश ब्रांच को बंद करने का सुझाव देता है (`git checkout main && git pull`)।

कोई पैरामीटर नहीं।

<Note>
यह नीति [GitHub CLI](https://cli.github.com/) (`gh`) को स्थापित और प्रमाणित होने की आवश्यकता है।
pull requests को पढ़ने की पहुंच के लिए `repo` स्कोप वाले व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएं। यदि `gh` स्थापित या प्रमाणित नहीं है,
तो नीति fail-open करती है और कारण की रिपोर्ट करती है।
</Note>

---

### `require-no-conflicts-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान ब्रांच बेस ब्रांच में स्वच्छ रूप से मर्ज नहीं हो सकता तो रोकने को नकारता है। नीति पहले पुष्टि करती है कि GitHub पर ब्रांच के लिए एक `OPEN` PR है — बिना एक के, मर्ज लक्ष्य को लागू नहीं है, इसलिए पूरी नीति short-circuit करती है अनुमति के लिए। एक बार `OPEN` PR की पुष्टि हो जाने के बाद, दो स्वतंत्र जांच चलाई जाती हैं:

1. **स्थानीय** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`। विरोध पर, नकार संदेश विरोधित फाइलों का नाम देता है ताकि Claude को बिल्कुल पता हो कि क्या हल करना है।
2. **GitHub** — पहले से ही पूर्वजांच में लाए गए `gh pr view --json mergeable,state` परिणाम को फिर से उपयोग करता है। विरोध को पकड़ता है जिसे एक स्टेल स्थानीय `origin/<baseBranch>` मिस करेगा (जैसे कोई किसी को बेस पर `main` पर एक विरोधी PR उतारा)। एक `CONFLICTING` परिणाम नकारता है। एक `UNKNOWN` परिणाम भी नकारता है और Claude को ~10 सेकंड प्रतीक्षा करने और फिर से जांचने का निर्देश देता है रुकने से पहले — यह GitHub पुनर्गणना करते समय गलत नकारात्मक को रोकता है।

पूरी तरह छोड़ देता है (अनुमति देता है) जब: `gh` स्थापित नहीं है, ब्रांच के लिए कोई PR मौजूद नहीं है, PR की स्थिति `OPEN` नहीं है (जैसे `MERGED`, `CLOSED`), या `gh pr view` अपार्सेबल आउटपुट लौटाता है। स्थानीय रूप से `origin/<baseBranch>` गायब होने या जब कोई commits बेस से आगे न हों तो भी fail-open होता है — वे Layer 1 fall-through अनुमति देने से पहले अभी भी cached PR mergeability से परामर्श लेते हैं।

**पैरामीटर:**

| पैरामीटर | प्रकार | डिफ़ॉल्ट | विवरण |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | विरोध के लिए जांचने के लिए बेस ब्रांच। |

<Note>
यह नीति GitHub CLI (`gh`) की आवश्यकता है। नीति किसी भी विरोध जांच को चलाने से पहले
एक `OPEN` PR की पुष्टि करने के लिए `gh pr view` का उपयोग करती है — बिना `gh` के,
नीति allow के लिए short-circuit करती है। pull requests को पढ़ने की पहुंच के लिए `repo` स्कोप के साथ व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएं।
</Note>

### `require-ci-green-before-stop`

**इवेंट:** Stop  
**डिफ़ॉल्ट:** जब वर्तमान ब्रांच पर CI जांच विफल हो रही हों या चल रही हों तो रोकने को नकारता है। GitHub Actions वर्कफ़्लो रन और तृतीय-पक्ष बॉट जांच दोनों को जांचता है (जैसे CodeRabbit, SonarCloud, Codecov)। `skipped` और `cancelled` निष्कर्षों को सफलता के रूप में व्यवहार करता है। सभी जांच पास होने पर एक सूचनात्मक संदेश लौटाता है।

कोई पैरामीटर नहीं।

<Note>
यह नीति [GitHub CLI](https://cli.github.com/) (`gh`) को स्थापित और प्रमाणित होने की आवश्यकता है।
GitHub Actions वर्कफ़्लो रन और Checks API को पढ़ने की पहुंच के लिए `repo` स्कोप वाले व्यक्तिगत एक्सेस टोकन के साथ `gh auth login` चलाएं। यदि `gh` स्थापित या प्रमाणित नहीं है, तो नीति fail-open करती है और कारण की रिपोर्ट करती है।
</Note>

## व्यक्तिगत नीतियों को अक्षम करना

अपने कॉन्फ़िग में `enabledPolicies` से एक विशिष्ट नीति हटाएं, या डैशबोर्ड के Policies टैब में इसे टॉगल बंद करें।

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

`enabledPolicies` में सूचीबद्ध नहीं की गई नीतियां नहीं चलती हैं, भले ही उनके लिए `policyParams` प्रविष्टियां मौजूद हों।
