failproofai JSON configuration files का उपयोग करके नियंत्रित करता है कि कौन सी policies सक्रिय हैं, वे कैसे व्यवहार करती हैं, और कस्टम policies कहाँ से लोड की जाती हैं। Configuration को आपकी टीम के साथ साझा करना आसान है - इसे अपने repo में commit करें और हर developer को समान agent safety net मिलता है।

---

## Configuration scopes

तीन configuration scopes हैं, जिनका मूल्यांकन प्राथमिकता क्रम में किया जाता है:

| Scope | File path | Purpose |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | Per-repo settings, version control में commit किए गए |
| **local** | `.failproofai/policies-config.local.json` | Personal per-repo overrides, gitignored |
| **global** | `~/.failproofai/policies-config.json` | सभी projects में user-level defaults |

जब failproofai को hook event मिलता है, तो यह वर्तमान working directory के लिए सभी तीनों files को लोड करता है और merge करता है।

### Merge rules

**`enabledPolicies`** - सभी तीनों scopes का union। कोई भी policy जो किसी भी level पर enabled है, सक्रिय है।

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** - पहला scope जो किसी policy के लिए params define करता है, पूरी तरह जीत जाता है। policy के params के भीतर values की कोई deep merging नहीं है।

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project जीतता है, global ignore होता है
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← global तक फॉलथ्रू होता है
```

**`customPoliciesPath`** - पहला scope जो इसे define करता है, जीत जाता है।

**`llm`** - पहला scope जो इसे define करता है, जीत जाता है।

## Config file format

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Field reference

### `enabledPolicies`

Type: `string[]`

Enable करने के लिए policy names की list। Names को `failproofai policies` द्वारा दिखाए गए policy identifiers से बिल्कुल match करना चाहिए। संपूर्ण list के लिए [Built-in Policies](/hi/built-in-policies) देखें।

`enabledPolicies` में नहीं होने वाली policies निष्क्रिय हैं, भले ही उनके `policyParams` में entries हों।

### `policyParams`

Type: `Record<string, Record<string, unknown>>`

Per-policy parameter overrides। बाहरी key policy name है; भीतरी keys policy-specific हैं। प्रत्येक policy [Built-in Policies](/hi/built-in-policies) में अपने उपलब्ध parameters को document करता है।

यदि किसी policy के parameters हैं लेकिन आप उन्हें specify नहीं करते, तो policy के built-in defaults का उपयोग किया जाता है। जो users `policyParams` को बिल्कुल configure नहीं करते वे पिछले versions के समान व्यवहार प्राप्त करते हैं।

policy के params block के अंदर Unknown keys को hook-fire time पर silently ignore किया जाता है लेकिन `failproofai policies` चलाने पर warnings के रूप में flagged किया जाता है।

#### `hint` (cross-cutting)

Type: `string` (optional)

एक message जो policy के `deny` या `instruct` return करने पर reason में जोड़ा जाता है। इसका उपयोग Claude को actionable guidance देने के लिए करें policy को modify किए बिना।

किसी भी policy type के साथ काम करता है — built-in, custom (`custom/`), project convention (`.failproofai-project/`), या user convention (`.failproofai-user/`)।

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

जब `block-force-push` deny करता है, Claude देखता है: *"Force-pushing is blocked. Try creating a fresh branch instead."*

Non-string values और empty strings को silently ignore किया जाता है। यदि `hint` set नहीं है, तो behavior unchanged है (backward-compatible)।

### `customPoliciesPath`

Type: `string` (absolute path)

कस्टम hook policies वाली JavaScript file का path। यह `failproofai policies --install --custom <path>` द्वारा automatically set किया जाता है (path को store करने से पहले absolute में resolve किया जाता है)।

File को हर hook event पर fresh लोड किया जाता है - कोई caching नहीं है। Authoring details के लिए [Custom Policies](/hi/custom-policies) देखें।

### Convention-based policies

Explicit `customPoliciesPath` के अलावा, failproofai automatically `.failproofai/policies/` directories से policy files को discover और load करता है:

| Level | Directory | Scope |
|-------|-----------|-------|
| Project | `.failproofai/policies/` | Version control के माध्यम से टीम के साथ shared |
| User | `~/.failproofai/policies/` | Personal, सभी projects पर लागू होता है |

**File matching:** केवल `*policies.{js,mjs,ts}` matching files load होती हैं (जैसे `security-policies.mjs`, `workflow-policies.js`)। Directory में अन्य files ignore होती हैं।

**कोई config आवश्यक नहीं:** Convention policies को `policies-config.json` में कोई entries की आवश्यकता नहीं है। बस directory में files डालें और वे अगली hook event पर pick up होती हैं।

**Union loading:** Project और user दोनों convention directories को scan किया जाता है। दोनों levels से सभी matching files load होती हैं (`customPoliciesPath` के विपरीत जो first-scope-wins का उपयोग करता है)।

अधिक विवरण और उदाहरणों के लिए [Custom Policies](/hi/custom-policies) देखें।

### `llm`

Type: `object` (optional)

उन policies के लिए LLM client configuration जो AI calls करती हैं। अधिकांश setups के लिए आवश्यक नहीं है।

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI से configuration को manage करना

`policies --install` और `policies --uninstall` commands आपके agent CLI की hook settings file (hook entry points) में लिखते हैं, जबकि `policies-config.json` वह file है जिसे आप directly manage करते हैं। दोनों अलग हैं:

- **Agent CLI settings** — agent को हर tool use पर `failproofai --hook <event>` call करने के लिए बताता है:
  - **Claude Code**: `~/.claude/settings.json` (user), `<cwd>/.claude/settings.json` (project), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (user), `<cwd>/.codex/hooks.json` (project) — Codex के पास `local` scope नहीं है
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (user), `<cwd>/.github/hooks/failproofai.json` (project) — Copilot के पास `local` scope नहीं है। Hook entries Copilot के OS-keyed `bash`/`powershell` command fields को `timeoutSec` के साथ उपयोग करते हैं; file एक top-level `version: 1` marker रखता है। Copilot CLI support **beta** है जबकि हम `events.jsonl` record schema को (जो public docs specify नहीं करते) अधिक real-world sessions के विरुद्ध verify करते हैं।
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (user), `<cwd>/.cursor/hooks.json` (project) — Cursor के पास `local` scope नहीं है। Hook entries Claude-shaped `{type, command, timeout}` form का उपयोग करते हैं (कोई `bash`/`powershell` split नहीं), लेकिन camelCase event keys (`preToolUse`, `beforeSubmitPrompt`, …) के अंतर्गत Cursor के [hooks schema](https://cursor.com/docs/hooks) के अनुसार एक flat array में stored हैं; file एक top-level `version: 1` marker रखता है। Handler camelCase → PascalCase को `CURSOR_EVENT_MAP` के माध्यम से canonicalize करता है ताकि existing builtin policies unchanged fire हों। Cursor Agent support **beta** है जबकि हम Cursor के transcript on-disk format (public docs में specified नहीं) को अधिक real-world installs के विरुद्ध verify करते हैं।
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (user), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (project) — OpenCode के पास `local` scope नहीं है। अन्य चार CLIs के विपरीत, OpenCode के पास **कोई external-command hook system नहीं है**: यह in-process JS/TS plugins को explicitly registered में लोड करता है `plugin: []` array के माध्यम से `opencode.json` में (`.opencode/plugins/` से auto-discovery **नहीं** है कि कैसे plugins OpenCode v1.14.33 पर लोड होते हैं)। Install एक small generated plugin shim को drop करता है जो failproofai binary को subprocess-call करता है और binary के Claude-shape JSON response को plugin semantics में translate करता है (`throw new Error()` deny के लिए, `client.session.prompt(...)` instruct के लिए, allow के लिए no-op)। Sessions OpenCode के SQLite DB में `~/.local/share/opencode/opencode.db` पर रहते हैं; dashboard का session viewer उन्हें `opencode db --format json` और `opencode export <id>` के माध्यम से पढ़ता है। OpenCode support **beta** है जबकि हम versions के पार और अधिक real-world sessions के विरुद्ध behavior verify करते हैं। [OpenCode plugins docs](https://opencode.ai/docs/plugins/) देखें।
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (user), `<cwd>/.pi/settings.json` (project) — Pi के पास `local` scope नहीं है। Pi startup पर TypeScript extension packages को लोड करता है; settings file एक flat string array `{"packages": ["./relative/path", …]}` है। failproofai एक single packages-array entry को लिखता है जो अपनी bundled `pi-extension/` directory की ओर point करता है। Extension internally Pi के `tool_call` / `user_bash` / `input` / `session_start` events को subscribe करता है और `failproofai --hook <Event> --cli pi` में shell out करता है; handler underscore_lower_snake_case → PascalCase को `PI_EVENT_MAP` के माध्यम से canonicalize करता है ताकि existing builtin policies unchanged fire हों। Pi support **beta** है जबकि Pi के extension API और session-log layout stabilize होते हैं।
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (user), `<cwd>/.gemini/settings.json` (project) — Gemini के पास `local` scope नहीं है (यह `/etc/gemini-cli/settings.json` पर एक `system` scope को document करता है जिसे failproofai expose नहीं करता)। Hook entries Claude के `{type, command, timeout}` form को Gemini के `{matcher, hooks: [...]}` matcher schema में wrap करते हैं जहाँ default रूप से `matcher: "*"` होता है। Events PascalCase हैं (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); handler `GEMINI_EVENT_MAP` के माध्यम से Claude canonical names में maps करता है। Tool names snake_case हैं (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — handler `GEMINI_TOOL_MAP` के माध्यम से canonicalize करता है ताकि existing builtin policies unchanged fire हों। Policy evaluator Gemini के flat `{decision: "deny", reason}` shape को emit करता है (Gemini के "Golden Rule" exit-0 contract per preferred), `{hookSpecificOutput: {hookEventName, additionalContext}}` context injection के लिए BeforeAgent / AfterTool / SessionStart पर, और `{decision: "block", reason}` AfterAgent पर force-retry semantics के लिए। Gemini CLI support **beta** है जबकि हम real-world coverage को widen करते हैं। [Gemini CLI hooks docs](https://geminicli.com/docs/hooks/) देखें।
- **`policies-config.json`** — failproofai को बताता है कि कौन सी policies को evaluate करना है और किन params के साथ (सभी agent CLIs में shared)

किसी specific agent को target करने के लिए `--cli claude|codex|copilot|cursor|opencode|pi|gemini` pass करें (space-separated या किसी भी subset के लिए repeated):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

जब `--cli` omit होता है, `failproofai` detect करता है कि कौन से agent CLIs installed हैं (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **एक CLI detected** — बिना prompting के उस CLI को auto-select करता है।
- **Multiple CLIs detected** एक interactive terminal में — एक arrow-key single-select prompt दिखाता है जो एक `Detected (N)` section में grouped होता है (एक `Install for all N detected` aggregate row + प्रत्येक detected CLI individually के साथ) और एक `Not installed (M) · install hooks ahead of time` section जो हर undetected supported CLI को forward-install option के रूप में list करता है (↑↓ move करने के लिए, Enter select करने के लिए, ^C quit करने के लिए)। Uninstall flow केवल Detected section दिखाता है।
- **Multiple CLIs detected** एक non-interactive run में (CI, no TTY) — बिना prompting के सभी detected CLIs के लिए install करता है।
- **कोई नहीं detected** — `claude` में fallback करता है, एक warning के साथ कि PATH में कोई agent binary नहीं मिला; hook command फिर भी लिखा जाता है ताकि यह activate हो जब आप एक install करते हैं।

आप `policies-config.json` को किसी भी समय directly edit कर सकते हैं; changes अगली hook event पर बिना restart के तुरंत प्रभाव लेते हैं।

## Example: project-level config with team defaults

`.failproofai/policies-config.json` को अपने repo में commit करें:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

प्रत्येक developer फिर `.failproofai/policies-config.local.json` (gitignored) create कर सकता है personal overrides के लिए teammates को प्रभावित किए बिना।
