failproofai JSON कॉन्फ़िगरेशन फ़ाइलों का उपयोग करता है यह नियंत्रित करने के लिए कि कौन सी नीतियां सक्रिय हैं, वे कैसे काम करती हैं, और कस्टम नीतियां कहां से लोड की जाती हैं। कॉन्फ़िगरेशन को आपकी टीम के साथ साझा करना आसान है - इसे अपने रिपो में कमिट करें और हर डेवलपर को एक जैसी एजेंट सुरक्षा मिलती है।

---

## कॉन्फ़िगरेशन स्कोप

तीन कॉन्फ़िगरेशन स्कोप हैं, जिनका मूल्यांकन प्राथमिकता क्रम में किया जाता है:

| स्कोप | फ़ाइल पाथ | उद्देश्य |
|-------|-----------|---------|
| **project** | `.failproofai/policies-config.json` | प्रति-रिपो सेटिंग, संस्करण नियंत्रण में कमिट की गई |
| **local** | `.failproofai/policies-config.local.json` | व्यक्तिगत प्रति-रिपो ओवरराइड, gitignored |
| **global** | `~/.failproofai/policies-config.json` | सभी परियोजनाओं में उपयोगकर्ता-स्तरीय डिफ़ॉल्ट |

जब failproofai को एक हुक ईवेंट मिलता है, तो यह मौजूदा कार्यशील निर्देशिका के लिए सभी तीन फ़ाइलों को लोड और मर्ज करता है।

### मर्ज नियम

**`enabledPolicies`** - सभी तीन स्कोप का संघ। कोई भी नीति किसी भी स्तर पर सक्षम हो तो सक्रिय है।

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← डीडुप्लिकेटेड संघ
```

**`policyParams`** - किसी दी गई नीति के लिए पैरामीटर को परिभाषित करने वाला पहला स्कोप पूरी तरह जीतता है। नीति के पैरामीटर के भीतर मूल्यों का कोई गहरा मर्ज नहीं है।

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project जीतता है, global को नजरअंदाज किया जाता है
```

```text
project:  (कोई block-sudo प्रविष्टि नहीं)
local:    (कोई block-sudo प्रविष्टि नहीं)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← global तक गिरता है
```

**`customPoliciesPath`** - पहला स्कोप जो इसे परिभाषित करता है जीतता है।

**`llm`** - पहला स्कोप जो इसे परिभाषित करता है जीतता है।

## कॉन्फ़िग फ़ाइल प्रारूप

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## फ़ील्ड संदर्भ

### `enabledPolicies`

प्रकार: `string[]`

सक्षम करने के लिए नीति नामों की सूची। नामों को `failproofai policies` द्वारा दिखाए गए नीति पहचानकर्ताओं से बिल्कुल मेल खाना चाहिए। संपूर्ण सूची के लिए [Built-in Policies](/hi/built-in-policies) देखें।

`enabledPolicies` में न होने वाली नीतियां निष्क्रिय हैं, भले ही उनके पास `policyParams` में प्रविष्टियां हों।

### `policyParams`

प्रकार: `Record<string, Record<string, unknown>>`

प्रति-नीति पैरामीटर ओवरराइड। बाहरी कुंजी नीति का नाम है; भीतरी कुंजियां नीति-विशिष्ट हैं। प्रत्येक नीति [Built-in Policies](/hi/built-in-policies) में अपने उपलब्ध पैरामीटर को दस्तावेज करती है।

यदि किसी नीति के पैरामीटर हैं लेकिन आप उन्हें निर्दिष्ट नहीं करते हैं, तो नीति के built-in डिफ़ॉल्ट का उपयोग किया जाता है। जो उपयोगकर्ता `policyParams` को बिल्कुल कॉन्फ़िगर नहीं करते हैं वे पिछले संस्करणों के समान व्यवहार प्राप्त करते हैं।

किसी नीति के पैरामीटर ब्लॉक के अंदर अज्ञात कुंजियों को hook-fire समय पर चुप चाप नजरअंदाज किया जाता है लेकिन जब आप `failproofai policies` चलाते हैं तो चेतावनियों के रूप में फ़्लैग किया जाता है।

#### `hint` (क्रॉस-कटिंग)

प्रकार: `string` (वैकल्पिक)

एक संदेश जो कारण में जोड़ा जाता है जब कोई नीति `deny` या `instruct` देता है। इसे Claude को नीति को स्वयं संशोधित किए बिना कार्यकारी मार्गदर्शन देने के लिए उपयोग करें।

किसी भी नीति प्रकार के साथ काम करता है — built-in, कस्टम (`custom/`), परियोजना सम्मेलन (`.failproofai-project/`), या उपयोगकर्ता सम्मेलन (`.failproofai-user/`)।

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

जब `block-force-push` को अस्वीकार करता है, Claude देखता है: *Force-pushing is blocked. Try creating a fresh branch instead.*

गैर-स्ट्रिंग मान और खाली स्ट्रिंग को चुप चाप नजरअंदाज किया जाता है। यदि `hint` सेट नहीं है, तो व्यवहार अपरिवर्तित है (पीछे की ओर संगत)।

### `customPoliciesPath`

प्रकार: `string` (पूर्ण पाथ)

कस्टम हुक नीतियों वाली JavaScript फ़ाइल का पाथ। यह स्वचालित रूप से `failproofai policies --install --custom <path>` द्वारा सेट किया जाता है (पाथ को स्टोर करने से पहले पूर्ण में हल किया जाता है)।

फ़ाइल हर हुक ईवेंट पर ताज़ा लोड होती है - कोई कैशिंग नहीं है। लेखन विवरण के लिए [Custom Policies](/hi/custom-policies) देखें।

### परंपरा-आधारित नीतियां

स्पष्ट `customPoliciesPath` के अलावा, failproofai स्वचालित रूप से `.failproofai/policies/` निर्देशिकाओं से नीति फ़ाइलों को खोजता और लोड करता है:

| स्तर | निर्देशिका | स्कोप |
|-------|-----------|-------|
| परियोजना | `.failproofai/policies/` | संस्करण नियंत्रण के माध्यम से टीम के साथ साझा |
| उपयोगकर्ता | `~/.failproofai/policies/` | व्यक्तिगत, सभी परियोजनाओं पर लागू |

**फ़ाइल मिलान:** केवल `*policies.{js,mjs,ts}` से मेल खाने वाली फ़ाइलें लोड की जाती हैं (उदा. `security-policies.mjs`, `workflow-policies.js`)। निर्देशिका में अन्य फ़ाइलें नजरअंदाज की जाती हैं।

**कोई कॉन्फ़िग की आवश्यकता नहीं:** परंपरा नीतियों को `policies-config.json` में कोई प्रविष्टि की आवश्यकता नहीं है। बस निर्देशिका में फ़ाइलें छोड़ें और अगली हुक ईवेंट पर उन्हें उठाया जाता है।

**संघ लोडिंग:** परियोजना और उपयोगकर्ता दोनों परंपरा निर्देशिकाओं को स्कैन किया जाता है। दोनों स्तरों से सभी मेल खाने वाली फ़ाइलें लोड की जाती हैं (`customPoliciesPath` के विपरीत जो पहली-स्कोप-जीतता है का उपयोग करता है)।

अधिक विवरण और उदाहरणों के लिए [Custom Policies](/hi/custom-policies) देखें।

### `llm`

प्रकार: `object` (वैकल्पिक)

AI कॉल करने वाली नीतियों के लिए LLM क्लाइंट कॉन्फ़िगरेशन। अधिकांश सेटअप के लिए आवश्यक नहीं है।

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## CLI से कॉन्फ़िगरेशन प्रबंधित करना

`policies --install` और `policies --uninstall` कमांड आपके एजेंट CLI की हुक सेटिंग फ़ाइल (हुक प्रविष्टि बिंदु) में लिखते हैं, जबकि `policies-config.json` वह फ़ाइल है जिसे आप सीधे प्रबंधित करते हैं। दोनों अलग हैं:

- **एजेंट CLI सेटिंग्स** — एजेंट को प्रत्येक टूल उपयोग पर `failproofai --hook <event>` कॉल करने के लिए बताता है:
  - **Claude Code**: `~/.claude/settings.json` (user), `<cwd>/.claude/settings.json` (project), `<cwd>/.claude/settings.local.json` (local)
  - **OpenAI Codex**: `~/.codex/hooks.json` (user), `<cwd>/.codex/hooks.json` (project) — Codex के पास `local` स्कोप नहीं है
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (user), `<cwd>/.github/hooks/failproofai.json` (project) — Copilot के पास कोई `local` स्कोप नहीं है। हुक प्रविष्टियां Copilot के OS-keyed `bash`/`powershell` कमांड फ़ील्ड का उपयोग करती हैं `timeoutSec` के साथ; फ़ाइल एक top-level `version: 1` मार्कर रखती है। Copilot CLI समर्थन **beta** है जबकि हम `events.jsonl` रिकॉर्ड स्कीमा को सत्यापित करते हैं (जिसे public docs में निर्दिष्ट नहीं किया जाता)।
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (user), `<cwd>/.cursor/hooks.json` (project) — Cursor के पास कोई `local` स्कोप नहीं है। हुक प्रविष्टियां Claude-shaped `{type, command, timeout}` फॉर्म का उपयोग करती हैं (कोई `bash`/`powershell` विभाजन नहीं), लेकिन Cursor के [hooks schema](https://cursor.com/docs/hooks) के अनुसार एक फ्लैट array में camelCase event keys (`preToolUse`, `beforeSubmitPrompt`, …) के तहत संग्रहीत। फ़ाइल एक top-level `version: 1` मार्कर रखती है। हैंडलर `CURSOR_EVENT_MAP` के माध्यम से camelCase → PascalCase को canonicalize करता है ताकि मौजूदा builtin नीतियां अपरिवर्तित फायरिंग करें। Cursor Agent समर्थन **beta** है जबकि हम Cursor के transcript on-disk फॉर्मेट को सत्यापित करते हैं (public docs में निर्दिष्ट नहीं)।
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (user), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (project) — OpenCode के पास कोई `local` स्कोप नहीं है। अन्य छह CLIs के विपरीत, OpenCode के पास **कोई external-command हुक सिस्टम नहीं है**: यह `opencode.json` में `plugin: []` array के माध्यम से स्पष्ट रूप से पंजीकृत in-process JS/TS plugins लोड करता है (`.opencode/plugins/` से auto-discovery यह नहीं है कि opencode v1.14.33 में plugins कैसे लोड होते हैं)। Install एक छोटा generated plugin shim छोड़ता है जो failproofai binary को subprocess-call करता है और binary के Claude-shape JSON response को plugin semantics में वापस अनुवाद करता है: tool-event deny के लिए `throw new Error()` (tool call को रद्द करता है), instruct के लिए `client.session.prompt(...)` और `Stop` / `SubagentStop` deny (deny reason को अगला उपयोगकर्ता संदेश के रूप में जमा करता है — एकमात्र force-retry चैनल क्योंकि `session.idle` notification-only है और इससे फेंकना एक no-op है), और allow के लिए no-op। Sessions opencode के SQLite DB में रहते हैं `~/.local/share/opencode/opencode.db` पर; dashboard का session viewer उन्हें `opencode db --format json` और `opencode export <id>` के माध्यम से पढ़ता है। OpenCode समर्थन **beta** है जबकि हम versions और more real-world sessions के across behavior को सत्यापित करते हैं। [OpenCode plugins docs](https://opencode.ai/docs/plugins/) देखें।
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (user), `<cwd>/.pi/settings.json` (project) — Pi के पास कोई `local` स्कोप नहीं है। Pi startup पर TypeScript extension packages लोड करता है; settings file एक फ्लैट string array `{"packages": ["./relative/path", …]}` है। failproofai एक single packages-array entry लिखता है जो अपनी bundled `pi-extension/` निर्देशिका की ओर इशारा करता है। Extension आंतरिक रूप से Pi के `tool_call` / `user_bash` / `input` / `session_start` events को subscribe करता है और `failproofai --hook <Event> --cli pi` को shell out करता है; हैंडलर `PI_EVENT_MAP` के माध्यम से underscore_lower_snake_case → PascalCase को canonicalize करता है ताकि मौजूदा builtin नीतियां अपरिवर्तित फायरिंग करें। Pi समर्थन **beta** है जबकि Pi का extension API और session-log layout स्थिर होता है।
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (user), `<cwd>/.gemini/settings.json` (project) — Gemini के पास कोई `local` स्कोप नहीं है (यह `/etc/gemini-cli/settings.json` पर एक `system` स्कोप को document करता है जिसे failproofai expose नहीं करता)। हुक प्रविष्टियां Claude के `{type, command, timeout}` फॉर्म का उपयोग करती हैं Gemini के `{matcher, hooks: [...]}` matcher schema में wrapped जिसमें default `matcher: "*"` है। Events PascalCase में हैं (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); हैंडलर `GEMINI_EVENT_MAP` के माध्यम से Claude canonical names से मैप करता है। Tool names snake_case में हैं (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — हैंडलर `GEMINI_TOOL_MAP` के माध्यम से canonicalize करता है ताकि मौजूदा builtin नीतियां अपरिवर्तित फायरिंग करें। Policy evaluator Gemini के flat `{decision: "deny", reason}` shape को emit करता है (Gemini के "Golden Rule" exit-0 contract के अनुसार preferred), BeforeAgent / AfterTool / SessionStart पर context injection के लिए `{hookSpecificOutput: {hookEventName, additionalContext}}`, और AfterAgent पर `{decision: "block", reason}` force-retry semantics के लिए। Gemini CLI समर्थन **beta** है जबकि हम real-world coverage को चौड़ा करते हैं। [Gemini CLI hooks docs](https://geminicli.com/docs/hooks/) देखें।
- **`policies-config.json`** — failproofai को बताता है कि कौन सी नीतियों का मूल्यांकन करना है और किस params के साथ (सभी agent CLIs में साझा)

किसी विशिष्ट एजेंट को target करने के लिए `--cli claude|codex|copilot|cursor|opencode|pi|gemini` पास करें (space-separated या किसी subset के लिए दोहराया गया):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

जब `--cli` को छोड़ा जाता है, `failproofai` यह पहचानता है कि कौन से agent CLIs installed हैं (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **एक CLI पहचाना गया** — बिना prompt दिए उस CLI को auto-select करता है।
- **Multiple CLIs पहचाने गए** एक interactive terminal में — एक arrow-key single-select prompt दिखाता है जिसे `Detected (N)` section में grouped किया गया है (एक `Install for all N detected` aggregate row + प्रत्येक detected CLI individually के साथ) और एक `Not installed (M) · install hooks ahead of time` section जो हर undetected supported CLI को एक forward-install option के रूप में सूचीबद्ध करता है (↑↓ move करने के लिए, Enter select करने के लिए, ^C quit करने के लिए)। Uninstall flow केवल Detected section दिखाता है।
- **Multiple CLIs पहचाने गए** एक non-interactive run में (CI, no TTY) — बिना prompt दिए सभी detected CLIs के लिए install करता है।
- **कोई नहीं पहचाना गया** — `claude` में fallback करता है, एक warning के साथ कि कोई agent binary PATH में नहीं मिला; हुक कमांड अभी भी लिखा जाता है ताकि यह सक्रिय हो जाए जैसे ही आप एक install करते हैं।

आप किसी भी समय `policies-config.json` को सीधे edit कर सकते हैं; changes अगली हुक ईवेंट पर तुरंत प्रभावी होते हैं, कोई restart की आवश्यकता नहीं है।

## उदाहरण: टीम डिफ़ॉल्ट के साथ परियोजना-स्तरीय कॉन्फ़िगरेशन

`.failproofai/policies-config.json` को अपने रिपो में कमिट करें:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

हर डेवलपर बाद में `.failproofai/policies-config.local.json` (gitignored) बना सकता है व्यक्तिगत overrides के लिए बिना teammates को प्रभावित किए।
