यह दस्तावेज़ failproofai कैसे आंतरिक रूप से काम करता है, यह समझाता है: हुक सिस्टम एजेंट टूल कॉलों को कैसे इंटरसेप्ट करता है, कॉन्फिग कैसे लोड और मर्ज होता है, पॉलिसीज़ का मूल्यांकन कैसे होता है, और डैशबोर्ड एजेंट गतिविधि को कैसे मॉनिटर करता है।

---

## अवलोकन

failproofai के दो स्वतंत्र सबसिस्टम हैं:

1. **हुक हैंडलर** - एक तेज़ CLI सबप्रोसेस जिसे Claude Code हर एजेंट टूल कॉल पर इनवोक करता है। पॉलिसीज़ का मूल्यांकन करता है और एक निर्णय लौटाता है।
2. **एजेंट मॉनिटर (डैशबोर्ड)** - एजेंट सेशन की निगरानी और पॉलिसीज़ को प्रबंधित करने के लिए एक Next.js वेब एप्लिकेशन।

दोनों सबसिस्टम `~/.failproofai/` और प्रोजेक्ट की `.failproofai/` डायरेक्टरी में कॉन्फिग फ़ाइलें साझा करते हैं, लेकिन वे अलग-अलग प्रक्रियाओं के रूप में चलते हैं और केवल फाइलसिस्टम के माध्यम से संचार करते हैं।

## हुक हैंडलर

### Claude Code के साथ एकीकरण

जब आप `failproofai policies --install` चलाते हैं, तो यह `~/.claude/settings.json` में इस तरह की प्रविष्टियां लिखता है:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code फिर प्रत्येक टूल कॉल से पहले `failproofai --hook PreToolUse` को एक सबप्रोसेस के रूप में इनवोक करता है, stdin पर एक JSON पेलोड पास करता है।

### पेलोड प्रारूप

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

`PostToolUse` इवेंट के लिए, पेलोड में टूल के आउटपुट के साथ `tool_result` भी होता है।

हैंडलर 1 MB stdin सीमा को लागू करता है। इस सीमा से अधिक पेलोड त्यागे जाते हैं और सभी पॉलिसीज़ निहित रूप से अनुमति देती हैं।

### प्रतिक्रिया प्रारूप

**अस्वीकार करें (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**अस्वीकार करें (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**निर्देश दें (किसी भी इवेंट को छोड़कर Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Stop इवेंट निर्देश दें:**
- एक्जिट कोड: `2`
- कारण stderr को लिखा जाता है (stdout को नहीं)

**अनुमति दें:**
- एक्जिट कोड: `0`
- खाली stdout

**संदेश के साथ अनुमति दें:**

`allow(message)` एक पॉलिसी को Claude को सूचनात्मक संदर्भ भेजने देता है भले ही ऑपरेशन की अनुमति दी जाए। हुक हैंडलर **stdout** को निम्न JSON लिखता है (कॉन्फिग फ़ाइल नहीं — यह हैंडलर की Claude Code को प्रतिक्रिया है, बिल्कुल डेनी और निर्देश प्रतिक्रियाओं की तरह):

```json
// हुक हैंडलर प्रक्रिया द्वारा stdout को लिखा जाता है
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- एक्जिट कोड: `0` (ऑपरेशन की अनुमति है)
- जब एकाधिक पॉलिसीज़ संदेश के साथ `allow` लौटाते हैं, तो उनके संदेशों को नई लाइनों के साथ एक एकल `additionalContext` स्ट्रिंग में जोड़ा जाता है
- अगर कोई पॉलिसी संदेश प्रदान नहीं करती, तो stdout खाली होता है (पहले की तरह)

### प्रोसेसिंग पाइपलाइन

`src/hooks/handler.ts` पूरी पाइपलाइन को लागू करता है:

```text
stdin JSON
  → पेलोड को पार्स करें (अधिकतम 1 MB)
  → सेशन मेटाडेटा निकालें (session_id, cwd, tool_name, tool_input, आदि)
  → readMergedHooksConfig(cwd)    ← प्रोजेक्ट + स्थानीय + वैश्विक कॉन्फिग को मर्ज करता है
  → सक्षम बिल्टिन पॉलिसीज़ को हल किए गए पैरामीटर के साथ पंजीकृत करें
  → customPoliciesPath से कस्टम पॉलिसीज़ लोड करें (यदि सेट है)
  → कस्टम पॉलिसीज़ को पॉलिसी रजिस्ट्री में पंजीकृत करें
  → सभी पॉलिसीज़ का मूल्यांकन करें (पहले बिल्टिन, फिर कस्टम)
      → पहला डेनी शॉर्ट-सर्किट करता है
      → निर्देश निर्णय जमा होते हैं
      → संदेशों की अनुमति जमा होती है
  → stdout को JSON निर्णय लिखें
  → इवेंट को ~/.failproofai/hook-activity.jsonl में संरक्षित करें
  → बाहर निकलें
```

बिना LLM कॉल के विशिष्ट पेलोड के लिए पूरी प्रक्रिया 100ms से कम समय में चलती है।

## कॉन्फिग लोडिंग

`src/hooks/hooks-config.ts` तीन-स्कोप कॉन्फिग लोडिंग को लागू करता है।

```text
[1] {cwd}/.failproofai/policies-config.json        ← प्रोजेक्ट  (उच्चतम प्राथमिकता)
[2] {cwd}/.failproofai/policies-config.local.json  ← स्थानीय
[3] ~/.failproofai/policies-config.json             ← वैश्विक   (सबसे कम प्राथमिकता)
```

मर्ज लॉजिक:
- `enabledPolicies` - तीनों फ़ाइलों में डीडुप्लिकेटेड यूनियन
- `policyParams` - प्रति-पॉलिसी कुंजी, पहली फ़ाइल जो इसे परिभाषित करती है पूरी तरह जीतती है
- `customPoliciesPath` - पहली फ़ाइल जो इसे परिभाषित करती है जीतती है
- `llm` - पहली फ़ाइल जो इसे परिभाषित करती है जीतती है

वेब डैशबोर्ड पढ़ने और लिखने के लिए `readHooksConfig()` (केवल वैश्विक) का उपयोग करता है, क्योंकि इसे प्रोजेक्ट cwd के साथ इनवोक नहीं किया जाता है।

## पॉलिसी मूल्यांकन

`src/hooks/policy-evaluator.ts` पॉलिसीज़ को क्रम में चलाता है।

प्रत्येक पॉलिसी के लिए:

1. पॉलिसी के `params` स्कीमा को देखें (यदि इसके पास एक है)।
2. मर्ज किए गए कॉन्फिग से `policyParams[policy.name]` को पढ़ें।
3. `ctx.params` बनाने के लिए स्कीमा डिफ़ॉल्ट पर यूजर-प्रदान किए गए मानों को मर्ज करें।
4. हल किए गए संदर्भ के साथ `policy.fn(ctx)` को कॉल करें।
5. यदि परिणाम `deny` है, तो तुरंत रुकें और वह निर्णय लौटाएं।
6. यदि परिणाम `instruct` है, तो संदेश को जमा करें और जारी रखें।
7. यदि परिणाम `allow` है, तो अगली पॉलिसी पर जाएं।

सभी पॉलिसीज़ चलने के बाद:
- यदि कोई `deny` लौटाया गया था, तो डेनी प्रतिक्रिया उत्सर्जित करें।
- यदि कोई `instruct` रिटर्न जमा किए गए थे, तो सभी संदेशों को जोड़कर एक एकल निर्देश प्रतिक्रिया उत्सर्जित करें।
- अन्यथा, अनुमति प्रतिक्रिया उत्सर्जित करें (खाली stdout, exit 0)।

## बिल्टिन पॉलिसीज़

`src/hooks/builtin-policies.ts` सभी 26 बिल्टिन पॉलिसीज़ को `BuiltinPolicyDefinition` ऑब्जेक्ट के रूप में परिभाषित करता है:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

जो पॉलिसीज़ `params` को स्वीकार करते हैं, वे प्रत्येक पैरामीटर के लिए प्रकार और डिफ़ॉल्ट के साथ एक `PolicyParamsSchema` घोषित करते हैं। पॉलिसी मूल्यांकन `fn` को कॉल करने से पहले हल किए गए मानों को `ctx.params` में इंजेक्ट करता है। पॉलिसी फ़ंक्शन `ctx.params` को पढ़ते हैं क्योंकि डिफ़ॉल्ट हमेशा पहले लागू होते हैं।

पॉलिसीज़ के अंदर पैटर्न मिलान, कच्ची स्ट्रिंग मिलान के बजाय पार्स किए गए कमांड टोकन (argv) का उपयोग करता है। यह shell ऑपरेटर इंजेक्शन के माध्यम से बाईपास को रोकता है (उदाहरण के लिए `sudo systemctl status *` के लिए एक पैटर्न को `; rm -rf /` जोड़कर बाईपास नहीं किया जा सकता)।

## कस्टम पॉलिसीज़

`src/hooks/custom-hooks-registry.ts` एक `globalThis`-समर्थित रजिस्ट्री को लागू करता है:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // परीक्षणों में उपयोग किया जाता है
```

`src/hooks/custom-hooks-loader.ts` यूजर की पॉलिसी फ़ाइल को लोड करता है:

1. कॉन्फिग से `customPoliciesPath` को पढ़ें; अनुपस्थित होने पर छोड़ दें।
2. पूर्ण पथ में हल करें; फ़ाइल की जांच करें कि मौजूद है।
3. सभी `from "failproofai"` आयातों को वास्तविक dist पथ पर फिर से लिखें ताकि `customPolicies` एक ही `globalThis` रजिस्ट्री में हल हो।
4. ESM संगतता सुनिश्चित करने के लिए संक्रमणीय स्थानीय आयातों को पुनरावर्ती रूप से फिर से लिखें।
5. अस्थायी `.mjs` फ़ाइलें लिखें और प्रविष्टि फ़ाइल को `import()` करें।
6. पंजीकृत हुक को पुनः प्राप्त करने के लिए `getCustomHooks()` को कॉल करें।
7. `finally` ब्लॉक में सभी अस्थायी फ़ाइलें साफ़ करें।

किसी भी त्रुटि पर (फ़ाइल नहीं मिली, सिंटैक्स त्रुटि, आयात विफल), त्रुटि को `~/.failproofai/hook.log` में लॉग किया जाता है और लोडर एक खाली सरणी लौटाता है। बिल्टिन पॉलिसीज़ प्रभावित नहीं होती हैं।

कस्टम पॉलिसीज़ सभी बिल्टिन पॉलिसीज़ के बाद मूल्यांकन किए जाते हैं। एक कस्टम पॉलिसी `deny` अभी भी आगे की कस्टम पॉलिसीज़ को शॉर्ट-सर्किट करता है (लेकिन सभी बिल्टिन पहले ही चल चुकी हैं)।

## गतिविधि लॉगिंग

प्रत्येक हुक इवेंट के बाद, हैंडलर `~/.failproofai/hook-activity.jsonl` में एक JSONL लाइन जोड़ता है:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

एक लाइन प्रति पॉलिसी जो एक गैर-अनुमति निर्णय लेता है। अनुमति निर्णय लॉग नहीं किए जाते हैं (फ़ाइल को छोटा रखने के लिए)।

## डैशबोर्ड आर्किटेक्चर

डैशबोर्ड एक **Next.js 16** एप्लिकेशन है जो App Router के साथ React Server Components और Server Actions का उपयोग करता है।

```text
app/
  layout.tsx                  ← रूट लेआउट (थीम, टेलीमेट्री, नेव)
  projects/page.tsx           ← सर्वर कॉम्पोनेंट: सभी Claude प्रोजेक्ट सूचीबद्ध करें
  project/[name]/page.tsx     ← सर्वर कॉम्पोनेंट: प्रोजेक्ट में सेशन सूचीबद्ध करें
  project/[name]/session/
    [sessionId]/page.tsx      ← सर्वर कॉम्पोनेंट: सेशन व्यूअर रेंडर करें
  policies/page.tsx           ← क्लाइंट कॉम्पोनेंट: पॉलिसी प्रबंधन + गतिविधि लॉग
  actions/
    get-hooks-config.ts       ← कॉन्फिग + पॉलिसी सूची पढ़ें
    update-hooks-config.ts    ← पॉलिसी चालू/बंद करें
    update-policy-params.ts   ← पॉलिसी पैरामीटर अपडेट करें
    get-hook-activity.ts      ← गतिविधि लॉग को पृष्ठांकित करें/खोजें
    install-hooks-web.ts      ← ब्राउज़र से हुक इंस्टॉल/हटाएं
  api/
    download/[project]/[session]/route.ts   ← प्रति-CLI सेशन निर्यात (JSONL या JSON)
```

**डेटा प्रवाह:**

- पृष्ठ कॉम्पोनेंट `lib/projects.ts` और `lib/log-entries.ts` को कॉल करके प्रोजेक्ट/सेशन डेटा को सीधे फाइलसिस्टम से पढ़ते हैं (पढ़ने के लिए कोई API परत नहीं)।
- पॉलिसीज़ पृष्ठ सभी मिउटेशन के लिए Server Actions का उपयोग करता है (टॉगल, पैरामीटर अपडेट, इंस्टॉल/हटाएं)।
- सेशन दर्शक Claude के JSONL ट्रांसक्रिप्ट प्रारूप को पार्स करता है और संदेशों और टूल कॉल की समयरेखा रेंडर करता है।

**मुख्य डिज़ाइन निर्णय:**

- कोई डेटाबेस नहीं - सभी निरंतर स्थिति सादी फ़ाइलों में है (`~/.failproofai/`, `~/.claude/projects/`)।
- म्यूटेशन के लिए Server Actions - CRUD ऑपरेशन के लिए कोई REST API आवश्यक नहीं।
- पढ़ने वाले पृष्ठों के लिए React Server Components - तेज़ प्रारंभिक लोड, डेटा फेचिंग के लिए कोई क्लाइंट बंडल नहीं।
- क्लाइंट कॉम्पोनेंट केवल जहां इंटरएक्टिविटी की आवश्यकता है (पॉलिसी टॉगल, गतिविधि खोज, लॉग दर्शक)।
