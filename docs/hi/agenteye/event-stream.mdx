जिस पल आपका agent कुछ करता है, आप उसे देखते हैं। Event Stream प्रोडक्शन में हर agent का आपका लाइव पल्स है: कोई इंतज़ार नहीं, कोई लॉग grep करने की जरूरत नहीं, यह अनुमान लगाने की कोई आवश्यकता नहीं कि अभी क्या हुआ।

![लाइव Event Stream: रंग-कोडित event rows रीयल टाइम में tail हो रहे हैं, environment, agent, session, event type, और फ्री टेक्स्ट द्वारा फ़िल्टर योग्य](/agenteye/images/events-stream.png)

*आपके org के हर agent का हर event, सबसे नया पहले, जैसे ही यह होता है अपडेट हो रहा है।*

## हर agent पर आपका लाइव पल्स

जब कोई agent एक run शुरू करता है, कोई मॉडल कॉल करता है, कोई tool फायर करता है, कोई hook चलाता है, या कोई error को hit करता है, तो row उस पल stream के शीर्ष पर दिखाई देता है जब यह होता है। यह आपके पूरे organization के हर agent के सभी events को tail करता है, सबसे नया पहले, तो आप हमेशा एक वर्तमान चित्र रखते हैं एक पुरानी जानकारी की जगह।

इसका मतलब है कि किसी box पर log फ़ाइलों को tail करने की कोई जरूरत नहीं, मशीनों के बीच grep करने की कोई जरूरत नहीं, timestamps को हाथ से एक साथ जोड़ने की कोई जरूरत नहीं। आप एक पेज खोलते हैं और आप पहले से ही प्रोडक्शन को देख रहे हैं।

Rows प्रकार के आधार पर रंग-कोडित हैं, इसलिए आप हर लाइन को parse करने की जगह एक नज़र में stream को पढ़ सकते हैं। एक नज़र में, प्रत्येक row आपको दिखाता है:

- **इसका प्रकार**, रंग-कोडित: `agent_start`, `model_response`, `tool_use`, `hook_completed`, `error`, और बहुत कुछ।
- **क्या हुआ**, इसका एक-पंक्ति सारांश, इसलिए आपको शायद ही कभी सिर्फ gist पाने के लिए कुछ खोलने की जरूरत होती है।
- **Token counts** इस step के लिए।
- **एक context-window fill badge** जहां यह लागू होता है, इसलिए prompt growth और एक आने वाला compaction दिखाई देते हैं इससे पहले कि वे काटें।

इसे लाइव देखने का मतलब है कि आप एक bad deploy, एक runaway loop, या errors का एक burst पकड़ते हैं जैसे ही यह होता है, कल के log review में नहीं।

## वह एक run खोजें जो महत्वपूर्ण है

जब कुछ गलत दिखता है, तो आप firehose नहीं चाहते। आप वह single run चाहते हैं जो टूटा। Stream तेज़ी से फ़िल्टर करता है: environment द्वारा, agent द्वारा, session द्वारा, event type द्वारा, या फ्री टेक्स्ट द्वारा।

एक run को उसके पहले event से लेकर आखिरी तक follow करने के लिए session id या agent id द्वारा फ़िल्टर करें। एक गतिविधि के एक ही प्रकार को अलग करने के लिए event type द्वारा फ़िल्टर करें, उदाहरण के लिए पूरे org के हर `error` को एक view में। "सबकुछ, हर जगह" से "यह agent, prod में, error कर रहा है" तक कुछ clicks में filters को stack करें, फिर आपको जो मिलता है उस पर कार्य करें।

फ्री-टेक्स्ट search सीधे एक संदेश, एक tool का नाम, या एक id में कटता है जो आपके पास पहले से है, इसलिए एक customer report कुछ सेकंड में exact run में बदल जाता है।

## इसे कहां खोजें

Event Stream आपका org home है। साइन इन करें और यह पहली सतह है जहां आप उतरते हैं, `/<org-slug>/` पर, इसलिए triage आपके आने के दूसरे ही पल शुरू होता है।

इसके पीछे, आपके agents SDK के माध्यम से events emit करते हैं, collector उन्हें आपके Failproof AI Observability server को भेजता है, और stream उन्हें tail करता है जैसे ही वे उस infrastructure में पहुंचते हैं जिसे आप नियंत्रित करते हैं। जब आप raw trail की जगह rolled-up view चाहते हैं, तो प्रत्येक run के events एक single row में Sessions पर collapse हो जाते हैं, एक click दूर।

यह raw source of truth है जिस पर हर दूसरी observe surface बनता है, इसलिए जब एक नंबर कहीं और गलत दिखता है, तो stream वह जगह है जहां आप confirm करते हैं कि वास्तव में क्या हुआ।
