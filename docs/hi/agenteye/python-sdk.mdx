अपने AI agents ने production में ठीक क्या किया, यह देखें: हर agent run, tool call, model request, hook, और human intervention। FailproofAI Observability Python SDK आपके agent code के अंदर से उस trail को रिकॉर्ड करता है ताकि आप debug, audit, और evaluate कर सकें कि क्या हुआ। जब भी आप चाहते हैं कि FailproofAI Observability आपके agents को observe करे, तब इसका उपयोग करें।

SDK के अंदर, यह structured events को local JSONL files में लिखता है, और collector daemon उन्हें pick करके platform पर automatically ship करता है। आप इन files को स्वयं manage नहीं करते।

## Quick Start

```python
import agenteye

agenteye.configure(environment="production")

agenteye.event.agent_start(session_id="run-001", agent_id="planner", goal="answer user query")

agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",
    input={"query": "latest AI research"},
)

agenteye.event.tool_result(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",
    output={"results": ["..."]},
)

agenteye.event.agent_end(session_id="run-001", agent_id="planner", outcome="success")
```

### एक real call को instrument करना

व्यवहार में आप अपने existing agent code को wrap करते हैं। एक model call को `model_request` से पहले और `model_response` के बाद bracket करें, ताकि दोनों events real request को span करें और FailproofAI Observability उन्हें pair कर सके:

```python
import anthropic
import agenteye

agenteye.configure(environment="production")
client = anthropic.Anthropic()

messages = [{"role": "user", "content": "Summarise today's incidents."}]

agenteye.event.model_request(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6",
    messages=messages,
)

reply = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    messages=messages,
)

agenteye.event.model_response(
    session_id="run-001",
    agent_id="planner",
    model=reply.model,
    stop_reason=reply.stop_reason,
    input_tokens=reply.usage.input_tokens,
    output_tokens=reply.usage.output_tokens,
    content=[block.model_dump() for block in reply.content],
)
```

Tool calls को भी same तरीके से `tool_use` और `tool_result` से wrap करें, pair के across एक ही `tool_call_id` को reuse करते हुए।

यहाँ देखें कि ये events dashboard पर कैसे दिखते हैं, type के हिसाब से colour-coded और environment, agent, और session द्वारा filterable:

## configure()

```python
agenteye.configure(
    base_dir=None,           # Path | str | None. Default: $AGENTEYE_HOME या ~/.agenteye
    flush_interval=0.5,      # float, seconds between flush cycles
    environment=None,        # str | None. Deployment environment label
)
```

किसी भी `event.*` call से पहले एक बार call करें। Safe to omit; defaults out of the box काम करते हैं। सभी arguments keyword-only हैं; उन्हें ऊपर दिखाए अनुसार नाम से pass करें।

जब `base_dir` `None` है (default), SDK `$AGENTEYE_HOME` को पढ़ता है यदि set हो,
अन्यथा `~/.agenteye` पर fallback करता है। यह collector के अपने resolution से match करता है,
इसलिए एक single `AGENTEYE_HOME` env var SDK और collector दोनों के लिए shared event spool को configure करता है।

## Environment

हर event को एक deployment environment (`production`, `staging`, `qa`, `canary`, आदि) के साथ label करें। इसे एक बार set करें; SDK इसे हर event के लिए automatically attach करता है।

**Option 1: `configure()` के माध्यम से:**

```python
agenteye.configure(environment="production")
```

**Option 2: environment variable के माध्यम से:**

```bash
export AGENTEYE_ENVIRONMENT=production
```

**Priority:** `configure(environment=...)` environment variable पर win करता है। यदि कोई भी set नहीं है, तो `"dev"` पर default करता है।

Environment value dashboard में एक first-class filter के रूप में दिखाई देता है और fast queries के लिए server पर store किया जाता है।

> **Warning:** Environment values में literal `,` comma नहीं होना चाहिए। Dashboard filters wire पर comma-separated multi-select का उपयोग करते हैं (`?environment=prod,staging`), इसलिए `prod,blue` नाम वाली environment दो values में split हो जाएगी। Comma-containing environments वाले events ingest time पर reject कर दिए जाते हैं।

## Data और privacy

SDK केवल उन fields को record करता है जो आप explicitly pass करते हैं। Prompts, messages, tool inputs और outputs, और model content capture किए जाते हैं केवल इसलिए कि आप उन्हें एक `event.*` call को hand करते हैं। कुछ भी आपकी process से read नहीं किया जाता या implicitly capture नहीं किया जाता। कोई भी field जो आप unset छोड़ते हैं, event से पूरी तरह omit किया जाता है; यह disk पर नहीं लिखा जाता।

यह redaction को आपकी choice और आपकी responsibility बनाता है। यदि एक prompt या tool payload में PII या secrets हैं जो आप store नहीं करना चाहते, तो इसे strip या mask करें इससे पहले कि आप इसे event method को pass करें।

## Event Reference

अधिकांश events start/end pairs में आते हैं जो एक correlation ID share करते हैं: `tool_use` और `tool_result` एक `tool_call_id` share करते हैं, `hook_triggered` और `hook_completed` एक `hook_id` share करते हैं, और `human_wait` और `human_input` एक `input_id` share करते हैं। Start event emit करें, काम करें, फिर end event emit करें same ID के साथ। FailproofAI Observability pair को match करता है और आपके लिए `duration_ms` compute करता है, इसलिए आप `duration_ms` को स्वयं कभी pass नहीं करते।

![A session's git-style execution graph beside its event timeline, reconstructed from the paired events, with the tool/model/hook breakdown panel](/agenteye/images/session-detail.png)

सभी event methods को इन दो fields की आवश्यकता है:

| Field | Type | Description |
|---|---|---|
| `session_id` | `str` | Top-level agent run को identify करता है |
| `agent_id` | `str` | Identify करता है कि session के अंदर कौन सा agent ने event emit किया |

सभी methods arbitrary `**kwargs` को भी accept करते हैं custom metadata के लिए (देखें [Custom Fields](#custom-fields))।

---

### `event.agent_start()`

Emitted जब एक agent काम शुरू करता है।

```python
agenteye.event.agent_start(
    session_id="run-001",
    agent_id="planner",
    goal="answer user query",   # str | None
    parent_id=None,             # str | None - nested agents के लिए parent agent_id
)
```

---

### `event.agent_end()`

Emitted जब एक agent काम finish करता है।

```python
agenteye.event.agent_end(
    session_id="run-001",
    agent_id="planner",
    outcome="success",          # str | None
    summary="Answered query",   # str | None
)
```

---

### `event.tool_use()`

Emitted जब एक agent एक tool invoke करता है। `tool_result` के साथ pair करें; SDK auto-computes `duration_ms`।

```python
agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",     # str, required
    tool_call_id="toolu_01",    # str, required - matching tool_result के लिए correlation key
    input={"query": "..."},     # dict | None
)
```

---

### `event.tool_result()`

Emitted जब एक tool return करता है। `tool_call_id` के माध्यम से `tool_use` से correlate करता है।

```python
agenteye.event.tool_result(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",        # prior tool_use से match करना चाहिए
    output={"results": ["..."]},    # Any | None
    error=None,                     # str | None - set करें यदि tool raised हो
    # duration_ms automatically computed है - इसे pass न करें
)
```

---

### `event.model_request()`

Emitted एक LLM को prompt भेजने से ठीक पहले।

```python
agenteye.event.model_request(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6", # str | None - कोई भी provider/model string; validated नहीं
    messages=[                   # list[dict] | None - conversation turns
        {"role": "user", "content": "..."},
    ],
    system="You are helpful.",   # Any | None - str या content blocks की list
    tools=[                      # list[dict] | None - tool schemas offered to the model
        {"name": "search", "input_schema": {"type": "object"}},
    ],
)
```

`messages` entries plain string `content` या Anthropic-style list-of-blocks `content` को accept करते हैं। Sampling params (`temperature`, `max_tokens`, आदि) को extra kwargs के रूप में pass किया जा सकता है।

---

### `event.model_response()`

Emitted जब LLM एक response return करता है।

```python
agenteye.event.model_response(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6", # str | None - कोई भी provider/model string; validated नहीं
    stop_reason="end_turn",     # str | None
    input_tokens=1024,          # int | None
    output_tokens=256,          # int | None
    content=[                    # Any | None - str, या content blocks की list
        {"type": "text", "text": "..."},
    ],
    role="assistant",            # str | None
)
```

`content` plain string (generic providers) या Anthropic-style content blocks की list को accept करता है। Tool calls `content` के अंदर `{"type": "tool_use", ...}` blocks के रूप में रहते हैं, कोई अलग `tool_calls` field नहीं।

---

### `event.hook_triggered()`

Emitted जब एक hook fires। `hook_completed` के साथ pair करें; SDK auto-computes `duration_ms`।

```python
agenteye.event.hook_triggered(
    session_id="run-001",
    agent_id="planner",
    hook_name="pre_tool_use",   # str, required
    hook_id="hook-abc",         # str, required - correlation key
    trigger_event="tool_use",   # str | None
    input={"tool": "search"},   # Any | None
)
```

---

### `event.hook_completed()`

Emitted जब एक hook finish हो जाता है। `hook_id` के माध्यम से `hook_triggered` से correlate करता है।

```python
agenteye.event.hook_completed(
    session_id="run-001",
    agent_id="planner",
    hook_name="pre_tool_use",
    hook_id="hook-abc",         # prior hook_triggered से match करना चाहिए
    outcome="allow",            # str | None
    output=None,                # Any | None
    error=None,                 # str | None
    # duration_ms automatically computed है - इसे pass न करें
)
```

---

### `event.error()`

Emitted जब एक unhandled error occur करता है।

```python
agenteye.event.error(
    session_id="run-001",
    agent_id="planner",
    error_type="TimeoutError",  # str, required
    message="timed out",        # str, required
    traceback="Traceback...",   # str | None
)
```

## Human-in-the-Loop Events

Human-in-the-loop events आपको oversight देते हैं उन moments पर जहाँ एक person agent के execution में step करता है (approval के लिए wait करना, input provide करना, pause करना, या agent को stop करना)। ये आपको measure करने देते हैं कि humans को respond करने में कितना समय लगता है (SDK auto-computes `duration_ms` paired events पर), audit करते हैं कि किसने एक agent को pause या interrupt किया, और approval और oversight workflows build करते हैं जो dashboard में surface होती हैं।

### `event.human_wait()`

Emitted जब agent execution को pause करता है एक human को input provide करने के लिए wait करने के लिए। `human_input` के साथ pair करें; SDK auto-computes `duration_ms` (human को respond करने में कितना समय लगा)।

```python
agenteye.event.human_wait(
    session_id="run-001",
    agent_id="planner",
    input_id="inp-abc",                          # str, required - matching human_input के लिए correlation key
    prompt="Do you approve this action?",        # str | None - human को दिखाया गया question
    options=["approve", "reject", "defer"],      # list[str] | None - human को presented choices
    reason="approval_required",                  # str | None - agent क्यों wait कर रहा है
)
```

### `event.human_input()`

Emitted जब एक human input provide करता है और agent resume होता है। `input_id` के माध्यम से `human_wait` से correlate करता है। `duration_ms` auto-computed है और caller द्वारा pass नहीं किया जाना चाहिए।

```python
agenteye.event.human_input(
    session_id="run-001",
    agent_id="planner",
    input_id="inp-abc",    # str, required - prior human_wait से match करना चाहिए
    response="approve",    # str | None - human का answer (free text या selected option)
    # duration_ms automatically computed है - इसे pass न करें
)
```

### `event.human_pause()`

Emitted जब एक human actively agent को pause करता है (जैसे dashboard control के माध्यम से)। Agent suspend है लेकिन terminated नहीं।

```python
agenteye.event.human_pause(
    session_id="run-001",
    agent_id="planner",
    reason="user_requested",  # str | None
    user_id="usr_42",         # str | None - किसने agent को pause किया
)
```

### `event.human_interrupt()`

Emitted जब एक human actively agent को mid-execution में stop करता है। `human_pause` के विपरीत, agent का काम terminate होता है rather than suspended।

```python
agenteye.event.human_interrupt(
    session_id="run-001",
    agent_id="planner",
    reason="output_incorrect",       # str | None
    user_id="usr_42",                # str | None - किसने agent को interrupt किया
    at_step="tool_use:web_search",   # str | None - agent क्या कर रहा था जब stopped हुआ
)
```

## Custom Fields

कोई भी extra keyword arguments standard fields के बाद event में appended होते हैं:

```python
agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="db_query",
    tool_call_id="toolu_02",
    tenant_id="acme",           # custom field
    region="us-east-1",         # custom field
)
```

`timestamp`, `type`, और `environment` reserved हैं और `ValueError` raise करते हैं (`Reserved field names cannot be used as custom fields: [...]`) यदि custom fields के रूप में pass किए जाएं। `session_id` और `agent_id` हर event method पर required parameters हैं और दूसरी बार supply नहीं किए जा सकते; Python `TypeError` raise करता है यदि आप करते हैं। Instead `configure(environment=...)` के साथ environment set करें (या `AGENTEYE_ENVIRONMENT` variable)।

## Events कैसे लिखे जाते हैं

Events को in-process buffer किया जाता है और हर `flush_interval` सेकंड पर (default 500 ms) disk पर flush किया जाता है। हर flush एक JSONL file लिखता है:

```text
~/.agenteye/events/event-2026-04-01T12-00-00-000Z.jsonl
```

Collector इस directory को watch करता है और files को automatically upload करता है। आपको इन files को directly manage करने की आवश्यकता नहीं है।

हर file atomically लिखा जाता है: SDK एक temporary file पर लिखता है और फिर इसे rename करता है, इसलिए collector कभी एक half-written file नहीं देखता। एक final flush भी run होता है जब आपकी process exit होती है, इसलिए last interval में buffered events नहीं खोते हैं। यदि collector offline है, events बस disk पर files के रूप में accumulate होते हैं और एक बार यह वापस आता है तो ship होते हैं।
