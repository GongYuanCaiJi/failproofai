देखें कि आपके models, tools, या hooks कब धीमे हो जाते हैं या बिल बढ़ता है, और tail-latency spike को अपने users महसूस करने से पहले ही पकड़ लें। तीन dedicated pages raw timings को p50, p95, और p99 में बदल देते हैं जिन्हें आप एक नज़र में पढ़ सकते हैं।

![The Models page showing a latency heat-map, a percentile band, and per-model token, cost, and context-window figures](/agenteye/images/models.png)
*Models page: एक latency heat-map, एक percentile band, और per-model tokens, अनुमानित cost, और context-window fill।*

## औसत को अपनी सबसे ख़राब runs को छिपाने न दें

औसत latency संख्या आरामदायक होती है और बेकार है: यह उन पचास में से एक call को smooth कर देती है जो रुक जाती है और सुबह 2 बजे आपके on-call को alert भेजती है। Models, Tools, और Hooks pages ऐसा नहीं करते। प्रत्येक एक ही आकार साझा करता है, इसलिए आप इसे एक बार सीखते हैं:

- एक **24-bin sparkline** trend के लिए एक नज़र में: क्या यह बदतर हो रहा है?
- एक **vitals strip** p50, p95, और p99 latency के साथ, ताकि typical run और tail एक दूसरे के बगल में हों।
- एक **latency heat-map**, 24 time bins by latency buckets, जो दिखाता है कि *कब* slow calls clustered हुई थीं।
- एक **percentile band**: एक p50 line जिसमें p25 to p75 और p10 to p90 shaded ribbons और p99 dots हों, ताकि spread visible रहे averaged away के बजाय।

एक shared hover crosshair heat-map और band को link करता है, इसलिए एक tail spike समय में दोनों में line हो जाता है, एक single mean line के पीछे छिपने के बजाय। तीनों pages को अपने dashboard के **observe** section में खोजें, प्रत्येक अपने organization को scope किया हुआ और date range, environment, agent, और session के आधार पर filterable।

## Models: देखें कि प्रत्येक model आपको बिल्कुल कितना खर्च करता है

Models page (ऊपर दिखाया गया) दो सवालों का जवाब देता है जो bill हमेशा उठाता है: कौन सा model, और कितना। shared latency view के ऊपर, यह **per-model token consumption**, **अनुमानित cost**, और **context-window fill** जोड़ता है, ताकि runaway prompt growth और एक impending compaction visible हों surprise से पहले।

Failproof AI Observability common model IDs को automatically recognize करता है। अगर कोई window गलत दिख रहा है, या आप अपना private model चलाते हैं, तो इसे **Settings** में, **model context windows** के अंतर्गत correct या add करें, और fill readouts follow करेंगे।

## Tools: slow को broken से अलग करें

एक tool call slow हो सकती है, या यह quietly fail हो सकती है, और आप चाहते हैं कि यह जान लें कि कौन सी है सेकंड में, logs के माध्यम से dig करने के बाद नहीं।

![The Tools page showing the shared latency heat-map and percentile band beside a success and failure breakdown and a tool-distribution bar](/agenteye/images/tools.png)
*Tools page: वही heat-map और percentile band, साथ ही एक success और failure breakdown और एक tool-distribution bar।*

Shared latency view के साथ-साथ, Tools page एक **success and failure breakdown** और एक **tool-distribution bar** जोड़ता है, ताकि आप एक नज़र में देखें कि आप कौन से tools पर सबसे ज्यादा भरोसा करते हैं और कौन से आपके error budget को खा रहे हैं।

## Hooks: बिल्कुल exact hook और trigger को pinpoint करें

जब एक lifecycle hook एक run को drag करता है, तो "hooks are slow" कुछ ऐसा नहीं है जिस पर आप act कर सकते हैं। Hooks page आपको उस तक ले जाता है जो मायने रखता है।

![The Hooks page showing latency broken down by hook name and trigger event over the shared heat-map and percentile band](/agenteye/images/hooks.png)
*Hooks page: latency को hook name और trigger event के आधार पर break किया गया है।*

उसी latency heat-map और percentile band पर, Hooks page activity को **hook name** और **trigger event** के आधार पर break down करता है, ताकि आप उस single hook और single event पर आएं जिसे attention की जरूरत है।
