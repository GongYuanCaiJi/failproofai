*每项审计都是一个定期任务，它会挖掘您的会话数据并输出按优先级排列、有据可查的改进建议。*

## 不再凭猜测决定下一步该修什么

告警能捕捉您已知晓并主动关注的问题，而审计则能发现您尚未意识到的问题。按照您设定的频率，审计会读取您所有的 Agent 会话，主动寻找值得修复的模式——让您把时间花在处理发现的问题上，而不是翻阅日志、碰运气地自己去发现它们。

每次运行都会针对生产环境中真正导致 Agent 失效的故障模式进行排查：

- **错误聚类**：由同一根本原因重复出现的故障。
- **与基线的偏差**：行为悄然偏离已知正常窗口的情况。
- **转录中的目标失败**：技术上已完成、但实际上未完成任务的运行。
- **工具误用**：使用了错误的工具、传入了错误的参数，或导致调用耗尽的循环。
- **质量与成本的权衡**：您为某些输出付出了过高代价，而本可以更低成本获得。
- **覆盖盲区**：没有任何评估或告警在监控的行为。

您只需通过一个**灵敏度**设置（低、中或高）来决定审计的深度，这样嘈杂的预发布 Agent 和严格管控的生产 Agent 都可以分别调整到适合自己的信号级别。

## 每项建议都有据可查

您无需对任何发现无条件信任。每项建议都会引用其来源的具体会话以及用于发现问题的 SQL，您只需点击一下即可查看证据、确认问题，而无需自行逆向推断。

这也是确保审计诚实可信的机制。服务器会验证每个被引用的会话确实存在，并**丢弃任何证据站不住脚的建议**，因此审计只会调查，绝不会凭空捏造。出现在您清单上的内容都是真实存在、可复现的，并按重要程度排序，影响最大的问题排在最前面。

## 将修复转化为防护措施

修复一个问题只完成了一半。另一半是确保它不会悄然卷土重来。每项发现都附带一个**一键生成复发告警的快捷方式**，预填了合理的初始触发条件供您调整。关闭该发现、启用告警，下次该模式再次出现时，您会收到通知，而不是在未来的审计中重新发现它。

## 在哪里找到它

审计功能位于仪表板的 **`/<org-slug>/audits`**（侧边栏 → *分析* → *审计*）。查看运行记录和发现结果需要 **`audits:read`** 权限；创建、编辑和处理审计需要 **`audits:write`** 权限。设置审计的范围和频率，然后点击**立即运行**即可立即获取结果，无需等待下一次定时执行。
