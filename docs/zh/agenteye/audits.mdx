*每项审计都是一个定期任务，会挖掘您的会话数据并生成按优先级排列、有据可查的建议报告。*

## 不再猜测下一步该修复什么

告警能捕获您已知需要关注的问题，而审计能捕获您尚未意识到的问题。按照您设定的计划，审计会读取所有 Agent 会话，主动寻找值得修复的模式，让您将时间花在落实发现上，而不是翻查日志碰运气。

单次运行会针对以下生产环境中真正会破坏 Agent 的故障模式展开排查：

- **错误聚类**：在共同根因下反复出现的同类故障。
- **与基线的偏差**：行为悄然偏离已知正常状态的情况。
- **转录中的目标失败**：技术上已完成但实际未完成任务的运行。
- **工具误用**：使用了错误的工具、传入了错误的参数，或浪费调用次数的循环。
- **质量与成本的权衡**：您为本可以用更低成本获得的输出支付了过高费用的场景。
- **覆盖盲区**：没有任何评估或告警在监控的行为。

您可以通过单一的**灵敏度**设置（低、中或高）来控制排查力度，从而让嘈杂的测试环境 Agent 和严格管控的生产环境 Agent 各自调整到所需的信噪比。

## 每条建议均附有佐证

您无需对任何发现盲目信任。每条建议都会引用其所来源的具体会话以及用于发现问题的 SQL，让您只需点击一下即可查阅证据、确认问题，而无需费力还原一个未经证实的论断。

这也是审计保持可信度的关键所在。服务器会核验每条被引用的会话确实存在，并**丢弃任何证据站不住脚的建议**，因此审计的作用是调查，而非凭空捏造。最终出现在您清单上的内容都是真实存在、可复现的，并按重要程度排序，影响最大的问题排在最前面。

## 将修复转化为防护措施

修复一个问题只是成功的一半，另一半是确保它不会悄悄卷土重来。每条发现都附带一个**一键创建复发告警的快捷方式**，预填了合理的初始触发条件供您调整。关闭发现、启用告警，下次该模式再次出现时，您将收到通知，而不是在未来的审计中重新发现它。

## 在哪里找到它

审计功能位于仪表板的 **`/<org-slug>/audits`** 路径下（侧边栏 → *分析* → *审计*）。查看运行记录和发现结果需要 **`audits:read`** 权限；创建、编辑和处理审计则需要 **`audits:write`** 权限。设置审计的范围和频率，随时点击 **立即运行** 即可获取结果，无需等待下次定时运行。
