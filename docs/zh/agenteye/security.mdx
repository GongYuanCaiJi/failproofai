Failproof AI Observability 被设计为紧密部署在您的生产环境 Agent 旁边，这意味着它能够访问您的提示词、工具输入和输出内容。本页说明它如何保持数据隔离、受控，并始终掌握在您手中。如果您正在对 Failproof AI Observability 进行安全审查评估，请从这里开始。

---

## 您的数据始终保留在您的环境中

Failproof AI Observability 采用自托管方式部署。事件、提示词、模型响应和分析数据都存储在您自己的数据库中、您自己的环境里。不会将任何数据发送至第三方 SaaS 平台存储，您的数据始终保留在您自己的云账户中。

---

## 租户隔离

一个 Failproof AI Observability 实例可以托管多个组织，每个组织在存储层面相互隔离——这由数据库强制执行，而不仅仅是界面层面的限制：

- 组织的运营数据（用户、密钥、仪表盘、已保存查询）均限定在该组织范围内，跨组织读取由数据库本身阻断。
- 每个摄入的事件都会打上所属组织的标记，因此一个组织的事件永远无法被另一个组织读取。

每个仪表盘路由都限定在组织标识符下（`/<org-slug>/…`）。

---

## 登录

Failproof AI Observability 使用无密码、基于邮件的登录方式。不存在可被钓鱼或泄露的密码。用户请求一次性验证码（或一键魔法链接），通过邮件发送给用户，并在短时间内过期。登录受**白名单**限制：只有您允许的邮箱地址（或域名）才能通过身份验证。

## 使用 API 密钥进行范围化访问

每个客户端都使用携带精细化最小权限的 API 密钥进行身份验证。采集器只需要 `events:add`；仪表盘或助手密钥可以设为只读；破坏性操作（删除、重新生成）是您可以选择性授予的独立权限。

![API 密钥页面：每个密钥的权限授予，按读取、写入和破坏性范围以颜色区分](/agenteye/images/api-keys.png)

保留管理员引导密钥用于初始设置，其他所有用途均签发最小权限密钥。请参阅 [API 密钥](/zh/agenteye/api-keys)。

---

## 只读、需审批的 AI 助手

仪表盘内置的 [AI 助手](/zh/agenteye/assistant) 可针对您的数据回答问题，但在设计上受到严格约束：

- **默认只读**：其 SQL 查询通过守卫过滤，仅允许 `SELECT`/`WITH` 查询、单语句执行，并设有行数上限。
- 它创建的任何内容（已保存的查询、仪表盘）均**需经过审批**：每次写入操作发生前，您都需要审查并确认。
- 它**永远无法执行删除操作**。

因此，团队成员可以询问"本周哪些 Agent 出错最多？"并根据答案采取行动，而助手本身无法自行修改或删除您的数据。

---

## 传输安全

所有流量均通过 HTTPS 传输。您使用自己的证书终止 TLS，因此采集器到服务器以及浏览器到服务器的流量在传输过程中均经过加密。
