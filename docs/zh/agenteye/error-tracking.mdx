## 所有错误，自动汇聚

Agent 出现故障时，你不应该还要盯着实时事件流，担心红色条目在滚动消失前被遗漏。**Errors** 页面会自动完成收集工作。它将仪表板中所有标红的内容汇总到一个统一的分诊界面，让你第一眼就能看清什么在报错，而不是到处找线索。

它捕获的远不止显而易见的错误。除了明确的 `error` 事件，FailproofAI Observability 还会浮出那些沉默的故障：任何 payload 中携带失败信息的 `tool_result`、`hook_completed` 或 `agent_end` 都会显示在这里。某个工具返回了错误，或者某个 hook 异常退出，再也不会因为没有抛出明显异常而悄悄溜走。

页面顶部的直方图展示了错误随时间的分布情况。扫一眼便能判断这是持续的低频背景噪声，还是几分钟前突然爆发的尖峰，让你立刻决定是否需要放下手头的事情优先处理。

与所有观测界面一样，Errors 页面按组织维度划定范围，并支持按日期范围、环境、Agent 和 Session 进行筛选。这意味着你可以从全量列表出发，精准缩小到真正关心的某个 Agent 或某个环境。

## 一个事件，而非上百条重复行

一个依赖出现故障，可能每分钟触发同一条错误数百次。如果不加处理，呈现出来的就是一堵几乎一模一样的行墙，把你真正需要看到的信息彻底淹没。

FailproofAI Observability 会将同一 Session 和相同错误类型的重复故障合并为单行。密集爆发的错误只读作一个事件。你统计的是问题的数量，而不是日志行数——真正重要的信号始终置顶，不会被自身的量所淹没。

## 从"某处报红"直达精确事件

点击任意一行，即可直接进入该次运行的 Session，并定位到触发错误的那个确切事件。无需复制 Session ID，无需上下滚动寻找出问题的时刻：你直接抵达现场，完整的执行图一览无余，让你清晰看到 Agent 在崩溃前的每一步动作。

如果你拥有 `alerts:write` 权限，每行还会显示一个 **+ alert** 按钮。点击后，Observability 会打开一条新的告警规则，并预先填入能够捕获同类故障的条件。你刚刚处理过的事件，下次再发生时会主动通知你，而不是再次打你个措手不及。

**访问路径：** **Errors** 页面位于仪表板的观测区域，地址为 `/<org-slug>/errors`。
