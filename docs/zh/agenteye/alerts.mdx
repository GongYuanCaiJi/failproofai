在问题越过警戒线的第一时间收到通知，通过团队已在使用的渠道，而不是等到客户投诉才知道。规则只需配置一次，Failproof AI Observability 会按计划检查，并通过邮件、Slack、Webhook 或仪表板内通知提醒你。

![告警页面：一组告警规则卡片，每张卡片显示触发条件、评估窗口、通知渠道以及信息、警告或严重级别的徽章](/agenteye/images/alerts.png)
*一览所有告警规则：监控内容、检查频率、通知渠道及紧急程度。*

## 在用户发现之前掌握问题

不必死盯着仪表板等着捕捉回归。对于任何无人值守时也希望第一时间知道的信号，都可以设置告警，通知将直接送达你常用的地方：

- **邮件**，发送给所有需要了解的人。
- **Slack**，富文本消息，附带可直接跳转至事件详情的按钮。
- **Webhook**，向 PagerDuty、Opsgenie 或你自己的接入点发送 JSON POST 请求，可选签名验证以便接收方确认来源可信。
- **仪表板内通知**，设计上较为低调，适合在调试规则时使用，暂时不想打扰任何人。

可以为一条规则同时配置多种通知方式，规则的严重级别（信息、警告或严重）会随通知一并传递，紧急通知自然一目了然。

## 用表单配置规则，无需手写 JSON

你在表单中描述什么叫"出了问题"，Failproof AI Observability 会自动生成底层规则。JSON 规格不过是表单在底层生成的内容，了解它有助于读懂规则，但你几乎不需要手动编写。

![新建告警表单：名称与描述、启用开关，以及触发条件选择器（包含指标阈值、自定义 SQL、评估分数、复合评估和单事件条件）](/agenteye/images/alert-new.png)
*选择触发类型，表单自动切换相应字段；点击保存即写入规则。*

常规流程非常快捷：输入名称，选择**触发条件**（监控什么），设置**阈值和窗口**（多严重、持续多久），关联至少一个**通知渠道**，然后**保存**并点击**测试**，发送一条模拟通知，确认所有目标都已正确接入。底层生成的规格如下所示：

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

你不局限于单一类型的信号，可以选择最符合你对故障理解方式的触发条件：

| 触发条件 | 触发时机 |
|---|---|
| **指标阈值** | 预设指标（错误率、p95 或 p99 延迟、事件或错误计数、Token 消耗）在一个时间窗口内越过警戒线 |
| **自定义 SQL** | 你的只读查询返回结果行，或计算出的值越过阈值 |
| **评估分数** | 某项评估分数的均值（例如幻觉率）越过阈值 |
| **复合评估** | 多项分数检查通过 any、all 或至少 N 个的逻辑组合，捕捉只在跨分数维度才显现的回归 |
| **单事件** | 单个匹配事件发生：特定 Agent、特定错误类型，或消息包含特定子字符串 |

正在[错误页面](/zh/agenteye/error-tracking)排查故障？每一行都有一个 **+ 告警** 按钮，点击后将打开同一表单，并预填信息以精准捕获该错误的再次发生，让你刚刚处理过的事件成为下次自动提醒你的规则。

**入口：** 告警位于 `/<org-slug>/alerts`。创建、编辑、删除和测试规则需要 **`alerts:write`** 权限；仅查看只需 **`alerts:read`**。收件人选择器会按姓名列出组织成员，无需离开表单即可指定通知对象。

## 只在真正出问题时提醒我

一次异常测量不应触发告警。**M of N** 噪音过滤器控制最近几次检查中必须有多少次失败才会真正触发告警。设置为 **3 of 5**，规则只在最近五次检查中有三次超过阈值时才触发，避免抖动信号频繁误报；保持默认的 **1 of 1** 则在首次超出时立即触发。你还可以选择规则的运行频率，可从 1m、5m、15m、1h 等预设值中选择，根据信号实际变化速度匹配。

## 告警触发后会发生什么

一次超出阈值的事件会创建一个**事件（incident）**，并向你的通知渠道发送一次提醒。之后，团队成员可以确认事件、分配负责人、协作讨论并最终解决，整个过程都有清晰、有据可查的记录。事件处理流程有专属页面，详见[事件管理](/zh/agenteye/incidents)。
