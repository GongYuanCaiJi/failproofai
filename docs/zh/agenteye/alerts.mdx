在问题越过你的警戒线的那一刻即时获知，通过你团队已在使用的渠道，而不是从用户那里听说。只需设置一次规则，FailproofAI Observability 便会按计划检查，并通过邮件、Slack、Webhook 或直接在仪表板中通知你。

![告警页面：由告警规则卡片组成的网格，每张卡片显示其触发条件、评估窗口、通知渠道，以及信息、警告或严重的级别徽章](/agenteye/images/alerts.png)
*一览所有告警规则：监控内容、检查频率、通知位置及紧急程度。*

## 在用户察觉之前发现问题

不必盯着仪表板刷新以期捕捉到回归问题。只要有一个信号是你在无人值守时也希望第一时间获悉的，就为它设置一条告警，让通知直达你已在使用的地方：

- **邮件**，发送给所有需要知晓的人。
- **Slack**，发送一条富文本消息，附带可直接跳转到事件详情的按钮。
- **Webhook**，向 PagerDuty、Opsgenie 或你自己的接口发送 JSON POST 请求，可选签名以便接收方验证来源。
- **仪表板内通知**，设计上保持静默，适用于你在调试规则、暂时不想打扰任何人的阶段。

你可以为同一条规则绑定任意组合的渠道，告警的严重级别（信息、警告或严重）也会随之传递，让紧急的事件一眼看上去就显得紧急。

## 在表单中创建规则，而非编写 JSON

你在表单中描述"出了问题"意味着什么，FailproofAI Observability 会为你生成底层规则。JSON 规范只是该表单在底层产生的结果，你可以阅读它来理解一条规则，但几乎无需手动编写。

![新建告警表单：名称与描述、启用开关，以及提供指标阈值、自定义 SQL、评估得分、复合评估和单事件条件的触发器选择器](/agenteye/images/alert-new.png)
*选择触发器后，表单会自动切换为对应的字段；点击保存即完成规则创建。*

常规流程非常简洁：为规则命名，选择**触发器**（监控什么），设置**阈值和窗口**（严重程度和持续时长），绑定至少一个**通知渠道**，然后点击**保存**，再点击**测试**发送一条模拟通知，确认所有目标都已正确配置。在底层，这会生成一个类似如下的简短规范：

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

你不局限于单一类型的信号。根据你对故障的理解方式来选择对应的触发器：

| 触发器 | 触发时机 |
|---|---|
| **指标阈值** | 预设指标（错误率、p95 或 p99 延迟、事件或错误计数、Token 消耗）在一个时间窗口内超过设定值 |
| **自定义 SQL** | 你的只读查询返回了某行数据，或其计算值超过阈值 |
| **评估得分** | 某个评估器得分的平均值（如幻觉率）超过阈值 |
| **复合评估** | 多项得分检查通过 any、all 或 at-least-N 逻辑组合，以捕捉仅在多个得分维度综合表现下才显现的回归 |
| **单事件** | 单个匹配事件发生：特定 Agent、特定错误类型或包含特定子字符串的消息 |

已经在[错误页面](/zh/agenteye/error-tracking)盯着一个故障看了？每一行都有一个 **+ alert** 按钮，点击后会打开这个表单并预填内容，精准捕捉同类故障的再次发生——这样你刚处理完的事件，下次出现时就会主动通知你。

**访问路径：** 告警页面位于 `/<org-slug>/alerts`。创建、编辑、删除和测试规则需要 **`alerts:write`** 权限；仅查看需要 `alerts:read`。收件人选择器会按姓名列出你的组织成员，无需离开表单即可指定通知对象。

## 仅在真正需要时通知我

一次异常测量不应该把你从睡梦中吵醒。**M of N** 降噪过滤器控制在告警真正触发通知之前，最近几次检查中必须有多少次失败。设置为 **3 of 5** 后，规则仅在最近五次检查中有三次超阈值时才触发，避免抖动信号频繁误报；保持默认的 **1 of 1** 则在第一次超阈值时立即触发。你还可以选择规则的执行频率，预设选项有 1m、5m、15m 和 1h，可根据信号的实际变化速度来匹配。

## 告警触发后会发生什么

一次超阈值事件会开启一个**事件（incident）**，并向你的通知渠道发送一次通知。随后，你的团队可以对其进行确认、指派负责人、讨论处理方案并最终解决，整个过程都有清晰且可追溯的记录。该处理流程有独立的页面：请参阅[事件管理](/zh/agenteye/incidents)。
