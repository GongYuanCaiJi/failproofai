failproofai использует JSON-файлы конфигурации для управления активными политиками, их поведением и местоположением пользовательских политик. Конфигурация спроектирована так, чтобы её было легко делиться с командой — зафиксируйте её в репозитории, и каждый разработчик получит одинаковый уровень безопасности для агента.

---

## Области конфигурации

Существует три области конфигурации, оцениваемые в порядке приоритета:

| Область | Путь файла | Назначение |
|---------|-----------|-----------|
| **project** | `.failproofai/policies-config.json` | Параметры для конкретного репозитория, зафиксированные в системе контроля версий |
| **local** | `.failproofai/policies-config.local.json` | Личные переопределения для конкретного репозитория, добавленные в .gitignore |
| **global** | `~/.failproofai/policies-config.json` | Значения по умолчанию на уровне пользователя для всех проектов |

Когда failproofai получает событие хука, он загружает и объединяет все три файла, которые существуют для текущей рабочей директории.

### Правила слияния

**`enabledPolicies`** — объединение всех трёх областей. Политика, активированная на любом уровне, является активной.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← объединение без дубликатов
```

**`policyParams`** — первая область, которая определяет параметры для конкретной политики, побеждает полностью. Глубокое объединение значений внутри параметров политики не происходит.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project побеждает, global игнорируется
```

```text
project:  (нет записи block-sudo)
local:    (нет записи block-sudo)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← переходит на global
```

**`customPoliciesPath`** — первая область, которая её определяет, побеждает.

**`llm`** — первая область, которая её определяет, побеждает.

## Формат файла конфигурации

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## Справочник по полям

### `enabledPolicies`

Тип: `string[]`

Список имён политик для активации. Имена должны точно совпадать с идентификаторами политик, показанными командой `failproofai policies`. Полный список см. в [Встроенные политики](/ru/built-in-policies).

Политики, отсутствующие в `enabledPolicies`, неактивны, даже если у них есть записи в `policyParams`.

### `policyParams`

Тип: `Record<string, Record<string, unknown>>`

Переопределения параметров для конкретной политики. Внешний ключ — имя политики; внутренние ключи зависят от политики. Каждая политика описывает свои доступные параметры в [Встроенные политики](/ru/built-in-policies).

Если у политики есть параметры, но вы их не указали, используются встроенные значения по умолчанию. Пользователи, которые вообще не конфигурируют `policyParams`, получают идентичное поведение предыдущим версиям.

Неизвестные ключи внутри блока параметров политики молча игнорируются при срабатывании хука, но помечаются как предупреждения при выполнении `failproofai policies`.

#### `hint` (сквозной параметр)

Тип: `string` (необязательно)

Сообщение, добавляемое к причине, когда политика возвращает `deny` или `instruct`. Используйте его, чтобы дать Claude действенные рекомендации без изменения самой политики.

Работает с любым типом политики — встроенной, пользовательской (`custom/`), конвенции проекта (`.failproofai-project/`) или конвенции пользователя (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Попробуйте создать новую ветку вместо этого."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Используйте apt-get напрямую без sudo."
    },
    "custom/my-policy": {
      "hint": "Сначала попросите разрешение у пользователя."
    }
  }
}
```

Когда `block-force-push` отклоняет, Claude видит: *«Force-pushing заблокирован. Попробуйте создать новую ветку вместо этого.»*

Не строковые значения и пустые строки молча игнорируются. Если `hint` не установлен, поведение не изменяется (обратная совместимость).

### `customPoliciesPath`

Тип: `string` (абсолютный путь)

Путь к JavaScript-файлу, содержащему пользовательские политики хука. Этот параметр устанавливается автоматически командой `failproofai policies --install --custom <path>` (путь разрешается в абсолютный перед сохранением).

Файл загружается заново при каждом событии хука — кэширование отсутствует. Детали см. в [Пользовательские политики](/ru/custom-policies).

### Политики на основе конвенций

В дополнение к явному `customPoliciesPath` failproofai автоматически обнаруживает и загружает файлы политик из директорий `.failproofai/policies/`:

| Уровень | Директория | Область |
|---------|-----------|---------|
| Project | `.failproofai/policies/` | Используется командой через систему контроля версий |
| User | `~/.failproofai/policies/` | Личное, применяется ко всем проектам |

**Сопоставление файлов:** Загружаются только файлы, соответствующие `*policies.{js,mjs,ts}` (например `security-policies.mjs`, `workflow-policies.js`). Другие файлы в директории игнорируются.

**Конфигурация не требуется:** Политики конвенций не требуют записей в `policies-config.json`. Просто поместите файлы в директорию, и они будут обнаружены при следующем событии хука.

**Объединённая загрузка:** Сканируются обе директории конвенций (проекта и пользователя). Загружаются все подходящие файлы обоих уровней (в отличие от `customPoliciesPath`, который использует правило первой области).

Больше деталей и примеров см. в [Пользовательские политики](/ru/custom-policies).

### `llm`

Тип: `object` (необязательно)

Конфигурация клиента LLM для политик, которые делают вызовы AI. Не требуется для большинства конфигураций.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## Управление конфигурацией из командной строки

Команды `policies --install` и `policies --uninstall` записывают в файл параметров хука вашего агента CLI (точки входа хука), а `policies-config.json` — это файл, который вы управляете напрямую. Это два отдельных файла:

- **Параметры агента CLI** — указывает агенту вызывать `failproofai --hook <event>` при каждом использовании инструмента:
  - **Claude Code**: `~/.claude/settings.json` (пользователь), `<cwd>/.claude/settings.json` (проект), `<cwd>/.claude/settings.local.json` (локально)
  - **OpenAI Codex**: `~/.codex/hooks.json` (пользователь), `<cwd>/.codex/hooks.json` (проект) — Codex не имеет локальной области
  - **GitHub Copilot CLI _(бета)_**: `~/.copilot/hooks/failproofai.json` (пользователь), `<cwd>/.github/hooks/failproofai.json` (проект) — Copilot не имеет локальной области. Записи хуков используют поля команд Copilot, ключевые по ОС `bash`/`powershell`, с `timeoutSec`; файл содержит маркер `version: 1` верхнего уровня. Поддержка Copilot CLI находится в статусе **бета**, пока мы проверяем схему записей `events.jsonl` (которая не указана в публичной документации) на большем количестве реальных сессий.
  - **Cursor Agent _(бета)_**: `~/.cursor/hooks.json` (пользователь), `<cwd>/.cursor/hooks.json` (проект) — Cursor не имеет локальной области. Записи хуков используют форму, подобную Claude, `{type, command, timeout}` (без разделения `bash`/`powershell`), но хранятся под camelCase ключами событий (`preToolUse`, `beforeSubmitPrompt`, …) в плоском массиве согласно [схеме хуков](https://cursor.com/docs/hooks) Cursor; файл содержит маркер `version: 1` верхнего уровня. Обработчик канонизирует camelCase → PascalCase через `CURSOR_EVENT_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Поддержка Cursor Agent находится в статусе **бета**, пока мы проверяем формат транскрипта Cursor на диске (не указан в публичной документации) на большем количестве реальных установок.
  - **OpenCode _(бета)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (пользователь), `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (проект) — OpenCode не имеет локальной области. В отличие от других шести CLI, OpenCode **не имеет системы внешних команд хука**: он загружает встроенные JS/TS плагины, явно зарегистрированные через массив `plugin: []` в `opencode.json` (автообнаружение из `.opencode/plugins/` — это **не** способ загрузки плагинов в opencode v1.14.33). Install размещает небольшой сгенерированный шим плагина, который вызывает бинарный файл failproofai в подпроцессе и переводит ответ в формате Claude-shape JSON обратно в семантику плагина: `throw new Error()` для отклонения события инструмента (отменяет вызов инструмента), `client.session.prompt(...)` для instruct И для отклонения `Stop` / `SubagentStop` (отправляет причину отклонения как следующее сообщение пользователя — единственный канал принудительного повтора, так как `session.idle` — это только уведомление и выброс из него — это no-op), и no-op для allow. Сессии живут в БД SQLite OpenCode в `~/.local/share/opencode/opencode.db`; просмотрщик сессий приборной панели читает их через `opencode db --format json` и `opencode export <id>`. Поддержка OpenCode находится в статусе **бета**, пока мы проверяем поведение на версиях и большем количестве реальных сессий. См. [документацию по плагинам OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(бета)_**: `~/.pi/agent/settings.json` (пользователь), `<cwd>/.pi/settings.json` (проект) — Pi не имеет локальной области. Pi загружает пакеты расширений TypeScript при запуске; файл параметров — это плоский массив строк `{"packages": ["./relative/path", …]}`. failproofai записывает одну запись массива пакетов, указывающую на встроенную директорию `pi-extension/`. Расширение внутренне подписывается на события `tool_call` / `user_bash` / `input` / `session_start` Pi и вызывает `failproofai --hook <Event> --cli pi`; обработчик канонизирует snake_case_нижний_регистр → PascalCase через `PI_EVENT_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Поддержка Pi находится в статусе **бета**, пока API расширений Pi и макет журнала сессий стабилизируются.
  - **Gemini CLI _(бета)_**: `~/.gemini/settings.json` (пользователь), `<cwd>/.gemini/settings.json` (проект) — Gemini не имеет локальной области (она задаёт область `system` в `/etc/gemini-cli/settings.json`, которую failproofai не предоставляет). Записи хуков используют форму `{type, command, timeout}` Claude, обёрнутую в схему сопоставления `{matcher, hooks: [...]}` Gemini с `matcher: "*"` по умолчанию. События — это PascalCase (`SessionStart`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `BeforeTool`, `AfterTool`, `PreCompress`, `Notification`, `SessionEnd`); обработчик отображает канонические имена Claude через `GEMINI_EVENT_MAP`. Имена инструментов — это snake_case (`run_shell_command`, `read_file`, `write_file`, `replace`, …) — обработчик канонизирует через `GEMINI_TOOL_MAP`, поэтому существующие встроенные политики срабатывают без изменений. Оценяющий политику излучает плоскую форму `{decision: "deny", reason}` Gemini (предпочтение по правилу контракта exit-0 Gemini), `{hookSpecificOutput: {hookEventName, additionalContext}}` для внедрения контекста на BeforeAgent / AfterTool / SessionStart и `{decision: "block", reason}` на AfterAgent для семантики принудительного повтора. Поддержка Gemini CLI находится в статусе **бета**, пока мы расширяем реальное покрытие. См. [документацию по хукам Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — указывает failproofai, какие политики оценивать и с какими параметрами (общие для всех агентов CLI)

Передайте `--cli claude|codex|copilot|cursor|opencode|pi|gemini` для указания конкретного агента (разделённые пробелами или повторные для любого подмножества):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

Когда `--cli` опущен, `failproofai` обнаруживает, какие агенты CLI установлены (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **Один CLI обнаружен** — автоматически выбирает этот CLI без запроса.
- **Несколько CLI обнаружено** в интерактивном терминале — показывает приглашение с одиночным выбором клавишами со стрелками, сгруппированное в раздел `Detected (N)` (с строкой совокупности `Install for all N detected` + каждый обнаруженный CLI отдельно) и раздел `Not installed (M) · install hooks ahead of time`, в котором перечислены все необнаруженные поддерживаемые CLI в качестве опции предварительной установки (↑↓ для движения, Enter для выбора, ^C для выхода). Поток удаления показывает только раздел Detected.
- **Несколько CLI обнаружено** в неинтерактивном запуске (CI, нет TTY) — устанавливает для всех обнаруженных CLI без запроса.
- **Ни один не обнаружен** — откатывается на `claude` с предупреждением, что бинарный файл агента не найден в PATH; команда хука всё равно записывается, поэтому она активируется сразу же при установке.

Вы можете редактировать `policies-config.json` напрямую в любое время; изменения вступают в силу немедленно при следующем событии хука без необходимости перезагрузки.

## Пример: конфигурация уровня проекта с командными стандартами

Зафиксируйте `.failproofai/policies-config.json` в своём репозитории:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

Каждый разработчик может затем создать `.failproofai/policies-config.local.json` (добавленный в .gitignore) для личных переопределений без влияния на товарищей по команде.
