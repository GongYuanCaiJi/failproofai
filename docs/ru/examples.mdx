## Настройка hooks для Claude Code

Failproof AI интегрируется с Claude Code через его [систему hooks](https://docs.anthropic.com/en/docs/claude-code/hooks). Когда вы запускаете `failproofai policies --install`, он регистрирует команды hooks в `settings.json` Claude Code, которые срабатывают при каждом вызове инструмента.

<Steps>
  <Step title="Установите failproofai">
    ```bash
    npm install -g failproofai
    ```
  </Step>
  <Step title="Включите все встроенные политики">
    ```bash
    failproofai policies --install
    ```
  </Step>
  <Step title="Проверьте, что hooks зарегистрированы">
    ```bash
    cat ~/.claude/settings.json | grep failproofai
    ```

    Вы должны увидеть записи hooks для событий `PreToolUse`, `PostToolUse`, `Notification` и `Stop`.
  </Step>
  <Step title="Запустите Claude Code">
    ```bash
    claude
    ```

    Политики теперь запускаются автоматически при каждом вызове инструмента. Попробуйте попросить Claude выполнить `sudo rm -rf /` — это будет заблокировано.
  </Step>
</Steps>

## Настройка hooks для Agents SDK

Если вы разрабатываете с помощью [Agents SDK](https://docs.anthropic.com/en/docs/agents-sdk), вы можете использовать ту же систему hooks программно.

<Steps>
  <Step title="Установите failproofai в ваш проект">
    ```bash
    npm install failproofai
    ```
  </Step>
  <Step title="Настройте hooks в вашем агенте">
    Передайте команды hooks при создании процесса агента. Hooks срабатывают так же, как в Claude Code — через stdin/stdout JSON:

    ```bash
    failproofai --hook PreToolUse   # вызывается перед каждым инструментом
    failproofai --hook PostToolUse  # вызывается после каждого инструмента
    ```
  </Step>
  <Step title="Напишите пользовательскую политику для вашего агента">
    ```javascript
    import { customPolicies, allow, deny } from "failproofai";

    customPolicies.add({
      name: "limit-to-project-dir",
      description: "Держите агента внутри директории проекта",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        const path = String(ctx.toolInput?.file_path ?? "");
        if (path.startsWith("/") && !path.startsWith(ctx.session?.cwd ?? "")) {
          return deny("Агент ограничен директорией проекта");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="Установите пользовательскую политику">
    ```bash
    failproofai policies --install --custom ./my-agent-policies.js
    ```
  </Step>
</Steps>

## Блокировка деструктивных команд

Наиболее распространённая настройка — предотвращение необратимого ущерба от агентов.

```bash
failproofai policies --install block-sudo block-rm-rf block-force-push block-curl-pipe-sh
```

Что это делает:
- `block-sudo` — блокирует все команды `sudo`
- `block-rm-rf` — блокирует рекурсивное удаление файлов
- `block-force-push` — блокирует `git push --force`
- `block-curl-pipe-sh` — блокирует передачу удалённых скриптов в shell

## Предотвращение утечки секретов

Помешайте агентам видеть или раскрывать учётные данные в выводе инструментов.

```bash
failproofai policies --install sanitize-api-keys sanitize-jwt sanitize-connection-strings sanitize-bearer-tokens
```

Эти политики срабатывают на `PostToolUse` — после выполнения инструмента они очищают вывод перед тем, как агент его увидит.

## Получайте оповещения Slack, когда агенты требуют внимания

Используйте hook уведомлений для отправки оповещений простоя в Slack.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "slack-on-idle",
  description: "Оповестить Slack, когда агент ждёт ввода",
  match: { events: ["Notification"] },
  fn: async (ctx) => {
    const webhookUrl = process.env.SLACK_WEBHOOK_URL;
    if (!webhookUrl) return allow();

    const message = String(ctx.payload?.message ?? "Агент ждёт");
    const project = ctx.session?.cwd ?? "неизвестно";

    try {
      await fetch(webhookUrl, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          text: `*${message}*\nПроект: \`${project}\``,
        }),
        signal: AbortSignal.timeout(5000),
      });
    } catch {
      // никогда не блокируйте агента, если Slack недоступен
    }

    return allow();
  },
});
```

Установите это:

```bash
SLACK_WEBHOOK_URL=https://hooks.slack.com/... failproofai policies --install --custom ./slack-alerts.js
```

## Держите агентов на ветке

Предотвратите переключение веток агентами или отправку в защищённые ветки.

```javascript
import { customPolicies, allow, deny } from "failproofai";

customPolicies.add({
  name: "stay-on-branch",
  description: "Предотвратить переключение агента на другие ветки",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+checkout\s+(?!-b)/.test(cmd)) {
      return deny("Оставайтесь на текущей ветке. Создайте новую ветку с -b если необходимо.");
    }
    return allow();
  },
});
```

## Требуйте тесты перед коммитом

Напомните агентам запустить тесты перед коммитом.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "test-before-commit",
  description: "Напомнить агенту запустить тесты перед коммитом",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+commit/.test(cmd)) {
      return instruct("Запустите тесты перед коммитом. Сначала используйте `npm test` или `bun test`.");
    }
    return allow();
  },
});
```

## Заблокируйте production репозиторий

Закоммитьте конфигурацию на уровне проекта, чтобы каждый разработчик вашей команды получил одинаковые политики.

Создайте `.failproofai/policies-config.json` в вашем репозитории:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-force-push",
    "block-push-master",
    "block-env-files",
    "sanitize-api-keys",
    "sanitize-jwt"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "production"]
    }
  }
}
```

Затем закоммитьте это:

```bash
git add .failproofai/policies-config.json
git commit -m "Add failproofai team policies"
```

Каждый член команды, у которого установлена failproofai, автоматически получит эти правила.

## Создайте стандарт качества на уровне организации с политиками соглашений

Наиболее эффективная настройка: закоммитьте `.failproofai/policies/` в ваш репозиторий с политиками, адаптированными к вашему проекту. Каждый член команды получит их автоматически — без команд установки, без изменения конфигурации.

<Steps>
  <Step title="Создайте директорию и добавьте политики">
    ```bash
    mkdir -p .failproofai/policies
    ```

    ```js
    // .failproofai/policies/team-policies.mjs
    import { customPolicies, allow, deny, instruct } from "failproofai";

    // Навязывайте предпочитаемый менеджер пакетов вашей команды
    // (или вместо этого включите встроенную политику prefer-package-manager)
    customPolicies.add({
      name: "enforce-bun",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        const cmd = String(ctx.toolInput?.command ?? "");
        if (/\bnpm\b/.test(cmd)) return deny("Используйте bun вместо npm.");
        return allow();
      },
    });

    // Напомните агенту запустить тесты перед коммитом
    customPolicies.add({
      name: "test-before-commit",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        if (/git\s+commit/.test(ctx.toolInput?.command ?? "")) {
          return instruct("Запустите тесты перед коммитом.");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="Закоммитьте в git">
    ```bash
    git add .failproofai/policies/
    git commit -m "Add team quality policies"
    ```
  </Step>
  <Step title="Продолжайте совершенствоваться">
    Когда ваша команда столкнётся с новыми сбоями, добавляйте политики и отправляйте их. Каждый получит обновление при следующем `git pull`. Эти политики становятся живым стандартом качества, который растёт вместе с вашей командой.
  </Step>
</Steps>
