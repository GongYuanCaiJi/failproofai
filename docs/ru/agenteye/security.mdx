Failproof AI Observability спроектирован для работы рядом с вашими production-агентами, что означает, что он видит ваши промпты, входные и выходные данные инструментов. На этой странице объясняется, как он обеспечивает изоляцию, контроль и хранение этих данных в ваших руках. Если вы оцениваете Failproof AI Observability для проверки безопасности, начните отсюда.

---

## Ваши данные остаются в вашей среде

Failproof AI Observability — это самостоятельно размещаемое решение. События, промпты, ответы модели и аналитика хранятся в ваших собственных базах данных в вашей собственной среде. Никакие данные не отправляются в стороннее SaaS-приложение для хранения, и ваши данные остаются в вашем собственном облачном аккаунте.

---

## Изоляция тенантов

Один экземпляр Failproof AI Observability может размещать множество организаций, и каждая изолирована на уровне хранилища — это обеспечивается самой базой данных, а не только пользовательским интерфейсом:

- Операционные данные организации (пользователи, ключи, панели, сохранённые запросы) ограничены этой организацией, а чтение между организациями блокируется самой базой данных.
- Каждое поступившее событие отмечено идентификатором организации-владельца, поэтому события одной организации никогда не могут быть прочитаны другой.

Каждый маршрут панели ограничен слагом организации (`/<org-slug>/…`).

## Вход в систему

Failproof AI Observability использует вход без пароля на основе электронной почты. Нет пароля, который можно было бы фишить или утечь. Пользователь запрашивает одноразовый код (или односвязную волшебную ссылку), который отправляется на его электронную почту и быстро истекает. Вход защищен **списком разрешений**: аутентификацию могут использовать только адреса электронной почты (или домены), которые вы разрешили.

## Область доступа с использованием ключей API

Каждый клиент проходит аутентификацию с помощью ключа API, который имеет детальные разрешения с наименьшим привилегиям. Сборщику нужна только `events:add`; ключ панели или ассистента может быть доступен только для чтения; деструктивные действия (удаление, переинициализация) — это отдельные разрешения, которые вы выбираете включить.

![Страница ключей API: разрешения каждого ключа, цветно-закодированные по области чтения, записи и деструктивных действий](/agenteye/images/api-keys.png)

Сохраняйте главный ключ начальной загрузки для установки и выдавайте узкие ключи для всего остального. Смотрите [Ключи API](/ru/agenteye/api-keys).

## Ассистент только для чтения с одобрением

[AI ассистент](/ru/agenteye/assistant) в панели отвечает на вопросы о ваших данных, но он ограничен по замыслу:

- Он **доступен только для чтения по умолчанию**: его SQL запускается через защиту, которая допускает только `SELECT`/`WITH` запросы, однооператорные с ограничением количества строк.
- Все, что он создаёт (сохранённый запрос, панель), **требует одобрения**: вы просматриваете и одобряете каждую запись перед её выполнением.
- Он **никогда не может удалять**.

Таким образом, коллега может спросить „какие агенты чаще всего дали ошибку на этой неделе?" и действовать на основе ответа, без возможности ассистента самостоятельно менять или удалять ваши данные.
