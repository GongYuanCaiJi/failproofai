Пользовательские политики позволяют вам писать правила для любого поведения агента: соблюдать соглашения проекта, предотвращать отклонения, блокировать деструктивные операции, обнаруживать зависшие агенты или интегрироваться с Slack, workflow одобрения и многим другим. Они используют ту же систему hook-событий и решения `allow`, `deny`, `instruct`, что и встроенные политики.

---

## Быстрый пример

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

Установите это:

```bash
failproofai policies --install --custom ./my-policies.js
```

## Два способа загрузки пользовательских политик

### Вариант 1: На основе соглашений (рекомендуется)

Поместите файлы `*policies.{js,mjs,ts}` в `.failproofai/policies/` и они будут загружены автоматически — без флагов или изменения конфигурации. Это работает как git hooks: поместите файл, и он просто работает.

```
# Уровень проекта — коммитится в git, используется всей командой
.failproofai/policies/security-policies.mjs
.failproofai/policies/workflow-policies.mjs

# Уровень пользователя — личные настройки, применяются ко всем проектам
~/.failproofai/policies/my-policies.mjs
```

**Как это работает:**
- Оба каталога (проекта и пользователя) сканируются (объединение — не первый в области видимости)
- Файлы загружаются в алфавитном порядке в каждом каталоге. Используйте префиксы `01-`, `02-` для управления порядком
- Загружаются только файлы, соответствующие шаблону `*policies.{js,mjs,ts}`; остальные файлы игнорируются
- Каждый файл загружается независимо (fail-open для каждого файла)
- Работает параллельно с явным `--custom` и встроенными политиками

<Tip>
Политики по соглашению — это самый простой способ установить стандарт качества для вашей организации. Поместите `.failproofai/policies/` в git, и каждый член команды автоматически получит одинаковые правила — не требуется настройка для каждого разработчика. Когда ваша команда обнаружит новые режимы сбоев, добавьте политику и отправьте её. Со временем это станет живым стандартом качества, который совершенствуется с каждым вкладом.
</Tip>

### Вариант 2: Явный путь к файлу

```bash
# Установить с файлом пользовательских политик
failproofai policies --install --custom ./my-policies.js

# Заменить путь к файлу политик
failproofai policies --install --custom ./new-policies.js

# Удалить путь к пользовательским политикам из конфигурации
failproofai policies --uninstall --custom
```

Разрешённый абсолютный путь сохраняется в `policies-config.json` как `customPoliciesPath`. Файл загружается заново при каждом hook-событии — кэширование между событиями отсутствует.

### Использование обоих способов вместе

Политики по соглашению и явный файл `--custom` могут сосуществовать. Порядок загрузки:

1. Файл явной `customPoliciesPath` (если настроен)
2. Файлы соглашения проекта (`{cwd}/.failproofai/policies/`, в алфавитном порядке)
3. Файлы соглашения пользователя (`~/.failproofai/policies/`, в алфавитном порядке)

## API

### Импорт

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

Регистрирует политику. Вызовите это столько раз, сколько необходимо для нескольких политик в одном файле.

```ts
customPolicies.add({
  name: string;                         // обязательно - уникальный идентификатор
  description?: string;                 // показывается в выводе `failproofai policies`
  match?: { events?: HookEventType[] }; // фильтр по типу события; опустите, чтобы совпадать со всеми
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### Вспомогательные функции решений

| Функция | Эффект | Используйте, когда |
|---------|--------|-------------------|
| `allow()` | Разрешить операцию без сообщения | Действие безопасно, не требуется сообщение |
| `deny(message)` | Заблокировать операцию | Агент не должен выполнять это действие |
| `instruct(message)` | Добавить контекст без блокировки | Дать агенту дополнительный контекст для правильного выполнения |

`deny(message)` — сообщение появляется Claude с префиксом `"Blocked by failproofai:"`. Один `deny` прерывает всю дальнейшую оценку.

`instruct(message)` — сообщение добавляется в контекст Claude для текущего вызова инструмента. Все сообщения `instruct` накапливаются и доставляются вместе.

<Tip>
Вы можете добавить дополнительное руководство к любому сообщению `deny` или `instruct`, добавив поле `hint` в `policyParams` — без изменения кода. Это работает для пользовательских (`custom/`), проектных (`failproofai-project/`) и пользовательских по соглашению (`failproofai-user/`) политик. Подробнее см. [Конфигурация → hint](/ru/configuration#hint-cross-cutting).
</Tip>

### Информационные сообщения allow

`allow(message)` разрешает операцию **и** отправляет информационное сообщение обратно Claude. Сообщение доставляется как `additionalContext` в ответе stdout обработчика hook — тот же механизм, что и `instruct`, но семантически отличается: это обновление статуса, а не предупреждение.

| Функция | Эффект | Используйте, когда |
|---------|--------|-------------------|
| `allow(message)` | Разрешить и отправить контекст Claude | Подтвердить, что проверка пройдена, или объяснить, почему проверка пропущена |

Сценарии использования:
- **Подтверждения статуса:** `allow("All CI checks passed.")` — говорит Claude, что всё хорошо
- **Объяснения fail-open:** `allow("GitHub CLI not installed, skipping CI check.")` — говорит Claude, почему проверка пропущена, чтобы оно имело полный контекст
- **Множественные сообщения накапливаются:** если несколько политик каждая вернут `allow(message)`, все сообщения объединяются с переводами строк и доставляются вместе

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### Поля `PolicyContext`

| Поле | Тип | Описание |
|------|-----|---------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | Вызываемый инструмент (например, `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Входные параметры инструмента |
| `payload` | `Record<string, unknown>` | Полный raw payload события от Claude Code |
| `session` | `SessionMetadata \| undefined` | Контекст сессии (см. ниже) |

### Поля `SessionMetadata`

| Поле | Тип | Описание |
|------|-----|---------|
| `sessionId` | `string` | Идентификатор сессии Claude Code |
| `cwd` | `string` | Рабочий каталог сессии Claude Code |
| `transcriptPath` | `string` | Путь к файлу JSONL транскрипта сессии |

### Типы событий

| Событие | Когда срабатывает | Содержимое `toolInput` |
|--------|------------------|----------------------|
| `PreToolUse` | Перед запуском инструмента Claude | Вход инструмента (например, `{ command: "..." }` для Bash) |
| `PostToolUse` | После завершения инструмента | Вход инструмента + `tool_result` (выход) |
| `Notification` | Когда Claude отправляет уведомление | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` — hooks должны всегда возвращать `allow()`, они не могут блокировать уведомления |
| `Stop` | Когда сессия Claude завершается | Пусто |

## Порядок оценки

Политики оцениваются в этом порядке:

1. Встроенные политики (в порядке определения)
2. Явные пользовательские политики из `customPoliciesPath` (в порядке `.add()`)
3. Политики по соглашению из проекта `.failproofai/policies/` (файлы в алфавитном порядке, `.add()` порядок внутри)
4. Политики по соглашению из пользователя `~/.failproofai/policies/` (файлы в алфавитном порядке, `.add()` порядок внутри)

<Note>
Первый `deny` прерывает все последующие политики. Все сообщения `instruct` накапливаются и доставляются вместе.
</Note>

## Переходные импорты

Файлы пользовательских политик могут импортировать локальные модули, используя относительные пути:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Все относительные импорты, достижимые из файла входа, разрешаются. Это реализуется путём переписания импортов `from "failproofai"` на фактический путь dist и создания временных файлов `.mjs` для обеспечения совместимости ESM.

## Фильтрация по типам событий

Используйте `match.events` для ограничения срабатывания политики:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // Срабатывает только при завершении сессии
    // ctx.session.transcriptPath содержит полный журнал сессии
    return allow();
  },
});
```

Опустите `match` полностью, чтобы срабатывать на каждом типе события.

## Обработка ошибок и режимы отказа

Пользовательские политики — это **fail-open**: ошибки никогда не блокируют встроенные политики и не вызывают крах обработчика hook.

| Сбой | Поведение |
|------|-----------|
| `customPoliciesPath` не установлен | Явные пользовательские политики не запускаются; встроенные и политики по соглашению продолжают работу нормально |
| Файл не найден | Предупреждение записывается в `~/.failproofai/hook.log`; встроенные продолжают работу |
| Синтаксическая/ошибка импорта (явная) | Ошибка записывается в `~/.failproofai/hook.log`; явные пользовательские политики пропускаются |
| Синтаксическая/ошибка импорта (соглашение) | Ошибка записывается; этот файл пропускается, остальные файлы по соглашению всё ещё загружаются |
| `fn` выбрасывает исключение во время выполнения | Ошибка записывается; этот hook рассматривается как `allow`; остальные hooks продолжают работу |
| `fn` занимает больше 10 сек | Timeout записывается; рассматривается как `allow` |
| Каталог соглашений отсутствует | Политики по соглашению не запускаются; не возникает ошибка |

<Tip>
Для отладки ошибок пользовательских политик смотрите файл лога:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## Полный пример: несколько политик

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Предотвратить запись агентом в каталог secrets/
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Держать агента на курсе: проверить тесты перед коммитом
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Предотвратить незапланированные изменения зависимостей во время freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```

## Примеры

Каталог `examples/` содержит готовые к использованию файлы политик:

| Файл | Содержимое |
|------|-----------|
| `examples/policies-basic.js` | Пять начальных политик, охватывающих распространённые режимы сбоев агентов |
| `examples/policies-advanced/index.js` | Продвинутые паттерны: переходные импорты, асинхронные вызовы, очистка выходов и hooks при завершении сессии |
| `examples/convention-policies/security-policies.mjs` | Политики безопасности по соглашению (блокировка записи .env, предотвращение переписания истории git) |
| `examples/convention-policies/workflow-policies.mjs` | Политики workflow по соглашению (напоминания о тестах, аудит записей файлов) |

### Использование примеров явных файлов

```bash
failproofai policies --install --custom ./examples/policies-basic.js
```

### Использование примеров на основе соглашений

```bash
# Скопировать на уровень проекта
mkdir -p .failproofai/policies
cp examples/convention-policies/*.mjs .failproofai/policies/

# Или скопировать на уровень пользователя
mkdir -p ~/.failproofai/policies
cp examples/convention-policies/*.mjs ~/.failproofai/policies/
```

Команда установки не требуется — файлы будут подобраны автоматически при следующем hook-событии.
