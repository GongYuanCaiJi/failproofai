Erfahren Sie sofort, wenn etwas eine Grenze überschreitet – über den Kanal, den Ihr Team bereits nutzt – anstatt erst von einem Kunden davon zu hören. Legen Sie eine Regel einmal fest, und FailproofAI Observability prüft sie regelmäßig und benachrichtigt Sie per E-Mail, Slack, Webhook oder direkt im Dashboard.

![Die Alerts-Seite: ein Raster aus Alert-Regelkarten, jede mit Auslöser, Auswertungsfenster, Kanälen und einem Info-, Warn- oder Kritisch-Schweregrad-Badge](/agenteye/images/alerts.png)
*Alle Alert-Regeln auf einen Blick: was überwacht wird, wie oft, wohin die Benachrichtigung geht und wie dringend.*

## Probleme erkennen, bevor Ihre Nutzer es tun

Hören Sie auf, ein Dashboard zu aktualisieren und auf einen Rückschritt zu warten. Richten Sie einen Alert ein, sobald es ein Signal gibt, über das Sie auch dann informiert werden möchten, wenn niemand hinschaut – und lassen Sie ihn dort ankommen, wo Sie ohnehin sind:

- **E-Mail**, an alle, die es wissen sollten.
- **Slack**, eine informative Nachricht mit einer Schaltfläche, die direkt zum Vorfall führt.
- **Webhook**, ein JSON-POST für PagerDuty, Opsgenie oder Ihren eigenen Endpunkt, mit optionaler Signatur, damit der Empfänger die Quelle verifizieren kann.
- **Im Dashboard**, bewusst dezent – für den Fall, dass Sie eine Regel abstimmen und noch niemanden benachrichtigen möchten.

Kombinieren Sie beliebig viele Kanäle für eine einzige Regel. Der Schweregrad (Info, Warnung oder Kritisch) wird dabei übertragen, sodass dringende Meldungen auch als dringend erscheinen.

## Regeln per Formular erstellen, nicht per JSON

Sie beschreiben, was „fehlerhaft" bedeutet, in einem Formular – und FailproofAI Observability erstellt die zugrunde liegende Regel für Sie. Die JSON-Spezifikation ist lediglich das, was dieses Formular im Hintergrund erzeugt. Sie können sie lesen, um eine Regel zu verstehen, müssen sie aber kaum manuell eingeben.

![Das Formular für neue Alerts: Name und Beschreibung, ein Aktivierungs-Toggle und eine Auslöserauswahl mit Metrikschwellenwert, benutzerdefiniertem SQL, Auswertungsscore, zusammengesetzter Auswertung und ereignisbezogenen Bedingungen](/agenteye/images/alert-new.png)
*Wählen Sie einen Auslöser, und das Formular zeigt die passenden Felder an; Speichern schreibt die Regel.*

Der Standardweg geht schnell: Benennen Sie die Regel, wählen Sie einen **Auslöser** (was überwacht werden soll), legen Sie **Schwellenwert und Zeitfenster** fest (wie schwerwiegend, über welchen Zeitraum), fügen Sie mindestens einen **Kanal** hinzu, klicken Sie auf **Speichern** und anschließend auf **Testen**, um eine synthetische Benachrichtigung auszulösen und zu prüfen, ob alle Ziele korrekt konfiguriert sind. Im Hintergrund entsteht dabei eine kompakte Spezifikation wie:

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

Sie sind nicht auf eine einzige Art von Signal beschränkt. Wählen Sie den Auslöser, der Ihrer Vorstellung vom Fehler entspricht:

| Auslöser | Löst aus, wenn |
|---|---|
| **Metrikschwellenwert** | eine vordefinierte Metrik (Fehlerrate, p95- oder p99-Latenz, Ereignis- oder Fehlerzählungen, Token-Ausgaben) innerhalb eines Zeitfensters Ihren Grenzwert überschreitet |
| **Benutzerdefiniertes SQL** | Ihre eigene schreibgeschützte Abfrage eine Zeile zurückgibt oder ein berechneter Wert einen Schwellenwert überschreitet |
| **Auswertungsscore** | der Durchschnittswert eines Evaluators (z. B. Halluzination) einen Schwellenwert überschreitet |
| **Zusammengesetzte Auswertung** | mehrere Score-Prüfungen mit any-, all- oder at-least-N-Logik kombiniert werden, um eine Regression zu erkennen, die sich nur über mehrere Scores hinweg zeigt |
| **Pro Ereignis** | ein einzelnes passendes Ereignis eintrifft: ein bestimmter Agent, ein bestimmter Fehlertyp oder ein Nachrichtensubstring |

Schauen Sie sich gerade einen Fehler auf der [Errors-Seite](/de/agenteye/error-tracking) an? Jede Zeile dort hat eine Schaltfläche **+ Alert**, die dieses Formular vorausgefüllt öffnet, um genau diesen Fehler künftig abzufangen – damit der Vorfall, den Sie gerade triagiert haben, Sie beim nächsten Mal benachrichtigt.

**Wo Sie es finden:** Alerts befinden sich unter `/<org-slug>/alerts`. Zum Erstellen, Bearbeiten, Löschen und Testen von Regeln wird **`alerts:write`** benötigt; `alerts:read` reicht zum Anzeigen. Die Empfängerauswahl listet die Mitglieder Ihrer Organisation namentlich auf, sodass Sie eine Person benachrichtigen können, ohne das Formular zu verlassen.

## Nur bei echten Problemen benachrichtigen

Eine einzelne fehlerhafte Messung sollte Sie nicht aufwecken. Der **M of N**-Rauschfilter steuert, wie viele der letzten Prüfungen fehlschlagen müssen, bevor der Alert tatsächlich ausgelöst wird. Setzen Sie ihn auf **3 of 5**, und die Regel löst nur aus, wenn drei der letzten fünf Prüfungen den Schwellenwert überschritten haben – ein unruhiges Signal hört dadurch auf, falschen Alarm zu schlagen. Belassen Sie den Standardwert **1 of 1**, um beim ersten Überschreiten auszulösen. Sie wählen auch, wie häufig die Regel ausgeführt wird: aus vordefinierten Intervallen von 1m, 5m, 15m und 1h, abgestimmt auf die tatsächliche Dynamik des Signals.

## Was passiert, wenn ein Alert ausgelöst wird

Ein Regelverstoß öffnet einen **Incident** und benachrichtigt Ihre Kanäle einmalig. Von dort aus bestätigt Ihr Team den Vorfall, weist einen Verantwortlichen zu, bespricht ihn und löst ihn auf – alles in einem übersichtlichen, nachvollziehbaren Protokoll. Dieser Triage-Workflow hat eine eigene Seite: siehe [Incidents](/de/agenteye/incidents).

## Verwandte Themen

- [Incidents](/de/agenteye/incidents): Verfolgen Sie einen ausgelösten Alert von offen über bestätigt bis gelöst.
- [Error tracking](/de/agenteye/error-tracking): Gruppieren Sie Agent-Fehler und befördern Sie einen mit einem Klick in einen Alert.
- [Dashboards](/de/agenteye/dashboards): Behalten Sie die gemeinsamen Boards im Blick, aus denen die überwachten Schwellenwerte stammen.
- [CLI und Agents](/de/agenteye/cli-and-agents): Erstellen Sie Alerts und bestätigen Sie Incidents über Ihr Terminal oder integrieren Sie sie in CI-Skripte.
