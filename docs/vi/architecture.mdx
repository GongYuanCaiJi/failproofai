Tài liệu này giải thích cách failproofai hoạt động nội bộ: cách hệ thống hook chặn các lệnh gọi công cụ của agent, cách cấu hình được tải và hợp nhất, cách chính sách được đánh giá, và cách dashboard giám sát hoạt động của agent.

---

## Tổng quan

failproofai có hai hệ thống con độc lập:

1. **Hook handler** - Một CLI subprocess nhanh được Claude Code gọi trên mọi lệnh gọi công cụ của agent. Đánh giá chính sách và trả về quyết định.
2. **Agent Monitor (Dashboard)** - Một ứng dụng web Next.js để giám sát các phiên làm việc của agent và quản lý chính sách.

Cả hai hệ thống con chia sẻ các tệp cấu hình trong `~/.failproofai/` và thư mục `.failproofai/` của dự án, nhưng chúng chạy dưới dạng các tiến trình riêng biệt và chỉ giao tiếp thông qua hệ thống tệp.

## Hook handler

### Tích hợp với Claude Code

Khi bạn chạy `failproofai policies --install`, nó ghi các mục như thế này vào `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code sau đó gọi `failproofai --hook PreToolUse` dưới dạng một subprocess trước mỗi lệnh gọi công cụ, truyền một tải JSON trên stdin.

### Định dạng tải

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

Đối với các sự kiện `PostToolUse`, tải cũng chứa `tool_result` với đầu ra của công cụ.

Handler thực thi giới hạn stdin 1 MB. Các tải vượt quá giới hạn này sẽ bị loại bỏ và tất cả chính sách ngầm cho phép.

### Định dạng phản hồi

**Từ chối (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Từ chối (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Hướng dẫn (bất kỳ sự kiện nào ngoại trừ Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Hướng dẫn sự kiện Stop:**
- Mã thoát: `2`
- Lý do được ghi vào stderr (không phải stdout)

**Cho phép:**
- Mã thoát: `0`
- stdout trống

**Cho phép với thông báo:**

`allow(message)` cho phép một chính sách gửi thông tin ngữ cảnh trở lại Claude ngay cả khi hoạt động được phép. Hook handler ghi JSON sau đây vào **stdout** (không phải tệp cấu hình — đây là phản hồi của handler đối với Claude Code, giống như các phản hồi từ chối và hướng dẫn ở trên):

```json
// Written to stdout by the hook handler process
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Mã thoát: `0` (hoạt động được phép)
- Khi nhiều chính sách trả về `allow` với một thông báo, các thông báo của chúng được nối với dòng mới thành một chuỗi `additionalContext` duy nhất
- Nếu không có chính sách nào cung cấp thông báo, stdout trống (giống như trước đây)

### Quy trình xử lý

`src/hooks/handler.ts` triển khai toàn bộ quy trình:

```text
stdin JSON
  → parse payload (max 1 MB)
  → extract session metadata (session_id, cwd, tool_name, tool_input, etc.)
  → readMergedHooksConfig(cwd)    ← merges project + local + global config
  → register enabled builtin policies with resolved params
  → load custom policies from customPoliciesPath (if set)
  → register custom policies into policy registry
  → evaluate all policies (builtins first, then custom)
      → first deny short-circuits
      → instruct decisions accumulate
      → allow messages accumulate
  → write JSON decision to stdout
  → persist event to ~/.failproofai/hook-activity.jsonl
  → exit
```

Toàn bộ quá trình chạy dưới 100ms cho các tải điển hình không có lệnh gọi LLM.

## Tải cấu hình

`src/hooks/hooks-config.ts` triển khai tải cấu hình ba phạm vi.

```text
[1] {cwd}/.failproofai/policies-config.json        ← project  (highest priority)
[2] {cwd}/.failproofai/policies-config.local.json  ← local
[3] ~/.failproofai/policies-config.json             ← global   (lowest priority)
```

Logic hợp nhất:
- `enabledPolicies` - liên hợp được khử trùng lặp trên cả ba tệp
- `policyParams` - mỗi chính sách là một key, tệp đầu tiên định nghĩa nó sẽ thắng hoàn toàn
- `customPoliciesPath` - tệp đầu tiên định nghĩa nó sẽ thắng
- `llm` - tệp đầu tiên định nghĩa nó sẽ thắng

Dashboard web sử dụng `readHooksConfig()` (chỉ toàn cầu) để đọc và ghi, vì nó không được gọi với một cwd dự án.

## Đánh giá chính sách

`src/hooks/policy-evaluator.ts` chạy chính sách theo thứ tự.

Đối với mỗi chính sách:

1. Tra cứu schema `params` của chính sách (nếu có).
2. Đọc `policyParams[policy.name]` từ cấu hình được hợp nhất.
3. Hợp nhất các giá trị do người dùng cung cấp so với các giá trị mặc định của schema để tạo ra `ctx.params`.
4. Gọi `policy.fn(ctx)` với ngữ cảnh được giải quyết.
5. Nếu kết quả là `deny`, dừng ngay lập tức và trả về quyết định đó.
6. Nếu kết quả là `instruct`, tích lũy thông báo và tiếp tục.
7. Nếu kết quả là `allow`, tiếp tục đến chính sách tiếp theo.

Sau khi tất cả chính sách chạy:
- Nếu bất kỳ `deny` nào được trả về, phát ra phản hồi từ chối.
- Nếu bất kỳ kết quả `instruct` nào được thu thập, phát ra một phản hồi hướng dẫn duy nhất với tất cả các thông báo được nối.
- Ngược lại, phát ra phản hồi cho phép (stdout trống, thoát 0).

## Chính sách tích hợp sẵn

`src/hooks/builtin-policies.ts` định nghĩa tất cả 26 chính sách tích hợp sẵn dưới dạng các đối tượng `BuiltinPolicyDefinition`:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

Các chính sách chấp nhận `params` khai báo một `PolicyParamsSchema` với các loại và giá trị mặc định cho mỗi tham số. Trình đánh giá chính sách tiêm các giá trị đã được giải quyết vào `ctx.params` trước khi gọi `fn`. Các hàm chính sách đọc `ctx.params` mà không cần kiểm tra null vì các giá trị mặc định luôn được áp dụng trước tiên.

Khớp mẫu bên trong chính sách sử dụng các token lệnh được phân tích cú pháp (argv), không phải khớp chuỗi thô. Điều này ngăn chặn vượt qua bằng cách tiêm toán tử shell (ví dụ: một mẫu cho `sudo systemctl status *` không thể bị vượt qua bằng cách thêm `; rm -rf /` vào lệnh).

## Chính sách tùy chỉnh

`src/hooks/custom-hooks-registry.ts` triển khai một registry được hỗ trợ bởi `globalThis`:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

`src/hooks/custom-hooks-loader.ts` tải tệp chính sách của người dùng:

1. Đọc `customPoliciesPath` từ cấu hình; bỏ qua nếu vắng mặt.
2. Giải quyết đến đường dẫn tuyệt đối; kiểm tra tệp tồn tại.
3. Ghi lại tất cả các import `from "failproofai"` vào đường dẫn dist thực tế để `customPolicies` giải quyết cùng một registry `globalThis`.
4. Ghi lại các import địa phương chuyên chở theo cách đệ quy để đảm bảo tính tương thích ESM.
5. Ghi các tệp `.mjs` tạm thời và `import()` tệp mục nhập.
6. Gọi `getCustomHooks()` để truy xuất các hook được đăng ký.
7. Dọn sạch tất cả các tệp tạm thời trong khối `finally`.

Trên bất kỳ lỗi nào (tệp không tìm thấy, lỗi cú pháp, lỗi import), lỗi được ghi vào `~/.failproofai/hook.log` và loader trả về một mảng trống. Các chính sách tích hợp sẵn không bị ảnh hưởng.

Các chính sách tùy chỉnh được đánh giá sau tất cả các chính sách tích hợp sẵn. Một chính sách tùy chỉnh `deny` vẫn làm ngắn mạch các chính sách tùy chỉnh tiếp theo (nhưng tất cả các tích hợp sẵn đã chạy trước đó).

## Ghi nhật ký hoạt động

Sau mỗi sự kiện hook, handler thêm một dòng JSONL vào `~/.failproofai/hook-activity.jsonl`:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

Một dòng cho mỗi chính sách đưa ra quyết định không cho phép. Các quyết định cho phép không được ghi (để giữ tệp nhỏ).

## Kiến trúc dashboard

Dashboard là một ứng dụng **Next.js 16** sử dụng App Router với React Server Components và Server Actions.

```text
app/
  layout.tsx                  ← Root layout (theme, telemetry, nav)
  projects/page.tsx           ← Server component: list all Claude projects
  project/[name]/page.tsx     ← Server component: list sessions in a project
  project/[name]/session/
    [sessionId]/page.tsx      ← Server component: render session viewer
  policies/page.tsx           ← Client component: policy management + activity log
  actions/
    get-hooks-config.ts       ← Read config + policy list
    update-hooks-config.ts    ← Toggle policy on/off
    update-policy-params.ts   ← Update policy parameters
    get-hook-activity.ts      ← Paginate/search activity log
    install-hooks-web.ts      ← Install/remove hooks from the browser
  api/
    download/[project]/[session]/route.ts   ← Per-CLI session export (JSONL or JSON)
```

**Luồng dữ liệu:**

- Các thành phần trang gọi `lib/projects.ts` và `lib/log-entries.ts` để đọc dữ liệu dự án/phiên trực tiếp từ hệ thống tệp (không có lớp API cho các phép đọc).
- Trang Chính sách sử dụng Server Actions cho tất cả các thay đổi (bật/tắt, cập nhật tham số, cài đặt/xóa).
- Trình xem phiên phân tích định dạng transcript JSONL của Claude và hiển thị dòng thời gian của các thông báo và lệnh gọi công cụ.

**Quyết định thiết kế chính:**

- Không có cơ sở dữ liệu - tất cả trạng thái liên tục nằm trong các tệp thuần túy (`~/.failproofai/`, `~/.claude/projects/`).
- Server Actions cho các thay đổi - không cần REST API cho các hoạt động CRUD.
- React Server Components cho các trang đọc - tải ban đầu nhanh hơn, không có gói client cho tìm nạp dữ liệu.
- Các thành phần client chỉ khi cần tương tác (bật/tắt chính sách, tìm kiếm hoạt động, trình xem nhật ký).
