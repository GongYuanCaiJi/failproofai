## إعداد الخطافات لـ Claude Code

يتكامل Failproof AI مع Claude Code عبر [نظام الخطافات الخاص به](https://docs.anthropic.com/en/docs/claude-code/hooks). عند تشغيل `failproofai policies --install`، يسجل أوامر الخطافات في `settings.json` الخاص بـ Claude Code والتي تُطلق عند كل استدعاء أداة.

<Steps>
  <Step title="تثبيت failproofai">
    ```bash
    npm install -g failproofai
    ```
  </Step>
  <Step title="تفعيل جميع السياسات المدمجة">
    ```bash
    failproofai policies --install
    ```
  </Step>
  <Step title="التحقق من تسجيل الخطافات">
    ```bash
    cat ~/.claude/settings.json | grep failproofai
    ```

    يجب أن تشاهد إدخالات الخطافات لأحداث `PreToolUse`، `PostToolUse`، `Notification`، و`Stop`.
  </Step>
  <Step title="تشغيل Claude Code">
    ```bash
    claude
    ```

    تعمل السياسات تلقائياً عند كل استدعاء أداة. جرب طلب Claude لتشغيل `sudo rm -rf /` - سيتم حظره.
  </Step>
</Steps>

## إعداد الخطافات لـ Agents SDK

إذا كنت تبني باستخدام [Agents SDK](https://docs.anthropic.com/en/docs/agents-sdk)، يمكنك استخدام نفس نظام الخطافات برمجياً.

<Steps>
  <Step title="تثبيت failproofai في مشروعك">
    ```bash
    npm install failproofai
    ```
  </Step>
  <Step title="تكوين الخطافات في وكيلك">
    مرّر أوامر الخطافات عند إنشاء عملية الوكيل الخاص بك. تعمل الخطافات بنفس الطريقة كما في Claude Code - عبر JSON في stdin/stdout:

    ```bash
    failproofai --hook PreToolUse   # يُستدعى قبل كل أداة
    failproofai --hook PostToolUse  # يُستدعى بعد كل أداة
    ```
  </Step>
  <Step title="كتابة سياسة مخصصة لوكيلك">
    ```javascript
    import { customPolicies, allow, deny } from "failproofai";

    customPolicies.add({
      name: "limit-to-project-dir",
      description: "ابقِ الوكيل داخل دليل المشروع",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        const path = String(ctx.toolInput?.file_path ?? "");
        if (path.startsWith("/") && !path.startsWith(ctx.session?.cwd ?? "")) {
          return deny("الوكيل مقيد بدليل المشروع");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="تثبيت السياسة المخصصة">
    ```bash
    failproofai policies --install --custom ./my-agent-policies.js
    ```
  </Step>
</Steps>

## حظر الأوامر المدمرة

الإعداد الأكثر شيوعاً - منع الوكلاء من إحداث أضرار لا يمكن إصلاحها.

```bash
failproofai policies --install block-sudo block-rm-rf block-force-push block-curl-pipe-sh
```

ما يفعله هذا:
- `block-sudo` - يحظر جميع أوامر `sudo`
- `block-rm-rf` - يحظر حذف الملفات العودي
- `block-force-push` - يحظر `git push --force`
- `block-curl-pipe-sh` - يحظر توجيه البرامج النصية البعيدة إلى الشل

## منع تسرب الأسرار

منع الوكلاء من رؤية أو تسريب بيانات المصادقة في مخرجات الأداة.

```bash
failproofai policies --install sanitize-api-keys sanitize-jwt sanitize-connection-strings sanitize-bearer-tokens
```

تعمل هذه على `PostToolUse` - بعد تشغيل الأداة، تنظف المخرجات قبل أن يراها الوكيل.

## احصل على تنبيهات Slack عند احتياج الوكلاء للانتباه

استخدم خطاف الإخطار لإرسال تنبيهات الخمول إلى Slack.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "slack-on-idle",
  description: "التنبيه إلى Slack عند انتظار الوكيل للإدخال",
  match: { events: ["Notification"] },
  fn: async (ctx) => {
    const webhookUrl = process.env.SLACK_WEBHOOK_URL;
    if (!webhookUrl) return allow();

    const message = String(ctx.payload?.message ?? "الوكيل ينتظر");
    const project = ctx.session?.cwd ?? "unknown";

    try {
      await fetch(webhookUrl, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          text: `*${message}*\nProject: \`${project}\``,
        }),
        signal: AbortSignal.timeout(5000),
      });
    } catch {
      // لا تحظر الوكيل أبداً إذا كان Slack غير متاح
    }

    return allow();
  },
});
```

ثبتها:

```bash
SLACK_WEBHOOK_URL=https://hooks.slack.com/... failproofai policies --install --custom ./slack-alerts.js
```

## ابقِ الوكلاء على فرع معين

منع الوكلاء من تبديل الفروع أو الدفع إلى الفروع المحمية.

```javascript
import { customPolicies, allow, deny } from "failproofai";

customPolicies.add({
  name: "stay-on-branch",
  description: "منع الوكيل من الانتقال إلى فروع أخرى",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+checkout\s+(?!-b)/.test(cmd)) {
      return deny("ابقَ على الفرع الحالي. أنشئ فرعاً جديداً باستخدام -b إذا لزم الأمر.");
    }
    return allow();
  },
});
```

## اطلب الاختبارات قبل التزام الكود

ذكّر الوكلاء بتشغيل الاختبارات قبل التزام الكود.

```javascript
import { customPolicies, allow, instruct } from "failproofai";

customPolicies.add({
  name: "test-before-commit",
  description: "ذكّر الوكيل بتشغيل الاختبارات قبل التزام الكود",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = String(ctx.toolInput?.command ?? "");
    if (/git\s+commit/.test(cmd)) {
      return instruct("شغّل الاختبارات قبل التزام الكود. استخدم `npm test` أو `bun test` أولاً.");
    }
    return allow();
  },
});
```

## تأمين مستودع الإنتاج

التزم بملف إعدادات على مستوى المشروع حتى يحصل كل مطور في فريقك على نفس السياسات.

أنشئ `.failproofai/policies-config.json` في مستودعك:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-force-push",
    "block-push-master",
    "block-env-files",
    "sanitize-api-keys",
    "sanitize-jwt"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "production"]
    }
  }
}
```

ثم التزم بها:

```bash
git add .failproofai/policies-config.json
git commit -m "Add failproofai team policies"
```

كل عضو فريق لديه failproofai مثبتاً سيلتقط هذه القواعد تلقائياً.

## بناء معيار جودة على مستوى المؤسسة مع سياسات الاتفاقية

الإعداد الأكثر تأثيراً: التزم بـ `.failproofai/policies/` في مستودعك مع سياسات مصممة لمشروعك. كل عضو فريق يحصل عليها تلقائياً — بدون أوامر تثبيت، بدون تغييرات إعدادات.

<Steps>
  <Step title="أنشئ الدليل وأضف السياسات">
    ```bash
    mkdir -p .failproofai/policies
    ```

    ```js
    // .failproofai/policies/team-policies.mjs
    import { customPolicies, allow, deny, instruct } from "failproofai";

    // فرض مدير الحزم المفضل لفريقك
    // (أو فعّل سياسة prefer-package-manager المدمجة بدلاً من ذلك)
    customPolicies.add({
      name: "enforce-bun",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        const cmd = String(ctx.toolInput?.command ?? "");
        if (/\bnpm\b/.test(cmd)) return deny("استخدم bun بدلاً من npm.");
        return allow();
      },
    });

    // ذكّر الوكيل بتشغيل الاختبارات قبل التزام الكود
    customPolicies.add({
      name: "test-before-commit",
      match: { events: ["PreToolUse"] },
      fn: async (ctx) => {
        if (ctx.toolName !== "Bash") return allow();
        if (/git\s+commit/.test(ctx.toolInput?.command ?? "")) {
          return instruct("شغّل الاختبارات قبل التزام الكود.");
        }
        return allow();
      },
    });
    ```
  </Step>
  <Step title="التزم بـ git">
    ```bash
    git add .failproofai/policies/
    git commit -m "Add team quality policies"
    ```
  </Step>
  <Step title="استمر في التحسين">
    عندما يواجه فريقك أنماط فشل جديدة، أضف السياسات وادفعها. يحصل الجميع على التحديث عند `git pull` التالية. تصبح هذه السياسات معيار جودة حي ينمو مع فريقك.
  </Step>
</Steps>

## أمثلة أخرى

يحتوي دليل [`examples/`](https://github.com/exospherehost/failproofai/tree/main/examples) في المستودع على:

| الملف | ما يوضحه |
|------|----------|
| `policies-basic.js` | السياسات الأساسية - حظر عمليات الكتابة في الإنتاج، الدفع القسري، البرامج النصية الموجهة |
| `policies-notification.js` | تنبيهات Slack لإخطارات الخمول ونهاية الجلسة |
| `policies-advanced/index.js` | الاستيرادات الموصولة، الخطافات غير المتزامنة، تنظيف مخرجات PostToolUse، معالجة حدث Stop |
