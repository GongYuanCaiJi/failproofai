السياسات المخصصة تتيح لك كتابة قواعد لأي سلوك وكيل: فرض اتفاقيات المشروع والحد من الانجراف وحماية العمليات المدمرة واكتشاف الوكلاء المعلقين أو التكامل مع Slack وسير العمل الموافقة وغير ذلك. وهي تستخدم نفس نظام أحداث الـ hook وقرارات `allow` و `deny` و `instruct` كما هو الحال مع السياسات المدمجة.

---

## مثال سريع

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

قم بتثبيته:

```bash
failproofai policies --install --custom ./my-policies.js
```

## طريقتان لتحميل السياسات المخصصة

### الخيار 1: على أساس الاتفاقية (موصى به)

ضع ملفات `*policies.{js,mjs,ts}` في `.failproofai/policies/` وسيتم تحميلها تلقائياً — لا حاجة إلى أعلام أو تغييرات في الإعدادات. يعمل هذا مثل git hooks: ضع ملف وسيعمل فقط.

```
# Project level — committed to git, shared with the team
.failproofai/policies/security-policies.mjs
.failproofai/policies/workflow-policies.mjs

# User level — personal, applies to all projects
~/.failproofai/policies/my-policies.mjs
```

**كيف يعمل:**
- يتم فحص كلا المجلدين (اتحاد — وليس first-scope-wins)
- يتم تحميل الملفات أبجدياً داخل كل مجلد. استخدم البادئات `01-` و `02-` للتحكم في الترتيب
- يتم تحميل الملفات المطابقة فقط `*policies.{js,mjs,ts}`؛ يتم تجاهل الملفات الأخرى
- يتم تحميل كل ملف بشكل مستقل (fail-open لكل ملف)
- يعمل بجانب `--custom` الصريح والسياسات المدمجة

<Tip>
سياسات الاتفاقية هي الطريقة الأسهل لبناء معيار جودة لمؤسستك. التزم بـ `.failproofai/policies/` على git وسيحصل كل عضو في الفريق على نفس القواعد تلقائياً — بدون إعداد لكل مطور. مع اكتشاف فريقك لأنماط فشل جديدة أضف سياسة واضغط. بمرور الوقت تصبح معايير جودة حية تتحسن مع كل مساهمة.
</Tip>

### الخيار 2: مسار الملف الصريح

```bash
# Install with a custom policies file
failproofai policies --install --custom ./my-policies.js

# Replace the policies file path
failproofai policies --install --custom ./new-policies.js

# Remove the custom policies path from config
failproofai policies --uninstall --custom
```

يتم حفظ مسار مطلق تم حله في `policies-config.json` كـ `customPoliciesPath`. يتم تحميل الملف بشكل جديد في كل حدث hook — لا يوجد تخزين مؤقت بين الأحداث.

### استخدام كليهما معاً

يمكن أن تتعايش سياسات الاتفاقية والملف الصريح `--custom`. ترتيب التحميل:

1. ملف `customPoliciesPath` الصريح (إن تم تكوينه)
2. ملفات اتفاقية المشروع (`{cwd}/.failproofai/policies/`، أبجدي)
3. ملفات اتفاقية المستخدم (`~/.failproofai/policies/`، أبجدي)

## API

### استيراد

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

تسجيل سياسة. اتصل بها عدة مرات حسب الحاجة للسياسات المتعددة في نفس الملف.

```ts
customPolicies.add({
  name: string;                         // required - unique identifier
  description?: string;                 // shown in `failproofai policies` output
  match?: { events?: HookEventType[] }; // filter by event type; omit to match all
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### مساعدات القرار

| الدالة | التأثير | الاستخدام عند |
|-------|--------|---------|
| `allow()` | السماح بالعملية بصمت | الإجراء آمن وليس هناك حاجة إلى رسالة |
| `deny(message)` | حظر العملية | يجب ألا يتخذ الوكيل هذا الإجراء |
| `instruct(message)` | إضافة سياق بدون حظر | إعطاء الوكيل سياق إضافي للبقاء على المسار الصحيح |

`deny(message)` - تظهر الرسالة إلى Claude بلاحقة `Blocked by failproofai:`، `deny` واحد يختصر كل التقييم الإضافي.

`instruct(message)` - يتم إلحاق الرسالة بسياق Claude للاستدعاء الحالي للأداة. يتم تجميع كل رسائل `instruct` وتسليمها معاً.

<Tip>
يمكنك إضافة إرشادات إضافية إلى أي رسالة `deny` أو `instruct` بإضافة حقل `hint` في `policyParams` — بدون تغيير الكود. يعمل هذا مع السياسات المخصصة (`custom/`) واتفاقيات المشروع (`.failproofai-project/`) واتفاقيات المستخدم (`.failproofai-user/`) أيضاً. انظر [Configuration → hint](/ar/configuration#hint-cross-cutting) للتفاصيل.
</Tip>

### رسائل السماح المعلوماتية

`allow(message)` تسمح بالعملية **و** ترسل رسالة معلوماتية إلى Claude. يتم تسليم الرسالة كـ `additionalContext` في استجابة stdout لمعالج الـ hook — نفس الآلية المستخدمة بـ `instruct` لكن مختلفة من الناحية الدلالية: إنها تحديث حالة وليس تحذير.

| الدالة | التأثير | الاستخدام عند |
|-------|--------|---------|
| `allow(message)` | السماح وإرسال سياق إلى Claude | تأكيد نجاح الفحص أو شرح سبب تخطي الفحص |

حالات الاستخدام:
- **تأكيدات الحالة:** `allow("All CI checks passed.")` — يخبر Claude أن كل شيء بخير
- **شرح fail-open:** `allow("GitHub CLI not installed, skipping CI check.")` — يخبر Claude لماذا تم تخطي الفحص لديه سياق كامل
- **تراكم الرسائل المتعددة:** إذا أعادت عدة سياسات `allow(message)`، يتم ربط جميع الرسائل بعلامات سطر جديدة وتسليمها معاً

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### حقول `PolicyContext`

| الحقل | النوع | الوصف |
|------|------|-------|
| `eventType` | `string` | `PreToolUse` أو `PostToolUse` أو `Notification` أو `Stop` |
| `toolName` | `string \| undefined` | الأداة التي يتم استدعاؤها (مثلاً `Bash` أو `Write` أو `Read`) |
| `toolInput` | `Record<string, unknown> \| undefined` | معاملات إدخال الأداة |
| `payload` | `Record<string, unknown>` | الحمولة الخام الكاملة من Claude Code |
| `session` | `SessionMetadata \| undefined` | سياق الجلسة (انظر أدناه) |

### حقول `SessionMetadata`

| الحقل | النوع | الوصف |
|------|------|-------|
| `sessionId` | `string` | معرف جلسة Claude Code |
| `cwd` | `string` | دليل العمل لجلسة Claude Code |
| `transcriptPath` | `string` | مسار ملف النسخة JSONL للجلسة |

### أنواع الأحداث

| الحدث | متى يتم تفعيله | محتويات `toolInput` |
|-----|------------|-----------|
| `PreToolUse` | قبل تشغيل Claude للأداة | إدخال الأداة (مثلاً `{ command: "..." }` لـ Bash) |
| `PostToolUse` | بعد اكتمال الأداة | إدخال الأداة + `tool_result` (الإخراج) |
| `Notification` | عند إرسال Claude إشعار | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - يجب على الـ hooks دائماً إعادة `allow()`، لا يمكنهم حظر الإشعارات |
| `Stop` | عند انتهاء جلسة Claude | فارغ |

## ترتيب التقييم

يتم تقييم السياسات بهذا الترتيب:

1. السياسات المدمجة (بترتيب التعريف)
2. السياسات المخصصة الصريحة من `customPoliciesPath` (بترتيب `.add()`)
3. سياسات الاتفاقية من `.failproofai/policies/` للمشروع (الملفات أبجدياً وترتيب `.add()` داخلياً)
4. سياسات الاتفاقية من `~/.failproofai/policies/` للمستخدم (الملفات أبجدياً وترتيب `.add()` داخلياً)

<Note>
أول `deny` يختصر جميع السياسات اللاحقة. يتم تجميع جميع رسائل `instruct` وتسليمها معاً.
</Note>

## الاستيرادات المتعدية

يمكن لملفات السياسة المخصصة استيراد الوحدات المحلية باستخدام المسارات النسبية:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

يتم حل جميع الاستيرادات النسبية القابلة للوصول من ملف الإدخال. يتم تطبيق هذا بإعادة كتابة استيرادات `from "failproofai"` إلى مسار dist الفعلي وإنشاء ملفات `.mjs` مؤقتة لضمان توافق ESM.

## تصفية نوع الحدث

استخدم `match.events` لتحديد متى تعمل السياسة:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // Only fires when the session ends
    // ctx.session.transcriptPath contains the full session log
    return allow();
  },
});
```

حذف `match` بالكامل للعمل على كل نوع حدث.

## معالجة الأخطاء ووسائط الفشل

السياسات المخصصة **fail-open**: الأخطاء لا تحظر السياسات المدمجة ولا تعطل معالج الـ hook.

| الفشل | السلوك |
|------|--------|
| عدم تعيين `customPoliciesPath` | لا تعمل السياسات المخصصة الصريحة؛ تستمر سياسات الاتفاقية والمدمجة بشكل طبيعي |
| الملف غير موجود | تحذير مسجل إلى `~/.failproofai/hook.log`؛ المدمجات تستمر |
| خطأ بصيغة/استيراد (صريح) | خطأ مسجل إلى `~/.failproofai/hook.log`؛ تم تخطي السياسات المخصصة الصريحة |
| خطأ بصيغة/استيراد (اتفاقية) | خطأ مسجل؛ تم تخطي هذا الملف، ملفات الاتفاقية الأخرى تستمر بالتحميل |
| `fn` يرمي في وقت التشغيل | خطأ مسجل؛ يتم التعامل مع هذا الـ hook كـ `allow`؛ الـ hooks الأخرى تستمر |
| `fn` يستغرق أطول من 10 ثوانٍ | timeout مسجل؛ يتم التعامل معه كـ `allow` |
| مجلد الاتفاقية مفقود | لا تعمل سياسات الاتفاقية؛ بدون خطأ |

<Tip>
لتصحيح أخطاء السياسة المخصصة شاهد ملف السجل:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## مثال كامل: سياسات متعددة

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```

## أمثلة

مجلد `examples/` يحتوي على ملفات السياسة الجاهزة للتشغيل:

| الملف | المحتويات |
|------|----------|
| `examples/policies-basic.js` | خمس سياسات بدء تغطي أنماط فشل الوكيل الشائعة |
| `examples/policies-advanced/index.js` | أنماط متقدمة: استيرادات متعدية واستدعاءات غير متزامنة وتنقية المخرجات وـ hooks نهاية الجلسة |
| `examples/convention-policies/security-policies.mjs` | سياسات أمان قائمة على الاتفاقية (حظر كتابة .env ومنع إعادة كتابة git history) |
| `examples/convention-policies/workflow-policies.mjs` | سياسات workflow قائمة على الاتفاقية (تذكيرات الاختبار وملفات الكتابة في المراجعة) |

### استخدام أمثلة الملف الصريح

```bash
failproofai policies --install --custom ./examples/policies-basic.js
```

### استخدام أمثلة قائمة على الاتفاقية

```bash
# Copy to project level
mkdir -p .failproofai/policies
cp examples/convention-policies/*.mjs .failproofai/policies/

# Or copy to user level
mkdir -p ~/.failproofai/policies
cp examples/convention-policies/*.mjs ~/.failproofai/policies/
```

لا حاجة لأمر التثبيت — يتم التقاط الملفات تلقائياً في حدث hook التالي.
