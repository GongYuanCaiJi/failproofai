يأتي failproofai مع 39 سياسة مدمجة تلتقط أنماط الفشل الشائعة للوكلاء. تعمل كل سياسة على نوع حدث hook محدد واسم أداة محدد. تقبل تسع عشرة سياسة معاملات تسمح لك بضبط سلوكها دون كتابة أي كود. تفرض خمس سياسات سير عمل عملية التزام → دفع → طلب دمج → CI قبل توقف Claude.

## نظرة عامة

تُجمّع السياسات في فئات:

| الفئة | السياسات | نوع Hook |
|----------|----------|-----------|
| [الأوامر الخطرة](#dangerous-commands) | block-sudo, block-rm-rf, block-curl-pipe-sh, block-failproofai-commands | PreToolUse |
| [أوامر البنية التحتية](#infra-commands) | block-kubectl, block-terraform, block-aws-cli, block-gcloud, block-az-cli, block-helm, block-gh-pipeline | PreToolUse |
| [الأسرار (المنظفات)](#secrets-sanitizers) | sanitize-jwt, sanitize-api-keys, sanitize-connection-strings, sanitize-private-key-content, sanitize-bearer-tokens | PostToolUse |
| [البيئة](#environment) | block-env-files, protect-env-vars | PreToolUse |
| [الوصول إلى الملفات](#file-access) | block-read-outside-cwd, block-secrets-write | PreToolUse |
| [Git](#git) | block-push-master, block-work-on-main, block-force-push, warn-git-amend, warn-git-stash-drop, warn-all-files-staged | PreToolUse |
| [قاعدة البيانات](#database) | warn-destructive-sql, warn-schema-alteration | PreToolUse |
| [التحذيرات](#warnings) | warn-large-file-write, warn-package-publish, warn-background-process, warn-global-package-install | PreToolUse |
| [مديري الحزم](#package-managers) | prefer-package-manager | PreToolUse |
| [سير العمل](#workflow) | require-commit-before-stop, require-push-before-stop, require-pr-before-stop, require-no-conflicts-before-stop, require-ci-green-before-stop | Stop |

- **`block-`** — وقف الوكيل عن المتابعة.
- **`warn-`** — إعطاء الوكيل سياقًا إضافيًا حتى يتمكن من تصحيح نفسه بنفسه.
- **`sanitize-`** — كشط البيانات الحساسة من مخرجات الأداة قبل أن يراها الوكيل.

### المساحات الاسمية

تعيش كل سياسة في فتحة `<namespace>/<name>`. تنتمي السياسات المدمجة إلى مساحة الاسم
**`exospherehost/`** — على سبيل المثال، `exospherehost/sanitize-jwt`. تمنع المساحة الاسمية
الاصطدامات عند تحميل سياسات مخصصة أو من طرف ثالث بأسماء قصيرة متشابهة.

في إعدادك، يمكنك الإشارة إلى سياسة مدمجة باستخدام اسمها القصير أو اسمها المؤهل؛ كلا الشكلين يحل إلى نفس السياسة:

```json
{
  "enabledPolicies": [
    "sanitize-jwt",
    "exospherehost/block-rm-rf"
  ]
}
```

إذا لم يكن للاسم `/`، يعامل failproofai كأنه ينتمي إلى مساحة الاسم الافتراضية `exospherehost`. الأسماء التي تحتوي بالفعل على `/` (مثل `myorg/foo`،
`custom/my-hook`) تُحتفظ بها كما هي.
- **`require-`** — حجب حدث Stop حتى تتحقق الشروط.

---

<Tip>
تدعم كل سياسة حقل `hint` اختياري في `policyParams`. يُلحق التلميح برسالة الرفض أو الإرشاد التي يراها Claude، مما يوفر إرشادات قابلة للتنفيذ دون تعديل كود السياسة. يعمل مع السياسات المدمجة والمخصصة والاتفاقية. انظر [الإعدادات → hint](/ar/configuration#hint-cross-cutting) للمزيد من التفاصيل.
</Tip>

## الأوامر الخطرة

منع الوكلاء من تشغيل العمليات التي يصعب التراجع عنها أو التي قد تضر نظام المضيف.

### `block-sudo`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أي أمر `sudo`.

يحجب الاستدعاءات التي تتضمن كلمة `sudo`. يتم مطابقة النمط على رموز الأوامر المحللة، وليس السلسلة الخام، لمنع الالتفاف عبر حقن مشغل shell.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات الأوامر الدقيقة المسموحة. تُطابق كل إدخالة ضد رموز argv المحللة. |

**مثال:**

```json
{
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    }
  }
}
```

مع هذا الإعداد، `sudo systemctl status nginx` مسموح، لكن `sudo rm /etc/hosts` مرفوض.

<Note>
تُطابق الأنماط ضد الرموز المحللة، وليس سلسلة الأوامر الخام. هذا يمنع الالتفاف عبر مشغلات shell الملحقة (مثل `sudo systemctl status x; rm -rf /` لا تطابق `sudo systemctl status *`).
</Note>

---

### `block-rm-rf`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `rm -rf`, `rm -fr`، والأشكال الحذف العودية المماثلة.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | المسارات الآمنة للحذف العودي (مثل `/tmp`). |

**مثال:**

```json
{
  "policyParams": {
    "block-rm-rf": {
      "allowPaths": ["/tmp", "/var/cache"]
    }
  }
}
```

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `curl <url> | bash`، `curl <url> | sh`، `wget <url> | bash`، والأنماط المماثلة.

**الحدث:** PreToolUse (Bash)

## أوامر البنية التحتية

منع وكلاء الترميز من تشغيل CLIs البنية التحتية أو تشغيل خطوط أنابيب CI/CD. جميع السياسات في هذه الفئة **اختيارية** (`defaultEnabled: false`) — الوكلاء الذين يحتاجون حقًا لاستدعاء `kubectl`، `terraform`، وما إلى ذلك لن يتعرضوا للانزعاج إلا إذا قمت بتفعيل السياسة. عند التفعيل، يُرفض كل استدعاء للـ CLI المتطابق ما لم تطابق الأمر إدخالًا في `allowPatterns`.

نحو النمط هو نفسه [`block-sudo`](#block-sudo): تُطابق الرموز ضد argv المحلل، `*` هو بطاقة برية لرمز واحد، وأي أمر يحتوي على مشغل shell مستقل (`&&`, `||`, `|`, `;`) أو رمز يحتوي على أحرف metacharacter shell مدمجة يُرفض قبل مطابقة القائمة البيضاء لمنع التجاوزات بالحقن.

### `block-kubectl`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أي استدعاء `kubectl`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | بادئات أوامر kubectl المسموحة. |

**مثال:**

```json
{
  "policyParams": {
    "block-kubectl": {
      "allowPatterns": ["kubectl get *", "kubectl describe *", "kubectl logs *"]
    }
  }
}
```

مع هذا الإعداد، `kubectl get pods` مسموح لكن `kubectl apply -f deploy.yaml` مرفوض.

### `block-gh-pipeline`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض أوامر فرعية `gh` CLI التالية التي تغير الحالة أو تشغل خطوط الأنابيب:

- `gh workflow run`, `gh workflow enable`, `gh workflow disable`
- `gh run rerun`, `gh run cancel`
- `gh pr merge`
- `gh release create`, `gh release delete`
- `gh cache delete`
- `gh secret set`, `gh secret delete`

أوامر فرعية `gh` للقراءة فقط مثل `gh pr view`, `gh pr list`, `gh run list`, `gh release view`, و `gh api repos/.../...` **لا تُطابق** بهذه السياسة — فهي ضرورية بشكل روتيني لفحوصات سير العمل (بما في ذلك `require-ci-green-before-stop` الخاص بـ failproofai).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPatterns` | `string[]` | `[]` | استدعاءات مرمجة محددة للسماح بها رغم أنها ستُرفض بخلاف ذلك. |

**مثال:**

```json
{
  "policyParams": {
    "block-gh-pipeline": {
      "allowPatterns": ["gh run rerun *"]
    }
  }
}
```

## الأسرار (المنظفات)

منع الوكلاء من تسرب بيانات الاعتماد إلى سياقهم أو مخرجاتهم. تعمل سياسات المنظف على أحداث **PostToolUse**. عندما يقوم Claude بتشغيل أمر Bash أو قراءة ملف أو استدعاء أي أداة، تفحص هذه السياسات المخرجات قبل إرجاعها إلى Claude. إذا تم اكتشاف نمط سري، تُرجع السياسة قرار رفض يمنع المخرجات من التمرير للخلف.

### `sanitize-jwt`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف رموز JWT (ثلاث قطاعات base64url مفصولة بـ `.`).

لا توجد معاملات.

---

### `sanitize-api-keys`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف تنسيقات مفاتيح API الشائعة: Anthropic (`sk-ant-`), OpenAI (`sk-`), GitHub PATs (`ghp_`), AWS access keys (`AKIA`), Stripe keys (`sk_live_`, `sk_test_`), و Google API keys (`AIza`).

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `{ regex: string; label: string }[]` | `[]` | أنماط regex إضافية تُعامل كأسرار. |

**مثال:**

```json
{
  "policyParams": {
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo internal API key" },
        { "regex": "pat_[0-9a-f]{40}", "label": "Internal PAT" }
      ]
    }
  }
}
```

---

### `sanitize-connection-strings`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف سلاسل اتصال قاعدة البيانات التي تحتوي على بيانات اعتماد مدمجة (مثل `postgresql://user:password@host/db`).

لا توجد معاملات.

---

### `sanitize-private-key-content`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف كتل PEM (`-----BEGIN PRIVATE KEY-----`, `-----BEGIN RSA PRIVATE KEY-----`، إلخ).

لا توجد معاملات.

---

### `sanitize-bearer-tokens`

**الحدث:** PostToolUse (جميع الأدوات)  
**الافتراضي:** يحذف رؤوس `Authorization: Bearer <token>` حيث الرمز 20 حرف أو أكثر.

لا توجد معاملات.

## البيئة

حماية إعدادات البيئة الحساسة من قراءتها أو كشفها من قبل الوكلاء.

### `block-env-files`

**الحدث:** PreToolUse (Bash, Read)  
**الافتراضي:** يرفض قراءة ملفات `.env` عبر `cat .env`, استدعاءات أداة Read بـ `.env` كمسار الملف، وما إلى ذلك.

لا يحجب `.envrc` أو ملفات أخرى مجاورة للبيئة — فقط الملفات المسماة بالضبط `.env`.

لا توجد معاملات.

---

### `protect-env-vars`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض الأوامر التي تطبع متغيرات البيئة: `printenv`, `env`, `echo $VAR`.

لا توجد معاملات.

---

## الوصول إلى الملفات

إبقاء الوكلاء يعملون داخل حدود المشروع وبعيدًا عن الملفات الحساسة.

### `block-read-outside-cwd`

**الحدث:** PreToolUse (Read, Bash)  
**الافتراضي:** يرفض قراءة الملفات خارج جذر المشروع. الحد هو `CLAUDE_PROJECT_DIR` (مضبوط مرة واحدة في الجلسة بواسطة Claude Code)، مع عودة إلى دليل العمل الحالي للجلسة عند عدم تعيين هذا المتغير. استخدام جذر المشروع بدلاً من `cwd` المباشر يعني أن الحد يبقى مستقرًا حتى بعد قيام Claude بـ `cd` إلى دليل فرعي.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `allowPaths` | `string[]` | `[]` | بادئات المسارات المطلقة المسموحة حتى خارج جذر المشروع. |

**مثال:**

```json
{
  "policyParams": {
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company/config"]
    }
  }
}
```

---

### `block-secrets-write`

**الحدث:** PreToolUse (Write, Edit)  
**الافتراضي:** يرفض الكتابة إلى الملفات المستخدمة عادة للمفاتيح الخاصة والشهادات: `id_rsa`, `id_ed25519`, `*.key`, `*.pem`, `*.p12`, `*.pfx`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `additionalPatterns` | `string[]` | `[]` | أنماط أسماء ملفات إضافية (نمط glob) للحجب. |

**مثال:**

```json
{
  "policyParams": {
    "block-secrets-write": {
      "additionalPatterns": [".token", ".secret"]
    }
  }
}
```

## Git

منع الدفع العرضي والدفع القسري وأخطاء الفرع التي يصعب التراجع عنها.

### `block-push-master`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git push origin main` و `git push origin master`.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي لا يمكن الدفع إليها مباشرة. |

**مثال:**

```json
{
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "master", "release", "prod"]
    }
  }
}
```

<Tip>
للسماح بالدفع إلى جميع الفروع (تعطيل هذه السياسة فعليًا دون إزالتها من `enabledPolicies`)، اضبط `protectedBranches: []`.
</Tip>

---

### `block-work-on-main`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git commit`, `git merge`, `git rebase`, و `git cherry-pick` بينما شجرة العمل على `main` أو `master`. إنشاء الفرع والتبديل (`git checkout`, `git checkout -b`, `git switch`, `git switch -c`) لا تتأثر.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `protectedBranches` | `string[]` | `["main", "master"]` | أسماء الفروع التي يُرفض عليها الالتزام/الدمج/إعادة الأساس/اختيار الكرز. |

---

### `block-force-push`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرفض `git push --force` و `git push -f`.

لا توجد معاملات خاصة بالسياسة. استخدم [`hint`](/ar/configuration#hint-cross-cutting) عبر القطع لاقتراح بدائل:

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Create a new branch from your current HEAD (e.g. `git checkout -b <new-branch>`) and push that instead."
    }
  }
}
```

---

### `warn-git-amend`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude للمتابعة بحذر عند تشغيل `git commit --amend`. لا يحجب الأمر.

لا توجد معاملات.

---

### `warn-git-stash-drop`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude لتأكيد قبل تشغيل `git stash drop`. لا يحجب الأمر.

لا توجد معاملات.

---

### `warn-all-files-staged`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude لمراجعة ما يتم تجميعه عند تشغيل `git add -A` أو `git add .`. لا يحجب الأمر.

لا توجد معاملات.

## قاعدة البيانات

اكتشاف العمليات المدمرة في SQL قبل تنفيذها على قاعدة البيانات.

### `warn-destructive-sql`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** يرشد Claude لتأكيد قبل تشغيل SQL يحتوي على `DROP TABLE`, `DROP DATABASE`, أو `DELETE` بدون جملة `WHERE`.

لا توجد معاملات.

---

### `warn-schema-alteration`

## مديري الحزم

فرض أي مدير حزم يُسمح للوكيل باستخدامه.

### `prefer-package-manager`

**الحدث:** PreToolUse (Bash)  
**الافتراضي:** معطّل. عند التفعيل، يحجب أي أمر مدير حزم ليس في قائمة `allowed` ويخبر Claude لإعادة كتابة الأمر باستخدام مدير مسموح.

يكتشف: pip, pip3, python -m pip, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo.

| المعامل | النوع | الافتراضي | الوصف |
|-----------|------|---------|-------------|
| `allowed` | string[] | `[]` | أسماء مديري الحزم المسموحة. يُحجب أي مدير مكتشف ليس في هذه القائمة. عند أن تكون فارغة، السياسة لا تفعل شيء. |
| `blocked` | string[] | `[]` | أسماء مديري حزم إضافية للحجب بعد القائمة المدمجة (مثل `['pdm', 'pipx']`). |

قائمة الحجب المدمجة تغطي: pip, pip3, npm, npx, yarn, pnpm, pnpx, bun, bunx, uv, poetry, pipenv, conda, cargo. استخدم `blocked` لإضافة مديرين ليسوا في هذه القائمة.

**مثال الإعدادات:**

```json
{
  "enabledPolicies": ["prefer-package-manager"],
  "policyParams": {
    "prefer-package-manager": {
      "allowed": ["uv", "bun"],
      "blocked": ["pdm", "pipx"]
    }
  }
}
```

مع هذا الإعداد، `pip install flask` و `pdm install flask` كلاهما مرفوض برسالة تخبر Claude لاستخدام `uv` أو `bun` بدلاً من ذلك. أوامر مثل `uv pip install flask` مسموحة لأن `uv` في قائمة السماح ويُفحص أولاً.

## سير العمل

فرض سير عمل منضبط في نهاية الجلسة. تعمل هذه السياسات على حدث **Stop** وترفض Claude من التوقف حتى يتم استيفاء كل شرط. تتبع سلسلة اعتماد طبيعية: التزام → دفع → طلب دمج → CI. إذا رفضت سياسة، يتم تخطي السياسات اللاحقة في السلسلة (الرفض يقصر الدارة).

جميع سياسات سير العمل **فتح الفشل**: إذا لم تكن الأداة المطلوبة متاحة (مثل `gh` غير مثبت، لا جهاز بعيد)، تسمح السياسة برسالة معلوماتية تشرح السبب في تخطي الفحص.

### `require-commit-before-stop`

**الحدث:** Stop  
**الافتراضي:** يرفض التوقف عندما تكون هناك تغييرات غير مرتكبة (ملفات معدلة أو مرحلة أو غير متتبعة). يُرجع رسالة معلوماتية عندما يكون دليل العمل نظيفًا.

لا توجد معاملات.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `remote` | `string` | `"origin"` | اسم الجهاز البعيد للدفع إليه. |

**مثال:**


لا توجد معاملات.

<Note>

طلبات الدمج. إذا لم يتم تثبيت `gh` أو لم يتم المصرح به، تفشل السياسة المفتوحة وتبلغ السبب إلى Claude.

### `require-no-conflicts-before-stop`

**الحدث:** Stop  
**الافتراضي:** يرفض التوقف عندما لا يستطيع الفرع الحالي الدمج بنظافة في الفرع الأساسي. تؤكد السياسة أولاً وجود طلب دمج `OPEN` على GitHub للفرع — بدون واحد، لا يوجد هدف دمج للفرض عليه، لذلك تقصر السياسة بأكملها للسماح. بمجرد تأكيد طلب دمج `OPEN`، يعمل مسحان مستقلان:

1. **محلي** — `git merge-tree --write-tree --name-only origin/<baseBranch> HEAD`. عند التضارب، تسمي رسالة الرفض الملفات المتضاربة حتى يعرف Claude بالضبط ما يجب حله.
2. **GitHub** — يعيد استخدام نتيجة `gh pr view --json mergeable,state` المجلوبة بالفعل في المسح المسبق. يلتقط التضاربات التي قد يفتقدها `origin/<baseBranch>` الراكدة محليًا (مثل هبوط شخص ما طلب دمج متضارب على `main` منذ الجلب الأخير). نتيجة `CONFLICTING` ترفض. نتيجة `UNKNOWN` ترفض أيضًا وترشد Claude للانتظار ~10 ثوان وإعادة الفحص قبل محاولة التوقف مرة أخرى — هذا يمنع الإيجابيات الكاذبة بينما تعيد GitHub حسابها.

تقصر بالكامل (تسمح) عندما: `gh` غير مثبت، لا يوجد طلب دمج للفرع، حالة طلب الدمج ليست `OPEN` (مثل `MERGED`, `CLOSED`)، أو `gh pr view` يُرجع إخراجًا غير قابل للتحليل. يفشل المفتوح أيضًا عندما `origin/<baseBranch>` مفقود محليًا أو عندما لا تكون هناك التزامات في المقدمة من الأساس — تلك الرقائق المتساقطة في الطبقة 1 تستشير دائمًا قابلية الدمج المخبأة لطلب الدمج قبل السماح.

**المعاملات:**

| المعامل | النوع | الافتراضي | الوصف |
|-------|------|---------|-------------|
| `baseBranch` | `string` | `"main"` | الفرع الأساسي للفحص للتضاربات ضده. |

<Note>
GitHub CLI (`gh`) مطلوب لهذه السياسة. تستخدم السياسة `gh pr view` لتأكيد
وجود طلب دمج `OPEN` قبل تشغيل أي مسح تضارب — بدون `gh`، تقصر السياسة للسماح. شغّل `gh auth login` برمز وصول شخصي له
نطاق `repo` للوصول بقراءة إلى طلبات الدمج.
</Note>

## تعطيل السياسات الفردية

أزل سياسة محددة من `enabledPolicies` في إعدادك، أو بدّل إيقافها في علامة التبويب السياسات في لوحة التحكم.

```json
{
  "enabledPolicies": [
    "block-rm-rf",
    "sanitize-api-keys"
  ]
}
```

السياسات غير المدرجة في `enabledPolicies` لا تعمل، حتى لو كانت إدخالات `policyParams` موجودة لها.
