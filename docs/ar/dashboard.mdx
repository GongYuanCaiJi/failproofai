لوحة التحكم failproofai هي تطبيق ويب محلي لمراقبة جلسات وكلائك الذكيين وإدارة السياسات. شاهد ما فعله وكلاؤك أثناء غيابك.

---

## بدء لوحة التحكم

```bash
failproofai
```

يتم فتحها على `http://localhost:8020`.

تقرأ لوحة التحكم مباشرة من نظام الملفات - مجلدات مشاريع Claude Code وملفات إعدادات failproofai. لا يتم كتابة أي شيء إلى خدمة بعيدة.

---

## الصفحات

### المشاريع

تسرد جميع مشاريع Claude Code و OpenAI Codex و GitHub Copilot CLI _(beta)_ و Cursor Agent _(beta)_ و OpenCode _(beta)_ و Pi _(beta)_ و Gemini CLI _(beta)_ الموجودة على جهازك. يتم اكتشاف مشاريع Claude من `~/.claude/projects/` (أو المسار المحدد بواسطة `CLAUDE_PROJECTS_PATH`)؛ يتم اكتشاف مشاريع Codex بمسح كل نسخة احتياطية تحت `~/.codex/sessions/<YYYY>/<MM>/<DD>/*.jsonl` وتجميعها حسب `cwd` المسجل في السجل الأول لكل جلسة؛ يتم اكتشاف مشاريع Copilot CLI بمسح كل `~/.copilot/session-state/<sessionId>/workspace.yaml` (قابل للتكوين عبر `COPILOT_HOME`) وتجميعها حسب حقل `cwd`؛ يتم اكتشاف مشاريع Cursor Agent بمسح البيانات الوصفية لكل جلسة تحت `~/.cursor/agent-sessions/<sessionId>/` (قابل للتكوين عبر `CURSOR_HOME`، مع اختبار `conversations/` و `sessions/` كخيارات بديلة) بحثاً عن عددية `cwd` في `meta.json` / `session.json` / `workspace.yaml`؛ يتم اكتشاف مشاريع OpenCode من خلال الاستعلام عن قاعدة بيانات SQLite الخاصة بها على `~/.local/share/opencode/opencode.db` عبر `opencode db --format json` (نقرأ جداول `session` و `project` وتجميعها حسب `project_id`)؛ يتم اكتشاف مشاريع Pi بمسح نسخ احتياطية JSONL لكل جلسة تحت `~/.pi/agent/sessions/<encoded-cwd>/<timestamp>_<uuid>.jsonl` (قابل للتكوين عبر `PI_SESSIONS_DIR`) وسحب `cwd` من السجل الأول لكل جلسة؛ يتم اكتشاف مشاريع Gemini CLI بمسح `~/.gemini/tmp/<basename>/chats/session-<timestamp>-<uuid-prefix>.jsonl` (قابل للتكوين عبر `GEMINI_SESSIONS_DIR`) واسترجاع cwd القانوني من علامة نص `.project_root` المجاورة. يتم عرض المشروع الذي تم استخدامه من قبل عدة CLIs كصف واحد مع جميع الشارات المطابقة. استخدم القائمة المنسدلة **CLI** أعلى الجدول للتصفية حسب CLI وكيل محدد؛ يحفظ عنوان URL اختيارك كـ `?cli=claude|codex|copilot|cursor|opencode|pi|gemini`.

يعرض كل مشروع:
- اسم المشروع (مشتق من مسار المجلد)
- شارة CLI — `Claude Code` (برتقالي)، `OpenAI Codex` (بنفسجي)، `GitHub Copilot` (أزرق)، `Cursor Agent` (أخضر زمردي)، `OpenCode` (كهرماني)، `Pi` (وردي)، و/أو `Gemini CLI` (سماوي)
- تاريخ آخر نشاط جلسة

انقر على مشروع لرؤية جلساته.

### الجلسات

تسرد جميع الجلسات داخل المشروع. تعرض كل جلسة:
- معرّف الجلسة
- طوابع زمنية البداية والنهاية
- عدد استدعاءات الأدوات
- عدد أنشطة الخطاف (السياسات التي تم تطبيقها)

استخدم مرشح نطاق التاريخ وبحث معرّف الجلسة لتضييق القائمة. يتم تقسيم الجلسات إلى صفحات.

انقر على جلسة لفتح عارض الجلسة.

### عارض الجلسة

يجيب عارض الجلسة على السؤال الرئيسي للوكلاء المستقلين: ماذا فعل الوكيل، وهل ظل على المسار الصحيح؟ شارة CLI بجانب الرأس تشير إلى ما إذا كانت الجلسة نسخة Claude Code أو OpenAI Codex أو GitHub Copilot CLI أو Cursor Agent أو OpenCode أو Pi أو Gemini CLI. يعرض خطاً زمنياً لكل شيء حدث في الجلسة:

- **الرسائل** - استجابات Claude النصية والمطالبات من المستخدم
- **استدعاءات الأدوات** - كل أداة استدعاها Claude، مع إدخالها ومخرجاتها
- **نشاط السياسة** - لكل استدعاء أداة، السياسات التي تم تطبيقها والقرار الذي أرجعته

يعرض شريط الإحصائيات في الأعلى مدة الجلسة والعدد الإجمالي لاستدعاءات الأدوات وملخصاً لقرارات الخطاف (عدد السماح / الرفض / التعليمات).

انقر على زر **تنزيل السجلات** لتصدير الجلسة. للجلسات Claude Code و Codex و Copilot و Cursor و Pi و Gemini تحصل على نسخة JSONL الأصلية على القرص بايت بايت؛ للجلسات OpenCode (التي تعيش في SQLite وليس على القرص) تحصل على وثيقة JSON تعكس جداول `session` و `messages` و `parts` الأساسية.

### التدقيق

تقرير مدفوع بالشخصية عن كيفية تصرف وكيلك فعلياً عبر الجلسات السابقة. يشغل نفس المسح مثل CLI `failproofai audit` لكنه يعرضه كلوحة تحكم بستة أقسام:

1. **الهوية** — تصنف وكيلك إلى أحد 8 نماذج أولية (`the optimist` و `the cowboy` و `the explorer` و `the goldfish` و `the paranoid architect` و `the precision builder` و `the hammer` و `the ghost`) بناءً على الكواشف والسياسات التي تم تطبيقها ومدى قوتها. يعرض شعاراً بحجم 8×8 بكسل والوسم الأولي وإطار `common in` / `primary risk` والبيان الختامي الواحد.
2. **تباهى بوكيلك** — يلتقط بطاقة الهوية كـ PNG بحجم 1200×630 مناسب للنشر على X / LinkedIn (انقر على `make poster`).
3. **نقاط القوة** — السلوكيات المؤشرة بالأخضر التي يقوم بها وكيلك بالفعل بشكل صحيح، المشتقة من بيانات التدقيق المباشرة (معدل استدعاء أداة نظيف، متوسط طول الجلسة، صفر تسريب بيانات الاعتماد، صفر عواصف إعادة المحاولة، إلخ.).
4. **النقاط + جدول الترتيب** — درجة من 0-100 مع درجة حرف (S/A/B/C/D/F)، رسم بياني للتوزيع يظهر مكان وقوعك في المجموعة، نثر (`a B starts at 71. you're 13 points away.`)، وجدول لوحة الترتيب مع تمييز صفك.
5. **النتائج** — بطاقات لكل نتيجة مرتبة حسب التأثير. تعرض كل بطاقة ما حدث والتكلفة وعينة دليل بأوامر فعلية تم التقاطها وسياسة failproofai التي ستمسك بنفس النمط (`$ failproofai policy add <slug>`، انقر للنسخ).
6. **السياسات المقترحة + حلقة العودة** — شبكة لكل سياسة مدمجة غير مفعلة ستغلق فجوة، مع شرح النقاط المتوقعة، بالإضافة إلى CTA `re-audit in 7 days`.

مدفوع بوقت التشغيل `failproofai audit` — انظر [Audit CLI](/ar/cli/audit) لمحرك المسح الأساسي والأعلام المدعومة وثوابت الذاكرة المؤقتة لكل نسخة احتياطية. تخزن لوحة التحكم آخر نتيجة مؤقتة في `~/.failproofai/audit-dashboard.json` (الوضع `0600`، فتحة واحدة، التشغيلات الجديدة تستبدل) بحيث تكون الزيارات اللاحقة فورية؛ **يتم رفض كل من ذاكرة النسخة الاحتياطية لكل نسخة احتياطية والنتيجة الكاملة عند القراءة بمجرد أن تصبح أقدم من 7 أيام** لذلك لن تعرض لوحة التحكم بصمت نتيجة قديمة بأسبوع. يظهر شريط في أعلى الصفحة دائماً يعرض آخر طابع زمني للتشغيل (`audited 3d ago`)، شريحة كهرمانية `expires in 14h — re-audit to refresh` في آخر 24 ساعة من النافذة، وشعار `audit expired` بعد TTL. عند النقر على أي من `[ re-audit ]` (الأعلى) أو `[ re-audit now ]` (الأسفل) POSTs إلى `/api/audit/run` ولوحة التحكم تجري استطلاع `/api/audit/status` بمعدل 1Hz حتى ينتهي التشغيل؛ شريط تقدم وردي لاصق يثبت في أعلى منفذ العرض أثناء التشغيل مع مؤقت قضى، والنتيجة تتبادل في مكانها عند النجاح (لا يوجد إعادة تحميل صفحة كاملة). عند الفشل يتحول الشريط إلى أحمر مع نسخ مفتاح تغيير حسب `RerunError.kind` (`timeout` / `network` / `post_failed`). تُعرض حالة فارغة (لا توجد ذاكرة مؤقتة) وحالة جلسات صفرية (توجد ذاكرة مؤقتة ولكن المسح لم يجد نسخاً احتياطية) بشكل منفصل.

## الإنعاش التلقائي

تحتوي لوحة التحكم على مبدل الإنعاش التلقائي في التنقل العلوي. عند تفعيله، يتم تحديث الصفحة الحالية بشكل دوري لعرض الجلسات الجديدة ونشاط السياسة كما يظهر. ضروري لمراقبة جلسات الوكيل المستقلة طويلة الأجل.

---

## تعطيل الصفحات

إذا كنت بحاجة فقط إلى بعض أجزاء لوحة التحكم، اضبط `FAILPROOFAI_DISABLE_PAGES` على قائمة مفصولة بفواصل من أسماء الصفحات:

```bash
FAILPROOFAI_DISABLE_PAGES=policies failproofai
```

القيم الصحيحة: `policies` و `projects` و `audit`.

---

## تكوين مسار المشاريع

بشكل افتراضي، تقرأ لوحة التحكم من دليل مشاريع Claude Code القياسي. تجاوزه للإعدادات المخصصة:

```bash
CLAUDE_PROJECTS_PATH=/custom/path/to/projects failproofai
```

## الوصول من مضيف غير localhost

عند تشغيل لوحة التحكم في **وضع التطوير** (`npm run dev`) والوصول إليها من اسم مضيف بخلاف `localhost` - على سبيل المثال، مجال مخصص أو عنوان IP بعيد أو عنوان URL متنقل - قد تشاهد تحذيراً مثل:

```text
⚠ Blocked cross-origin request to Next.js dev resource /_next/webpack-hmr from "dashboard.example.com".
```

هذا هو Next.js يحظر الوصول عبر الأصول إلى مورد HMR (hot module reload) الخاص به، وهو ميزة فقط للتطوير. للسماح بمضيفك، استخدم العلم `--allowed-origins`:

```bash
npm run dev -- --allowed-origins dashboard.example.com
```

لعدة مضيفات أو عناوين IP، مرر قائمة مفصولة بفواصل:

```bash
npm run dev -- --allowed-origins dashboard.example.com,192.168.1.5
```

يمكنك أيضاً تعيين متغير البيئة `FAILPROOFAI_ALLOWED_DEV_ORIGINS` بدلاً من ذلك:

```bash
FAILPROOFAI_ALLOWED_DEV_ORIGINS=dashboard.example.com npm run dev
```

<Note>
ينطبق هذا فقط على وضع التطوير. عند تشغيل `failproofai` (وضع الإنتاج)، لا توجد مقبس websocket HMR ولا مشكلة مورد مشترك الأصل.
</Note>
