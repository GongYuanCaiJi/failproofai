  قد يتغير كتالوج الكاشف وتنسيق التقرير قبل الإصدار المستقر التالي. يرجى فتح مشكلة إذا بدا شيء ما غير صحيح.
</Note>

يتم الآن عرض التدقيق كـ **صفحة لوحة معلومات /audit**، وليس كأمر CLI فرعي. افتحها من شريط التنقل في لوحة المعلومات (بين السياسات والمشاريع)، أو قم بزيارة `http://localhost:8020/audit` مباشرة عند تشغيل `failproofai` محليًا.

```bash
failproofai          # open the dashboard, then click "Audit"
```

تمسح لوحة المعلومات نصوص وكيل CLI السابقة على هذا الجهاز (Claude Code، Codex، Copilot، Cursor، OpenCode، Pi، Gemini) وتقرر عدد المرات التي قام فيها الوكيل بأشياء تم بناء failproofai لإيقافها — فحوصات متغيرات البيئة، الدفع القسري، بادئات `cd <cwd>` الزائدة، حلقات sleep-polling، إعادة قراءة الملفات المعدلة للتو، والمزيد.

بالنسبة لكل نص، يتم تشغيل كل حدث استخدام أداة من خلال 39 سياسة مدمجة **و** من خلال 8 كواشف خاصة بالتدقيق تلتقط الأنماط التي لم تغطها السياسات في وقت التشغيل بعد. يتم تجميع الأعداد لكل سياسة / كاشف عبر جميع الجلسات.

## ما الذي تحصل عليه

تتكون صفحة `/audit` من ستة أقسام:

1. **الهوية** — يتم تصنيف وكيلك إلى أحد 8 نماذج أولية (`optimist`، `cowboy`، `explorer`، `goldfish`، `paranoid architect`، `precision builder`، `hammer`، `ghost`) بناءً على الإشارة المرجحة عبر كل نص تم تدقيقه.
2. **نقاط القوة** — أرقام حقيقية مستمدة من الفحص (نسبة المكالمات النظيفة %، "0 credential leaks"، وما إلى ذلك) محدودة بسياسات sanitize ذات الصلة التي تعمل بالفعل.
3. **النتيجة** — 0-100 مع نطاقات S/A/B/C/D/F والرفع المتوقع إذا تم تمكين كل سياسة موصى بها.
4. **النتائج** — بطاقات لكل سياسة توضح ما حدث والتكلفة والأدلة المكتشفة وأمر `failproofai policy add <slug>` الدقيق لتمكين المدمج في الوقت الفعلي الذي كان سيمسكه.
5. **السياسات الموصى بها** — قائمة التثبيت المجمعة مع أمر `failproofai policies --install` أحادي الرمية.
6. **تذكير إعادة التدقيق** — "عد بشكل أفضل." اضبط تذكيرًا عبر البريد الإلكتروني لمدة 7 أيام عبر api-server (يتطلب تسجيل الدخول؛ انظر [`failproofai auth`](/ar/cli/auth)).

## كواشف خاصة بالتدقيق

تكتشف أنماط السلوك "الغبي" التي لم يتم فرضها (بعد) في الوقت الفعلي. تعمل فقط أثناء التدقيق ولا تحظر أبدًا استدعاء أداة مباشر.

| الكاشف | ما الذي يعده |
|---|---|
| `redundant-cd-cwd` | أوامر Bash تبدأ بـ `cd <cwd> && …` حتى لو كانت الأوامر تعمل بالفعل في `cwd`. |
| `prefer-edit-over-read-cat` | `cat`/`head`/`tail`/`less`/`more` على ملف مصدر واحد — استخدم أداة `Read`. |
| `prefer-edit-over-sed-awk` | تحريرات `sed -i` / `awk … > file` المضمنة — استخدم أداة `Edit`. |
| `prefer-write-over-heredoc` | Heredoc / كتابة ملفات `echo > file` متعددة الأسطر — استخدم أداة `Write`. |
| `sleep-polling-loop` | `sleep N` طويل (≥ 30s) أو حلقات استقصاء `while …; sleep …; done`. |
| `find-from-root` | `find /`، `find /home`، `find /usr`، إلخ. — قصر النطاق على `cwd` بدلاً من ذلك. |
| `git-commit-no-verify` | `git commit … --no-verify` / `-n`، تخطي الخطافات. |
| `reread-after-edit` | `Read` لملف تم `Edit`/`Write` للتو في نفس الجلسة. |

## الذاكرات المؤقتة

- **ذاكرة مؤقتة لكل نص** في `~/.failproofai/cache/audit/<sha1>.json` مفهرسة حسب `(mtime, size, engineVersion, detectorVersion)` — تبطل تلقائيًا عند تغيير النص أو رمز السياسة/الكاشف. تخزن كل إدخالة أيضًا طابع وقت `cachedAt` كـ **بيانات وصفية TTL** (ليست جزءًا من مفتاح الذاكرة المؤقتة)؛ يتم رفض الإدخالات الأقدم من **7 أيام** عند القراءة حتى لا تستمر النتائج طويلة العمر في تجاوز نية الكاشف المتطورة.
- **ذاكرة مؤقتة للنتيجة الكاملة** في `~/.failproofai/audit-dashboard.json` (الوضع 0600). يسمح بعرض لوحة المعلومات فورًا عند التنقل دون إعادة التشغيل. يتم رفضه أيضًا عند القراءة بعد **TTL لمدة 7 أيام** — ثم ينزلق `/audit` إلى الحالة الفارغة مع بافتة `audit expired — re-audit to refresh`. انقر على `[ re-audit ]` في أعلى الصفحة (أو `[ re-audit now ]` بالقرب من الأسفل) للتحديث؛ يقوم التشغيل الجديد بدفق التقدم عبر شريط ثابت في الأعلى ويبدل النتيجة في المكان عند النجاح (بدون إعادة تحميل الصفحة).

## ملاحظات

- **لا يوجد طفرة.** يتم إعادة التدقيق في الوضع للقراءة فقط. يتم تخطي `warn-repeated-tool-calls` لأن الملف الجانبي لكل جلسة سيتم تعديله بخلاف ذلك.
- **سياسات سير العمل المتخطاة.** سياسات `require-*-before-stop` تعمل فقط على أحداث `Stop` و `execSync` مقابل حالة git المباشرة — ليس لديها تفسير معنوي "ماذا كان سيحدث في 2025"، لذا لا تظهر في أعداد التدقيق.
- **السياسات المخصصة المتخطاة.** لا يتم إعادة تشغيل الخطافات المخصصة المرسلة من المستخدم (قد تكون قد تغيرت منذ الجلسة الأصلية).
