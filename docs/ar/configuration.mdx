يستخدم failproofai ملفات إعدادات JSON للتحكم في السياسات النشطة وكيفية تعاملها وأين يتم تحميل السياسات المخصصة. تم تصميم التكوين ليكون سهل المشاركة مع فريقك - التزم به في مستودع الكود والكل المطورين سيحصلون على نفس شبكة الأمان للعامل.

---

## نطاقات التكوين

هناك ثلاثة نطاقات تكوين، يتم تقييمها بترتيب الأولوية:

| النطاق | مسار الملف | الغرض |
|-------|-----------|-------|
| **المشروع** | `.failproofai/policies-config.json` | إعدادات لكل مستودع، مرتكبة لعناصر التحكم في الإصدار |
| **محلي** | `.failproofai/policies-config.local.json` | تجاوزات شخصية لكل مستودع، مدرجة في gitignore |
| **عام** | `~/.failproofai/policies-config.json` | الإعدادات الافتراضية على مستوى المستخدم عبر جميع المشاريع |

عندما يتلقى failproofai حدث hook، يقوم بتحميل ودمج جميع الملفات الثلاثة الموجودة للدليل العامل الحالي.

### قواعد الدمج

**`enabledPolicies`** - اتحاد جميع النطاقات الثلاثة. أي سياسة مفعلة على أي مستوى تكون نشطة.

```text
project:  ["block-sudo"]
local:    ["block-rm-rf"]
global:   ["block-sudo", "sanitize-api-keys"]

resolved: ["block-sudo", "block-rm-rf", "sanitize-api-keys"]  ← deduplicated union
```

**`policyParams`** - يكون أول نطاق يحدد المعاملات لسياسة معينة هو الفائز تماماً. لا يوجد دمج عميق للقيم داخل معاملات السياسة.

```text
project:  block-sudo → { allowPatterns: ["sudo apt-get update"] }
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo apt-get update"] }   ← project wins, global ignored
```

```text
project:  (no block-sudo entry)
local:    (no block-sudo entry)
global:   block-sudo → { allowPatterns: ["sudo systemctl status"] }

resolved: { allowPatterns: ["sudo systemctl status"] }  ← falls through to global
```

**`customPoliciesPath`** - يكون أول نطاق يحدده هو الفائز.

**`llm`** - يكون أول نطاق يحدده هو الفائز.

## صيغة ملف الإعدادات

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "sanitize-jwt",
    "block-env-files",
    "block-read-outside-cwd"
  ],
  "policyParams": {
    "block-sudo": {
      "allowPatterns": ["sudo systemctl status", "sudo journalctl"]
    },
    "block-push-master": {
      "protectedBranches": ["main", "release", "prod"]
    },
    "block-rm-rf": {
      "allowPaths": ["/tmp"]
    },
    "block-read-outside-cwd": {
      "allowPaths": ["/shared/data", "/opt/company"]
    },
    "sanitize-api-keys": {
      "additionalPatterns": [
        { "regex": "myco_[A-Za-z0-9]{32}", "label": "MyCo API key" }
      ]
    },
    "warn-large-file-write": {
      "thresholdKb": 512
    }
  },
  "customPoliciesPath": "/home/alice/myproject/my-policies.js"
}
```

## مرجع الحقول

### `enabledPolicies`

النوع: `string[]`

قائمة أسماء السياسات المراد تفعيلها. يجب أن تطابق الأسماء بالضبط معرّفات السياسات التي يظهرها `failproofai policies`. انظر [السياسات المدمجة](/ar/built-in-policies) للحصول على القائمة الكاملة.

السياسات غير المدرجة في `enabledPolicies` غير نشطة، حتى لو كانت لديها إدخالات في `policyParams`.

### `policyParams`

النوع: `Record<string, Record<string, unknown>>`

تجاوزات المعاملات لكل سياسة. المفتاح الخارجي هو اسم السياسة؛ المفاتيح الداخلية خاصة بالسياسة. توثق كل سياسة معاملاتها المتاحة في [السياسات المدمجة](/ar/built-in-policies).

إذا كانت للسياسة معاملات لكنك لم تحددها، يتم استخدام القيم الافتراضية المدمجة للسياسة. المستخدمون الذين لا يقومون بتكوين `policyParams` على الإطلاق يحصلون على سلوك متطابق لإصدارات سابقة.

المفاتيح غير المعروفة داخل كتلة معاملات السياسة يتم تجاهلها صامتة عند وقت إطلاق hook لكن يتم تجاهلها كتحذيرات عند تشغيل `failproofai policies`.

#### `hint` (شامل)

النوع: `string` (اختياري)

رسالة يتم إلحاقها بالسبب عندما تعيد السياسة `deny` أو `instruct`. استخدمها لإعطاء Claude إرشادات قابلة للتنفيذ دون تعديل السياسة نفسها.

يعمل مع أي نوع سياسة - مدمجة أو مخصصة (`custom/`) أو اتفاقية مشروع (`.failproofai-project/`) أو اتفاقية مستخدم (`.failproofai-user/`).

```json
{
  "policyParams": {
    "block-force-push": {
      "hint": "Try creating a fresh branch instead."
    },
    "block-sudo": {
      "allowPatterns": ["sudo apt-get"],
      "hint": "Use apt-get directly without sudo."
    },
    "custom/my-policy": {
      "hint": "Ask the user for approval first."
    }
  }
}
```

عندما يرفض `block-force-push`، يرى Claude: *"Force-pushing is blocked. Try creating a fresh branch instead."*

القيم غير النصية والسلاسل الفارغة يتم تجاهلها صامتة. إذا لم تكن `hint` محددة، السلوك لا يتغير (متوافق للإصدارات السابقة).

### `customPoliciesPath`

النوع: `string` (مسار مطلق)

المسار إلى ملف JavaScript يحتوي على سياسات hook مخصصة. يتم تعيين هذا تلقائياً بواسطة `failproofai policies --install --custom <path>` (يتم حل المسار إلى مسار مطلق قبل تخزينه).

يتم تحميل الملف بشكل جديد على كل حدث hook - لا يوجد تخزين مؤقت. انظر [السياسات المخصصة](/ar/custom-policies) لتفاصيل الإنشاء.

### سياسات قائمة على الاتفاقية

بالإضافة إلى `customPoliciesPath` الصريح، يكتشف failproofai تلقائياً ويحمل ملفات السياسات من أدلة `.failproofai/policies/`:

| المستوى | الدليل | النطاق |
|--------|--------|--------|
| المشروع | `.failproofai/policies/` | مشترك مع الفريق عبر التحكم في الإصدار |
| المستخدم | `~/.failproofai/policies/` | شخصي، ينطبق على جميع المشاريع |

**مطابقة الملفات:** يتم تحميل الملفات فقط المطابقة `*policies.{js,mjs,ts}` (على سبيل المثال `security-policies.mjs`، `workflow-policies.js`). الملفات الأخرى في الدليل يتم تجاهلها.

**لا توجد إعدادات مطلوبة:** سياسات الاتفاقية لا تتطلب إدخالات في `policies-config.json`. فقط ضع الملفات في الدليل وسيتم التقاطها عند حدث hook التالي.

**التحميل المتحد:** يتم مسح دليل الاتفاقية للمشروع والمستخدم. يتم تحميل جميع الملفات المطابقة من كلا المستويين (بخلاف `customPoliciesPath` الذي يستخدم أول نطاق فائز).

انظر [السياسات المخصصة](/ar/custom-policies) لمزيد من التفاصيل والأمثلة.

### `llm`

النوع: `object` (اختياري)

إعدادات عميل LLM للسياسات التي تقوم باستدعاءات AI. غير مطلوب لمعظم الإعدادات.

```json
{
  "llm": {
    "model": "claude-sonnet-4-6",
    "apiKey": "sk-ant-..."
  }
}
```

## إدارة التكوين من CLI

تكتب أوامر `policies --install` و `policies --uninstall` إلى ملف إعدادات hook الخاص بـ CLI العامل (نقاط دخول hook)، بينما `policies-config.json` هو الملف الذي تدير مباشرة. الاثنان منفصلان:

- **إعدادات Agent CLI** — يخبر العامل باستدعاء `failproofai --hook <event>` على كل استخدام أداة:
  - **Claude Code**: `~/.claude/settings.json` (مستخدم)، `<cwd>/.claude/settings.json` (مشروع)، `<cwd>/.claude/settings.local.json` (محلي)
  - **OpenAI Codex**: `~/.codex/hooks.json` (مستخدم)، `<cwd>/.codex/hooks.json` (مشروع) — Codex ليس له نطاق محلي
  - **GitHub Copilot CLI _(beta)_**: `~/.copilot/hooks/failproofai.json` (مستخدم)، `<cwd>/.github/hooks/failproofai.json` (مشروع) — Copilot ليس له نطاق محلي. تستخدم إدخالات Hook حقول أوامر Copilot المرتبطة بـ OS `bash`/`powershell` مع `timeoutSec`؛ الملف يحمل علامة `version: 1` على المستوى الأعلى. دعم Copilot CLI في **beta** بينما نتحقق من مخطط سجل `events.jsonl` (الذي لا تحدده الوثائق العامة) مقابل جلسات عالم حقيقي أكثر.
  - **Cursor Agent _(beta)_**: `~/.cursor/hooks.json` (مستخدم)، `<cwd>/.cursor/hooks.json` (مشروع) — Cursor ليس له نطاق محلي. تستخدم إدخالات Hook نموذج Claude-shaped `{type, command, timeout}` (بدون تقسيم `bash`/`powershell`)، لكن مخزنة تحت مفاتيح حدث camelCase (`preToolUse`، `beforeSubmitPrompt`، ...) في صفيف مسطح لكل [مخطط hooks](https://cursor.com/docs/hooks) Cursor؛ الملف يحمل علامة `version: 1` على المستوى الأعلى. يقوم المعالج بتقويس camelCase → PascalCase عبر `CURSOR_EVENT_MAP` بحيث تطلق السياسات المدمجة الحالية دون تغيير. دعم Cursor Agent في **beta** بينما نتحقق من نسخة Cursor على القرص (غير محددة في الوثائق العامة) مقابل مثبتات عالم حقيقي أكثر.
  - **OpenCode _(beta)_**: `~/.config/opencode/opencode.json` + `~/.config/opencode/plugins/failproofai.mjs` (مستخدم)، `<cwd>/.opencode/opencode.json` + `<cwd>/.opencode/plugins/failproofai.mjs` (مشروع) — OpenCode ليس له نطاق محلي. بخلاف CLIs الستة الأخرى، OpenCode ليس له **نظام hook أوامر خارجية**: يقوم بتحميل مكونات JS/TS مدمجة مسجلة بوضوح عبر صفيف `plugin: []` في `opencode.json` (الاكتشاف التلقائي من `.opencode/plugins/` **ليس** كيفية تحميل المكونات الإضافية على opencode v1.14.33). يضع Install مكون حشو صغير تم إنشاؤه يستدعي ثنائي failproofai عبر subprocess ويترجم استجابة JSON في شكل Claude الثنائي إلى دلالات المكونات الإضافية: `throw new Error()` لـ tool-event deny (يلغي استدعاء الأداة)، `client.session.prompt(...)` لـ instruct و `Stop` / `SubagentStop` deny (يرسل سبب الرفض كرسالة المستخدم التالية — القناة الوحيدة لإعادة المحاولة القسرية منذ أن `session.idle` إشعار فقط والرمي منه لا-op)، و no-op لـ allow. الجلسات تعيش في قاعدة بيانات OpenCode SQLite في `~/.local/share/opencode/opencode.db`؛ عارض لوحة البيانات يقرأها عبر `opencode db --format json` و `opencode export <id>`. دعم OpenCode في **beta** بينما نتحقق من السلوك عبر الإصدارات وضد جلسات عالم حقيقي أكثر. انظر [وثائق مكونات OpenCode](https://opencode.ai/docs/plugins/).
  - **Pi _(beta)_**: `~/.pi/agent/settings.json` (مستخدم)، `<cwd>/.pi/settings.json` (مشروع) — Pi ليس له نطاق محلي. يقوم Pi بتحميل حزم التوسيع TypeScript عند البدء؛ ملف الإعدادات هو صفيف سلسلة مسطح `{"packages": ["./relative/path", …]}`. يكتب failproofai إدخال صفيف packages واحد يشير إلى دليل `pi-extension/` المضمن الخاص به. يقوم التوسيع داخلياً بالاشتراك في أحداث Pi `tool_call` / `user_bash` / `input` / `session_start` وينادي `failproofai --hook <Event> --cli pi`؛ يقوم المعالج بتقويس underscore_lower_snake_case → PascalCase عبر `PI_EVENT_MAP` بحيث تطلق السياسات المدمجة الحالية دون تغيير. دعم Pi في **beta** بينما تستقر واجهة برمجة تطبيقات التوسيع Pi وتخطيط سجل الجلسة.
  - **Gemini CLI _(beta)_**: `~/.gemini/settings.json` (مستخدم)، `<cwd>/.gemini/settings.json` (مشروع) — Gemini ليس له نطاق محلي (يوثق نطاق `system` في `/etc/gemini-cli/settings.json` الذي لا يكشفه failproofai). تستخدم إدخالات Hook نموذج Claude `{type, command, timeout}` ملفوف في مخطط تطابق Gemini `{matcher, hooks: [...]}` مع `matcher: "*"` بشكل افتراضي. الأحداث هي PascalCase (`SessionStart`، `BeforeAgent`، `AfterAgent`، `BeforeModel`، `AfterModel`، `BeforeToolSelection`، `BeforeTool`، `AfterTool`، `PreCompress`، `Notification`، `SessionEnd`)؛ يقوم المعالج بالخريطة إلى أسماء Claude الكنسية عبر `GEMINI_EVENT_MAP`. أسماء الأدوات هي snake_case (`run_shell_command`، `read_file`، `write_file`، `replace`، ...) — يقوم المعالج بالتقويس عبر `GEMINI_TOOL_MAP` بحيث تطلق السياسات المدمجة الحالية دون تغيير. يصدر محيِّم السياسة شكل Gemini المسطح `{decision: "deny", reason}` (مفضل وفقاً لـ "القاعدة الذهبية" exit-0 لـ Gemini)، `{hookSpecificOutput: {hookEventName, additionalContext}}` لحقن السياق على BeforeAgent / AfterTool / SessionStart، و `{decision: "block", reason}` على AfterAgent لدلالات إعادة المحاولة القسرية. دعم Gemini CLI في **beta** بينما نوسع التغطية في العالم الحقيقي. انظر [وثائق hooks Gemini CLI](https://geminicli.com/docs/hooks/).
- **`policies-config.json`** — يخبر failproofai بأي السياسات يتم تقييمها وبأي معاملات (مشترك عبر جميع CLIs العامل)

مرر `--cli claude|codex|copilot|cursor|opencode|pi|gemini` للاستهداف عامل محدد (مفصول بمسافات أو مكرر لأي مجموعة فرعية):

```bash
failproofai policies --install --cli codex --scope project
failproofai policies --install --cli copilot --scope project
failproofai policies --install --cli cursor --scope project
failproofai policies --install --cli opencode --scope project
failproofai policies --install --cli pi --scope project
failproofai policies --install --cli gemini --scope project
failproofai policies --install --cli claude codex copilot cursor opencode pi gemini
```

عند حذف `--cli`، يكتشف failproofai CLIs العامل المثبتة (`which claude` / `which codex` / `which copilot` / `which cursor-agent` / `which opencode` / `which pi` / `which gemini`):

- **تم الكشف عن CLI واحد** — يختار هذا CLI تلقائياً دون موجه.
- **تم الكشف عن CLIs متعددة** في محطة تفاعلية — يعرض موجه اختيار أحادي المسار مع أسهم مرتب في قسم `Detected (N)` (مع صف إجمالي واحد `Install for all N detected` + كل CLI تم الكشف عنه بشكل فردي) وقسم `Not installed (M) · install hooks ahead of time` يسرد كل CLI مدعوم غير مكتشف كخيار تثبيت عاجل (↑↓ للتحرك، Enter للاختيار، ^C للإقلاع). يعرض تدفق الإلغاء فقط قسم Detected.
- **تم الكشف عن CLIs متعددة** في تشغيل غير تفاعلي (CI، لا TTY) — يثبت لجميع CLIs المكتشفة دون موجه.
- **لم يتم الكشف عن أي شيء** — يسقط إلى `claude`، مع تحذير بأنه لم يتم العثور على ثنائي عامل في PATH؛ لا يزال أمر hook مكتوباً بحيث يتفعل بمجرد تثبيت واحد.

يمكنك تحرير `policies-config.json` مباشرة في أي وقت؛ التغييرات تدخل حيز التنفيذ فوراً على حدث hook التالي بدون حاجة للإعادة.

## مثال: إعدادات مستوى المشروع مع افتراضيات الفريق

التزم `.failproofai/policies-config.json` بمستودع الكود الخاص بك:

```json
{
  "enabledPolicies": [
    "block-sudo",
    "block-rm-rf",
    "block-push-master",
    "sanitize-api-keys",
    "block-env-files"
  ],
  "policyParams": {
    "block-push-master": {
      "protectedBranches": ["main", "release", "hotfix"]
    }
  }
}
```

يمكن لكل مطور عندئذ إنشاء `.failproofai/policies-config.local.json` (مدرجة في gitignore) للتجاوزات الشخصية دون التأثير على زملائهم.
