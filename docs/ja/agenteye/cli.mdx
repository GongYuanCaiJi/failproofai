FailproofAI Observability のすべての操作をターミナルまたはスクリプトから実行できます。ダッシュボードを開く必要はありません。`agenteye` CLI はデータ（セッション、イベントログ、評価）のクエリと、組織の管理（APIキー、ユーザー、設定、アラート、インシデント、保存済みクエリ）を行います。チェックの自動化、ObservabilityのCI組み込み、またはコーディングエージェントによる本番環境の検査に活用してください。すべてのコマンドは `--json` フラグに対応しているため、プロンプトからの手動操作にも、シェルアウトして結果を解析するコーディングエージェント（Claude Code、Cursor）にも同様に利用できます。

1つのバイナリで以下が可能です：

- **データの読み取り**：`sessions`、`events`、`evals`、`errors`（時刻・エージェント・環境・スコアでフィルタ）
- **組織の管理**：`keys`、`users`、`settings`、`alerts`、`incidents`
- **分析の実行**：保存済みSQLとアドホッククエリランナー（`query`）
- **AIアシスタントへの質問**：ダッシュボードで利用できる読み取り専用アナリストと同じもの（`agent`）

> **注意：** これは `agenteye` CLI であり、コレクターデーモン（`agenteye-collector`）とは別のツールです。CLIはダッシュボードと通信し、コレクターはサーバーにイベントを送信します。

## クイックスタート

ゼロから最初の結果を得るまでの4行です。CLIをダッシュボードに向け、サインインし、自分を確認してから、直近1日分の実行記録を取得します：

```bash
pipx install agenteye
agenteye --base-url https://agenteye.example.com login --email you@example.com   # 6桁のコードがメールで届きます
agenteye whoami                                                                  # ユーザーとアクティブな組織を確認
agenteye --json sessions --since 24h                                             # エージェント実行1件につき1行、直近24時間分
```

最後のコマンドは最新のセッション（新しい順、デフォルトで最大50件）のJSONオブジェクトを出力します。`jq` にパイプしてフィルタするか、`--json` を外してボックス付きのカラー表示にすることもできます。各行には実行のステータスと、評価器によるスコアリングが行われた場合はそのメトリクススコアが含まれます（ここでは省略形）：

```json
{
  "sessions": [
    {
      "session_id": "run-8f2a",
      "agent_id": "checkout-bot",
      "environment": "prod",
      "status": "error",
      "scores": { "helpfulness": 0.42, "tool_efficiency": 0.55 },
      "event_count": 37,
      "started_at": "2026-07-16T09:14:02Z",
      "last_event_at": "2026-07-16T09:14:48Z"
    }
  ],
  "next_cursor": null
}
```

このページの残りでは各要素を説明します：[インストール](#installation)、[サインイン](#authentication)、[設定](#configuration)、すべてのコマンドに共通する[グローバル規則](#global-options--conventions)、および[コマンドリファレンス](#command-reference)です。

## インストール

CLIは **`agenteye`** という名前のパブリックPyPIパッケージです。独立した環境にインストールして、常に専用の依存関係を持つようにしてください：

```bash
pipx install agenteye
# または
uv tool install agenteye
```

Python 3.10以上が必要です。インストールされるコマンドは **`agenteye`** です：

```bash
agenteye --version
agenteye --help
```

> **注意：** FailproofAI Observability Python SDKも `agenteye` という配布名を使用しています。`pipx` または `uv tool`（共有の仮想環境への `pip install` ではなく）を使ってCLIをインストールすることで、両者の衝突を防げます。SDKが同じ環境にインストールされていない場合に限り、単純な `pip install agenteye` でも問題ありません。

## 認証

CLIはメールで送られるワンタイムコードを使って**ダッシュボード**に認証します：

```bash
agenteye login --email you@example.com
# 6桁のコードがメールで届きます。プロンプトに貼り付けてください。
```

セッショントークンは `~/.agenteye/cli.json`（自分だけが読めるパーミッション `0600`）に保存され、デフォルトで24時間有効です。期限が切れたら `agenteye login` を再実行してください。

```bash
agenteye whoami     # 現在のユーザー、アクティブな組織、権限を表示
agenteye logout     # セッションを無効化し、保存されたトークンを削除
```

`whoami` はセッションが存在しない場合や期限切れの場合でもエラーになりません。代わりに `logged_in: false` を返すため、スクリプトやエージェントが安全に認証状態を確認できます（ベースURLが未設定だったりダッシュボードに到達できない場合は、それでもゼロ以外の終了コードになります）。

**要件：** ダッシュボードへのサインインが許可されたメールアドレスが必要です（FailproofAI Observabilityの管理者に確認してください）。また、ダッシュボードはベースURLで到達可能でなければなりません（[設定](#configuration)を参照）。コードをリクエストしても届かない場合は、そのメールアドレスがまだダッシュボードアクセスに対して有効化されていない可能性があります。

## 組織の選択（マルチテナント）

アカウントが複数の組織に属している場合、**ログイン時**にアクティブな組織を選択します。選択内容は保存され、以降のすべてのコマンドで使用されます：

```bash
agenteye login --org acme           # 認証とアクティブなテナントの設定を一度に行う
agenteye orgs list                  # アクセスできる組織の一覧（アクティブな組織にマーク付き）
agenteye orgs switch globex         # 保存されているデフォルトを変更
agenteye --org globex sessions      # 単一コマンドのみオーバーライド
```

組織が1つだけの場合は自動的に選択されるため、`--org` を気にする必要はありません。複数の組織に属していて選択していない場合、CLIは一覧を表示し `--org <slug>` を付けて再実行するよう求めます。アクティブな組織はすべてのリクエストでダッシュボードに送信され、権限は**組織ごと**に解決されます。`agenteye whoami` はアクティブな組織、その中での権限、およびすべてのメンバーシップを表示します。

## 設定

| 設定 | フラグ | 環境変数 | デフォルト |
|---|---|---|---|
| ダッシュボードのベースURL | `--base-url` | `AGENTEYE_DASHBOARD_URL` | **必須**（デフォルトなし） |
| アクティブな組織/テナント | `--org` | `AGENTEYE_ORG` | ログイン時に選択。`~/.agenteye/cli.json` に保存 |
| セッショントークン | `--token` | `AGENTEYE_CLI_TOKEN` | `~/.agenteye/cli.json` から取得 |
| JSON出力 | `--json` | `AGENTEYE_CLI_JSON` | オフ |
| TLS検証のスキップ | `--insecure` / `--secure` | `AGENTEYE_INSECURE` | オフ（ログイン時に保存） |
| リクエストタイムアウト（秒） | `--timeout` | _（なし）_ | 30 |
| 使用状況テレメトリの無効化 | _（なし）_ | `AGENTEYE_ANALYTICS_DISABLED`（または `DO_NOT_TRACK`） | テレメトリは現在無効。何も送信されません |

解決の優先順位は **フラグ → 環境変数 → 設定ファイル** の順です。デフォルトはありません。CLIをダッシュボードに向ける必要があります。コマンドごとに指定するか（`--base-url https://agenteye.example.com`）、環境変数で一度設定してください（最初の `login` 後にも保存されます）：

```bash
export AGENTEYE_DASHBOARD_URL=https://agenteye.example.com
```

設定ディレクトリは `AGENTEYE_HOME`（SDKとコレクターが使う同じ規則）を優先します。設定されている場合、`cli.json` は `$AGENTEYE_HOME/cli.json` に配置されます。

### 自己署名証明書または内部TLS

ダッシュボードが自己署名証明書や内部証明書を使ったHTTPS（例：ロードバランサーの生のホスト名）で提供されている場合、TLS検証は `CERTIFICATE_VERIFY_FAILED` エラーで拒否します。`--insecure` を渡して証明書検証をスキップしてください：

```bash
agenteye --base-url https://agenteye.internal --insecure login
```

`--insecure` はログイン時に **`cli.json` に保存される**ため、以降のコマンドでは自動的に検証がスキップされます。フラグを毎回繰り返す必要はありません。一時的に検証ありの呼び出しをするには `--secure` を渡し、次回ログイン時に検証を元に戻すこともできます。検証が無効になっている状態でダッシュボードと通信するコマンドを実行すると、CLIはstderrに警告を出力します。検証をスキップすると中間者攻撃への防御が失われます。ダッシュボードへのネットワーク経路（VPN、プライベートサブネットなど）を信頼できる場合にのみ使用してください。

## テレメトリとプライバシー

> **注意：** 現在リリースされているCLIは**使用状況テレメトリを一切送信しません。** マスターキルスイッチがオンになっているため、環境に関わらず何も送信されません。以下のセクションでは、テレメトリが将来有効化された場合のオプトアウト機能について説明します。

有効化された場合でも、テレメトリは**匿名の使用状況分析のみ**であり、エージェント・セッション・イベントのデータは含まれません：

- **エージェント、セッション、イベントのデータがインフラの外に出ることは一切ありません。** 報告されるのはCLIの使用状況のみです：コマンドとサブコマンド名（例：`keys create`）、使用したフラグの**名前**（値は含まない）、成功/終了ステータス、実行時間、およびミューテーション操作ごとのイベント（例：`api_key_created`、`query_run`。静的な名前/enumと大まかなカウントのみ）。ダッシュボードURL、セッショントークン、メール、組織スラグ、リソースID、SQL、キーのシークレット、クエリフィルターは**一切送信されません**。オペレーターは不透明な内部IDのみで識別され、メールアドレスは使われません。
- **事前にオプトアウト**するには、CLIの環境に `AGENTEYE_ANALYTICS_DISABLED=1` を設定してください（ツール横断の規則 `DO_NOT_TRACK=1` も尊重されます）。これはテレメトリが有効化された瞬間から効果を発揮するため、プライバシーを重視する環境では永続的にオプトアウト状態を維持できます。
- テレメトリが有効化された場合、CLIはPostHog（`https://us.i.posthog.com`）に直接送信します。そのホストをブロックしているマシンは何も送信せず、CLIへの影響もありません。

## グローバルオプションと規則

一度読んでおいてください。すべてのコマンドに適用されます。

- **グローバルオプションはコマンドの前に指定します。** `agenteye --json sessions` が正しく、`agenteye sessions --json` は使用エラーです。グローバルオプションは `--json`、`--base-url`、`--org`、`--token`、`--insecure`/`--secure`、`--timeout`、`--quiet`、`--no-color` です。
- **`--json` はpure JSONのみをstdoutに出力し、それ以外は何も出力しません。** 人向けのステータス行、警告、エラーは**stderr**に出力されるため、ステータス行が表示される場合でも `--json` のstdoutキャプチャは `jq` へのパイプに対してクリーンな状態を保ちます。`--json` なしの場合は人間向けのボックス付きカラー表示になります。
- **`--help` で探索できます。** すべてのコマンドとサブコマンドには `--help`（と `-h` エイリアス）があります：`agenteye -h`、`agenteye sessions -h`、`agenteye keys create -h`。トップレベルのヘルプには終了コードとグローバルオプションも一覧表示されます。グローバルなマシン可読サーフェスダンプはありません。コマンドごとの `--help` と、2つのレジストリ向けの `agenteye query schema` および `agenteye settings schema` を使用してください。
- **スクリプトとエージェントでは確認プロンプトが自動スキップされます。** 作成/更新/削除コマンドはインタラクティブなターミナルでは「本当によろしいですか？」と確認を求めますが、**`--json` 指定時またはstdinがTTYでない場合**（TTYはインタラクティブなターミナルセッション。パイプやCIランナーはTTYではない）は自動スキップされるため、スクリプトやエージェントがハングすることはありません。明示的にスキップするには `--yes`/`-y` を渡してください。エージェントの場合はプロンプトが表示されないため、破壊的な操作の前に人間に確認を取るべきです。
- **ページネーション：** 結果は新しい順でカーソルページネーションされます（各ページは次のページを取得するためのトークンを返します）。`--limit N`（エイリアス `-n`）は行数を制限し、**デフォルトは50**です。`--all` は自動ページネーションを行いますが（200行チャンク単位）、**`--limit` の値まで**にとどまるため、単純な `--all` は依然として50件で停止します。全件取得するには明示的に大きい値を指定してください：`--all --limit 1000`。`--page-size N` はリクエストごとのチャンクサイズを制御します（最大200）。`--cursor <id>` は前のページの `next_cursor` から再開します。
- **時間フィルター：** `--since` は相対的なウィンドウを受け付けます：`15m`、`1h`、`6h`、`24h`、`7d`、または `all`（ダッシュボードのプリセット）。より長いまたはカスタムの範囲（例：直近30日）には `--from`/`--to` を使用してください：`--since` をオーバーライドする明示的なISO-8601 UTCタイムスタンプ（**`T` とタイムゾーン付き**、例：`2026-06-01T00:00:00Z`）が必要です。スペース区切りやタイムゾーンなしの値は使用エラーになります。
- **`--fields a,b,c`**（`events`、`sessions`、`evals`、`errors` で使用）は出力をそれらのキーに制限します。テーブル表示と `--json` の両方に適用されます。不明な名前は有効なリストとともに拒否されるため、フィールド名を探索する簡単な方法にもなります。
- **`--file payload.json`**（または `--file -` でstdinから読み込み）は、リソースが複雑な形状を持つ場合に完全なJSONリクエストボディを提供します（`alerts create/update`、`settings set`、`users create/update`）。保存済みクエリのSQLには代わりに `--sql @file.sql` を使用します。
- **複数値フィルター**はカンマ区切りで、セットとしてマッチします（1つのフィルター内はOR、フィルター間はAND）：`--event-type tool_use,tool_result`。Clickオプションは可変長引数ではないため、`--add a b` は機能しません。`--add a,b`、フラグの繰り返し（`--add a --add b`）、またはクォート（`--add "a b"`）を使用してください。

## コマンドリファレンス

### 最もよく使う5つのコマンド

日常作業のほとんどはいくつかの読み取りコマンドで完結します。まずここから始め、必要に応じて以下の全機能を参照してください：

| コマンド | 機能 | 試してみる |
|---|---|---|
| `sessions` | エージェント実行1件につき1行：時刻、環境、エージェント、ステータス、最新スコア | `agenteye --json sessions --since 24h --status error` |
| `events` | 実行内のステップごとの生のトレイル（`--full` でペイロード付き） | `agenteye --json events --session-id run-001 --all` |
| `evals` | 評価結果とスコア。`--aggregate` でロールアップ | `agenteye --json evals --aggregate --since 7d --env prod` |
| `errors` | エラーが発生したイベントのみ。`--aggregate` でタイプ別カウント | `agenteye --json errors --since 24h --aggregate` |
| `list` | 有効なフィルター値の探索（エージェント、環境、モデルなど） | `agenteye list agents` |

### CLIで可能なすべての操作

全機能の一覧を以下に示します。CLIには**18のトップレベルコマンド**があります。すべての読み取りコマンドは `--json` とグローバルオプションに対応しています。詳細なフラグ一覧と各コマンドのJSONの形状については `agenteye <command> -h`（または `<command> <subcommand> -h`）を実行してください。

### ID関連：`login` · `logout` · `whoami` · `orgs` · `version` · `help`

```bash
agenteye login --email you@example.com [--org acme]   # メールで送られるワンタイムコード。セッションを保存
agenteye logout                                       # このマシンの保存済みセッションを削除
agenteye whoami                                       # 現在のユーザー、アクティブな組織、権限
agenteye version                                      # CLIのバージョンを表示（--version と同じ）
agenteye help                                         # トップレベルのヘルプ（--help と同じ）
```

`orgs` はアクティブなテナントを確認・切り替えます：

```bash
agenteye orgs list        # 所属する組織と各組織での役割（アクティブな組織にマーク付き）
agenteye orgs switch acme # 保存済みのアクティブな組織を変更（スラグを省略するとTTYでリストから選択）
agenteye orgs current     # アクティブな組織のIDカード
agenteye orgs perms       # アクティブな組織での権限をリソース別にグループ表示
```

### 観察（読み取り専用）：`events` · `sessions` · `evals` · `errors` · `list`

これらのコマンドはいずれも確認プロンプトを必要としません。共通フィルター：`--session-id`、`--agent-id`、`--env`（`--environment` **ではない**）、および時間範囲（`--since` / `--from` / `--to`）。

```bash
# events（エイリアス：ステップごとの生のトレイル）、新しい順
agenteye --json events --session-id run-001 --event-type tool_use,tool_result --all --limit 1000
agenteye --json events --since 1h --search timeout --all | jq '.events[].payload'

# sessions：エージェント実行1件につき1行（時刻/環境/エージェント/セッション/ステータス。スコアフィルタなし）
agenteye --json sessions --since 24h --status error
agenteye --json sessions --agent-id checkout-bot --env prod --all --limit 1000

# evals：評価結果とスコア。--score でメトリクスフィルタ。--aggregate でロールアップ
agenteye --json evals --score helpfulness:0.5..0.8 --score tool_efficiency:..0.3
agenteye --json evals --aggregate --since 7d --env prod        # ステータスの分布 + キー別スコア統計

# errors：エラーが発生したイベント。--aggregate でカウント/セッション/エージェント/最終発生時刻
agenteye --json errors --since 24h --aggregate
agenteye --json errors --since 24h --error-type timeout --all --limit 1000

# list：フィルタリングの前に有効なフィルター値を探索
agenteye list envs        # 他にも：agents event_types score_filters models hooks tools error_types
```

`--score KEY:MIN..MAX`（**`evals`** に対して使用。`sessions` には使えない）は繰り返し可能でAND結合されます。どちらの境界も省略可能です（`..0.5` は ≤ 0.5、`0.9..` は ≥ 0.9 を意味します）。リクエストあたり最大20のスコアフィルター。`evals --scores-full` は**人間向けテーブルのみ**の表示フラグで、最初の数件と `+N` カウントの代わりにすべてのスコアペアを表示します。`--json` では常に完全なスコアオブジェクトが返されるため効果はありません。**1つのセッション全体を読む**には、イベントトレイルと評価を組み合わせます：

```bash
agenteye --json events --session-id run-001 --all --limit 1000 | jq '.events[] | {ts, event_type}'
agenteye --json evals  --session-id run-001                       # スコアとステータス
```

### 管理（権限が必要）：`keys` · `users` · `settings` · `alerts` · `incidents`

**`keys`**：APIキー。シークレットはローカルで生成され、サーバーに送信されます（サーバーはハッシュのみを保存）。シークレットは作成/再生成時に**一度だけ表示**されます。その場でコピーしてください。`--json` 指定時は `key` フィールドにのみ表示されます。**名前**で参照します。

```bash
agenteye keys list                                   # アクティブなキーを先に表示し、次に失効済みキー
agenteye keys show ci-bot
agenteye keys create ci-bot --add events:read.add    # 必要なスコープのみ付与。シークレットは一度だけ表示
agenteye keys create ops --permission-set standard --remove queries:run   # プリセットをベースにして調整
agenteye keys update ci-bot --add evaluations:read --yes
agenteye keys regenerate ci-bot --yes                # シークレットをローテーション（古いものは無効になる）
agenteye keys disable ci-bot --yes                   # 失効
```

権限は `(パーミッションセット ∪ --add) − --remove` として処理されます。トークンは `slug:action`（例：`events:read`）または `slug:action.action`（1つのリソースに複数のアクションを展開。`events:read.add` → `events:read`、`events:add`）の形式です。プリセット：`read-only`、`standard`、`admin`。人間専用の権限（`keys:update`）はキーに付与できません。

**`users`**：組織メンバー。**メール**で参照します（UUID形式のIDも使用可能）。

```bash
agenteye users list [--active-only]
agenteye users show dev@corp.com
agenteye users create dev@corp.com --permission-set standard
agenteye users update dev@corp.com --add alerts:write --remove queries:delete   # 変更内容を確認後に実行
agenteye users disable dev@corp.com --yes            # 保護されたユーザー・自分自身への保護あり
agenteye users enable dev@corp.com
```

**`settings`**：固定レジストリ（既存のキーの読み取りと変更のみ。新規キーは作成不可）。

```bash
agenteye settings list                               # キー・値・タイプ・更新日時（シークレットはマスク）
agenteye settings schema                             # 各キーが受け付ける値（タイプ・範囲・説明）
agenteye settings set session_ttl_secs --value 86400 --yes
```

**`alerts`**：アラート定義。**名前**で参照します。`create` は位置引数のNAMEと、フラグまたは `--file` による完全なJSONボディを受け付けます。

```bash
agenteye alerts list
agenteye alerts show high-errors
agenteye alerts create high-errors --file alert.json          # NAME は必須（位置引数）
agenteye alerts update high-errors --severity critical --yes
agenteye alerts test high-errors --yes                        # テスト通知を送信
agenteye alerts delete high-errors --yes
```

**`incidents`**：アラートインシデント。IDで参照します（短縮IDも使用可能）。`show` は完全なアクティビティログを表示します。操作前に確認してください。

```bash
agenteye incidents list --state firing                # 他にも：acknowledged、resolved
agenteye incidents count
agenteye incidents show <id>
agenteye incidents ack <id>
agenteye incidents assign <id> you@corp.com           # 担当者はオペレーターでなければならない
agenteye incidents resolve <id> --yes
agenteye incidents open --alert-id <id> --severity critical    # アラートに対して手動で開く
agenteye incidents comment-add <id> "root cause: upstream 5xx"
agenteye incidents comment-list <id> ; agenteye incidents comment-delete <id> <comment-id>
agenteye incidents subscribe <id> ; agenteye incidents unsubscribe <id> ; agenteye incidents subscribers <id>
```

### 分析とアシスタント：`query` · `agent`

**`query`**：分析ストアに対する保存済みSQLとアドホッククエリランナー。保存済みクエリは**名前**で参照します。SQLはサーバー側で検証されます（SELECT/WITHのみ、ステートメントタイムアウト、行数上限）。

```bash
agenteye query schema [TABLE]                         # 分析ビューのカラム構成
agenteye query run --sql "select count(*) from analytics.events"
agenteye query run errs --arg prod --limit 100        # 保存済みクエリと位置引数 $1 を実行
agenteye query list ; agenteye query show errs
agenteye query create errs --sql @errs.sql --description "errored events (24h)"
agenteye query update errs --sql @errs.sql --yes ; agenteye query delete errs --yes
```

**`agent`**：組み込みの**AIアシスタント**（ダッシュボードでチャットできる読み取り専用アナリストと同じもの）と通話します。チャットは短いchat-id（プレフィックス解決対応）で参照します。

```bash
agenteye agent health                                 # AIアシスタントが設定済み/到達可能かを確認
agenteye agent models                                 # --model に渡せるモデルの一覧（デフォルトにマーク付き）
agenteye agent ask "which agents errored most in the last day?"    # チャットを開始。短いIDを表示
agenteye agent ask --chat <short-id> "and which tools did they call?"   # チャットを継続
agenteye agent chats ; agenteye agent show <short-id>
agenteye agent rename <short-id> --title "error triage" ; agenteye agent delete <short-id>
```

## 終了コード

| コード | 意味 |
|---|---|
| 0 | 成功 |
| 1 | 予期しないエラー（例：ダッシュボードが5xxを返した） |
| 2 | 使用エラー（無効な引数、不明なコマンド/フラグ、名前の衝突） |
| 3 | ダッシュボードに到達できない |
| 4 | 未ログインまたはセッション期限切れ。`agenteye login` を実行してください |
| 5 | 認証済みだが、必要な権限がありません（メッセージに権限名が記載） |
| 6 | 要求されたリソースが見つかりません（例：不明なセッションIDまたはインシデントID） |

これらの終了コードにより、CLIをスクリプトで安全に使用できます。コーディングエージェントは `4` を受け取ったら再認証を促し、`5` を受け取ったら不足している権限を表示できます。終了コードの処理パターンとJSONの出力形式については、[エージェント向けCLIレシピ](/ja/agenteye/cli-recipes)を参照してください。
