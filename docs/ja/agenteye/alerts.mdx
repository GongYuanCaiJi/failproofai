何か問題が発生した瞬間に、チームがすでに使っているチャンネルで通知を受け取りましょう。顧客から報告される前に気付けます。ルールを一度設定するだけで、FailproofAI Observability がスケジュールに沿って監視し、メール、Slack、Webhook、またはダッシュボード上で通知します。

*すべてのアラートルールを一覧で確認：監視対象、実行頻度、通知先、緊急度。*

## ユーザーより先に問題を把握する

ダッシュボードを何度も更新して不具合を見つけようとするのは終わりにしましょう。誰も見ていないときでも知らせてほしいシグナルにはアラートを設定し、普段使っている場所に通知を届けましょう：

- **メール**：通知すべき担当者へ。
- **Slack**：インシデントへ直接ジャンプするボタン付きのリッチメッセージ。
- **Webhook**：PagerDuty、Opsgenie、または独自エンドポイントへの JSON POST。受信側が信頼性を検証できるオプションの署名付き。
- **ダッシュボード内**：ルールを調整中で誰にも通知したくないときのための、控えめな通知。

1 つのルールに任意の組み合わせで通知先を設定でき、重大度（情報・警告・重大）も一緒に伝わるため、緊急なものはそれとわかる形で届きます。

## JSON ではなくフォームでルールを作成する

「壊れている」状態をフォームで記述すると、FailproofAI Observability が内部ルールを自動生成します。JSON 仕様はそのフォームが裏側で生成するものなので、ルールを理解するために読むことはあっても、直接書く必要はほとんどありません。

![新規アラートフォーム：名前と説明、有効/無効トグル、メトリクス閾値・カスタム SQL・評価スコア・複合評価・イベント単位の条件を選択できるトリガーピッカー](/agenteye/images/alert-new.png)
*トリガーを選ぶと対応するフィールドが表示される。保存でルールが書き込まれる。*

基本的な流れは素早く完了します：名前を付け、**トリガー**（何を監視するか）を選び、**閾値とウィンドウ**（どの程度の問題を、どのくらいの期間で）を設定し、少なくとも 1 つの**チャンネル**を追加して、**保存**します。その後 **テスト** を実行して疑似通知を送信し、すべての通知先が正しく設定されているか確認しましょう。内部的には次のような小さな仕様が生成されます：

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

シグナルの種類は 1 つに限りません。障害の捉え方に合ったトリガーを選んでください：

| トリガー | 発火条件 |
|---|---|
| **メトリクス閾値** | エラー率、p95・p99 レイテンシ、イベント数・エラー数、トークン消費量などのプリセットメトリクスが、ウィンドウ内で閾値を超えたとき |
| **カスタム SQL** | 独自の読み取り専用クエリが行を返したとき、またはそのクエリが計算した値が閾値を超えたとき |
| **評価スコア** | 評価器スコア（例：ハルシネーション）の平均値が閾値を超えたとき |
| **複合評価** | any・all・N 件以上のロジックで複数のスコアチェックを組み合わせ、スコア全体にわたってのみ現れる回帰を検知するとき |
| **イベント単位** | 特定のエージェント、特定のエラータイプ、またはメッセージのサブ文字列に一致する単一イベントが発生したとき |

[エラーページ](/ja/agenteye/error-tracking)で障害を確認中ですか？各行には **+ アラート** ボタンがあり、クリックすると同じフォームがその障害を再度検知するための設定で開きます。今トリアージしたインシデントが、次回の通知トリガーになります。

**場所：** アラートは `/<org-slug>/alerts` にあります。ルールの作成・編集・削除・テストには **`alerts:write`** が必要です。閲覧は `alerts:read` で十分です。受信者ピッカーには組織メンバーが名前で表示されるため、フォームを離れずに担当者を指定できます。

## 本当に問題があるときだけ通知する

たった 1 回の計測ミスで起こされるべきではありません。**M of N** ノイズフィルターは、直近の数回のチェックのうち何回失敗した場合にアラートを実際に発報するかを制御します。**5 回中 3 回** に設定すると、直近 5 回のチェックのうち 3 回が閾値を超えた場合のみ発報されるため、不安定なシグナルが誤報を繰り返すことがなくなります。デフォルトの **1 of 1** のままにすると、最初の違反で即座に発報します。実行頻度は 1m・5m・15m・1h のプリセットから選択でき、シグナルの実際の変化速度に合わせて設定できます。

## アラートが発報したときの流れ

閾値超過が発生すると **インシデント** が開かれ、チャンネルに 1 回通知されます。その後チームが確認・担当者アサイン・議論・解決を行い、その過程はすべてクリーンで属性付きの記録として残ります。このトリアージワークフローには専用のページがあります：[インシデント](/ja/agenteye/incidents) を参照してください。

## 関連ドキュメント

- [インシデント](/ja/agenteye/incidents)：発報したアラートをオープンから確認済み、解決済みまで追跡する。
- [エラートラッキング](/ja/agenteye/error-tracking)：エージェントの障害をグループ化し、1 クリックでアラートに昇格させる。
- [ダッシュボード](/ja/agenteye/dashboards)：アラート閾値の基となる共有ボードを確認する。
- [CLI とエージェント](/ja/agenteye/cli-and-agents)：ターミナルからアラートの作成やインシデントの確認応答を行う。CI にスクリプトとして組み込むことも可能。
