顧客から報告を受ける前に、チームがすでに使っているチャンネルで、問題が発生した瞬間に通知を受け取りましょう。ルールを一度設定するだけで、Failproof AI Observability がスケジュールに従ってチェックを実行し、メール、Slack、webhook、またはダッシュボード上で通知します。

![アラートページ：アラートルールカードのグリッド。各カードにはトリガー、評価ウィンドウ、チャンネル、および情報・警告・重大の重要度バッジが表示されている](/agenteye/images/alerts.png)
*すべてのアラートルールを一覧で確認：監視対象、評価頻度、通知先、緊急度。*

## ユーザーより先に問題を把握する

ダッシュボードをリロードしながら回帰を見つけようとするのはやめましょう。誰も見ていないときでも把握しておきたいシグナルがあれば、アラートを設定して、普段使っている場所に通知を届けましょう。

- **メール**：通知すべき担当者へ。
- **Slack**：インシデントに直接ジャンプできるボタン付きのリッチメッセージ。
- **Webhook**：PagerDuty、Opsgenie、または独自のエンドポイントへの JSON POST。受信側が信頼できるよう任意の署名を付与可能。
- **ダッシュボード内**：ルールを調整中でまだ誰にも通知したくないときのための、静かな通知。

1つのルールに任意の組み合わせで通知先を設定でき、重要度（情報・警告・重大）も一緒に送信されるため、緊急のものは緊急に見えます。

## JSON ではなくフォームでルールを作成する

「壊れている」状態をフォームで記述するだけで、Failproof AI Observability が内部ルールを生成します。JSON 仕様はフォームが生成したものにすぎないため、ルールの内容を理解するために読むことはあっても、直接入力することはほとんどありません。

![新規アラートフォーム：名前と説明、有効化トグル、およびメトリクスしきい値・カスタム SQL・評価スコア・複合評価・イベント単位の条件を提示するトリガーピッカー](/agenteye/images/alert-new.png)
*トリガーを選択するとフォームが適切なフィールドに切り替わり、保存でルールが書き込まれます。*

基本的な手順はシンプルです。名前を付け、**トリガー**（監視対象）を選択し、**しきい値とウィンドウ**（どの程度悪化したら、どの期間で）を設定し、**チャンネル**を少なくとも1つ設定してから、**保存**し、**テスト**を実行して合成通知を送信し、すべての通知先が正しく設定されていることを確認します。内部的には以下のような小さなスペックが生成されます：

```json
{ "metric": "p95_latency_ms", "op": ">", "value": 5000, "window_secs": 900 }
```

シグナルの種類は1つに限りません。障害の性質に合ったトリガーを選択してください：

| トリガー | 発火条件 |
|---|---|
| **メトリクスしきい値** | エラー率、p95/p99 レイテンシ、イベント数やエラー数、トークン消費量などのプリセットメトリクスが、ウィンドウ期間中にしきい値を超えたとき |
| **カスタム SQL** | 独自の読み取り専用クエリが行を返すか、計算した値がしきい値を超えたとき |
| **評価スコア** | 評価スコア（例：ハルシネーション）の平均がしきい値を超えたとき |
| **複合評価** | 複数のスコアチェックを any・all・N件以上のロジックで組み合わせ、複数スコアにまたがる回帰を検出するとき |
| **イベント単位** | 特定のエージェント、特定のエラータイプ、またはメッセージの部分文字列に一致する単一のイベントが発生したとき |

[エラーページ](/ja/agenteye/error-tracking)で障害を確認中ですか？各行に **+ アラート** ボタンがあり、その障害を再度検出するようにプリフィルされた状態で同じフォームが開きます。これにより、今トリアージしたインシデントが次回の通知トリガーになります。

**場所：** アラートは `/<org-slug>/alerts` にあります。ルールの作成・編集・削除・テストには **`alerts:write`** が必要で、閲覧のみなら `alerts:read` で十分です。受信者ピッカーには組織メンバーが名前で一覧表示されるため、フォームを離れることなく特定の担当者に通知できます。

## 本当に必要なときだけ通知する

1回の悪い計測値で起こされるべきではありません。**M of N** ノイズフィルターは、アラートが実際に通知を送る前に、直近の数回のチェックのうち何回失敗する必要があるかを制御します。**3 of 5** に設定すると、直近5回のチェックのうち3回しきい値を超えた場合にのみ発火するため、不安定なシグナルによる誤報を防ぎます。デフォルトの **1 of 1** のままにすれば、最初のしきい値超過で即時発火します。また、シグナルの実際の変化速度に合わせて、1m・5m・15m・1h のプリセットからルールの実行頻度を選択できます。

## アラートが発火したときの動作

しきい値超過によって**インシデント**が作成され、チャンネルに1回通知が送信されます。その後、チームはインシデントを認識し、担当者を割り当て、議論し、解決します。これらはすべて整理された帰属記録に対して行われます。このトリアージワークフローには専用のページがあります：[インシデント](/ja/agenteye/incidents)をご覧ください。
