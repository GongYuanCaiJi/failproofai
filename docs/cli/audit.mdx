The audit is now exposed as the **/audit dashboard page**, not a CLI subcommand. Open it from the dashboard navbar (between Policies and Projects), or visit `http://localhost:8020/audit` directly when running `failproofai` locally.

```bash
failproofai          # open the dashboard, then click "Audit"
```

The dashboard scans past agent CLI transcripts on this machine (Claude Code, Codex, Copilot, Cursor, OpenCode, Pi, Gemini) and reports how often the agent did things failproofai is built to stop ‚Äî env-var checks, force pushes, redundant `cd <cwd>` prefixes, sleep-polling loops, re-reading files just edited, and more.

For each transcript, every tool-use event is replayed through the 39 builtin policies **and** through 8 audit-only detectors that catch patterns not yet covered by runtime policies. Counts are aggregated per policy / detector across all sessions.

## What you get

The `/audit` page composes six sections:

1. **Identity** ‚Äî your agent classified into one of 8 archetypes (`optimist`, `cowboy`, `explorer`, `goldfish`, `paranoid architect`, `precision builder`, `hammer`, `ghost`) based on the weighted signal across every audited transcript.
2. **Strengths** ‚Äî real numbers derived from the scan (clean-call %, "0 credential leaks", etc.) gated on the relevant sanitize policies actually firing.
3. **Score** ‚Äî 0-100 with S/A/B/C/D/F bands and a projected uplift if every recommended policy were enabled.
4. **Findings** ‚Äî per-policy cards with what happened, cost, captured evidence, and the exact `failproofai policy add <slug>` to enable the live-time builtin that would have caught it.
5. **Prescribed policies** ‚Äî aggregated install list with a one-shot `failproofai policies --install` command.
6. **Re-audit reminder** ‚Äî "come back better." Set a 7-day email reminder via the api-server (requires sign-in; see [`failproofai auth`](/cli/auth)).

## Audit-only detectors

These detect "stupid behavior" patterns not (yet) enforced in real time. They run only during the audit and never block a live tool call.

| Detector | What it counts |
|---|---|
| `redundant-cd-cwd` | Bash commands starting with `cd <cwd> && ‚Ä¶` even though commands already run in `cwd`. |
| `prefer-edit-over-read-cat` | `cat`/`head`/`tail`/`less`/`more` on a single source file ‚Äî use the `Read` tool. |
| `prefer-edit-over-sed-awk` | `sed -i` / `awk ‚Ä¶ > file` in-place edits ‚Äî use the `Edit` tool. |
| `prefer-write-over-heredoc` | Heredoc / multi-line `echo > file` writing files ‚Äî use the `Write` tool. |
| `sleep-polling-loop` | Long `sleep N` (‚â• 30s) or `while ‚Ä¶; sleep ‚Ä¶; done` polling loops. |
| `find-from-root` | `find /`, `find /home`, `find /usr`, etc. ‚Äî scope to `cwd` instead. |
| `git-commit-no-verify` | `git commit ‚Ä¶ --no-verify` / `-n`, skipping hooks. |
| `reread-after-edit` | `Read` of a file that was just `Edit`/`Write` in the same session. |

## Caches

- **Per-transcript cache** at `~/.failproofai/cache/audit/<sha1>.json` keyed by `(mtime, size, engineVersion, detectorVersion)` ‚Äî invalidates automatically when the transcript or the policy/detector code changes. Each entry also stores a `cachedAt` timestamp as **TTL metadata** (not part of the cache key); entries older than **7 days** are rejected on read so long-lived results don't outlive evolving detector intent.
- **Whole-result cache** at `~/.failproofai/audit-dashboard.json` (mode 0600). Lets the dashboard render instantly on navigation without re-running. Also rejected on read past the **7-day TTL** ‚Äî `/audit` then falls through to the empty state with an `audit expired ‚Äî re-audit to refresh` banner. Click `[ re-audit ]` at the top of the page (or `[ re-audit now ]` near the bottom) to refresh; the new run streams progress via a sticky top strip and swaps the result in place on success (no page reload).

## Notes

- **No mutation.** The audit replays in read-only mode. `warn-repeated-tool-calls` is skipped because its per-session sidecar would otherwise be modified.
- **Workflow policies skipped.** `require-*-before-stop` policies fire only on `Stop` events and `execSync` against the live git state ‚Äî they have no meaningful "what would have happened in 2025" interpretation, so they don't appear in audit counts.
- **Custom policies skipped.** User-supplied custom hooks are not replayed (they may have changed since the original session).
