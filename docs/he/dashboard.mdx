לוח הבקרה של failproofai הוא יישום אינטרנט מקומי לניטור הפעלות סוכני ה-AI שלך וניהול מדיניות. ראו מה עשו הסוכנים שלכם בזמן שלא הייתם שם.

---

## הפעלת לוח הבקרה

```bash
failproofai
```

נפתח ב-`http://localhost:8020`.

לוח הבקרה קורא ישירות מערכת הקבצים - תיקיות פרויקטי Claude Code ולוחות ההגדרות של failproofai שלכם. שום דבר לא כתוב לשירות מרחוק.

---

## דפים

### Projects

מציג את כל פרויקטי Claude Code, OpenAI Codex, GitHub Copilot CLI _(beta)_, Cursor Agent _(beta)_, OpenCode _(beta)_, Pi _(beta)_, ו-Gemini CLI _(beta)_ שנמצאו במחשב שלך. פרויקטי Claude מתגלים מתוך `~/.claude/projects/` (או הנתיב שנקבע על ידי `CLAUDE_PROJECTS_PATH`); פרויקטי Codex מתגלים על ידי סריקת כל תמלול תחת `~/.codex/sessions/<YYYY>/<MM>/<DD>/*.jsonl` וקיבוץ לפי ה-`cwd` שנרשם בשיא הראשון של כל הפעלה; פרויקטי Copilot CLI מתגלים על ידי סריקת כל `~/.copilot/session-state/<sessionId>/workspace.yaml` (ניתן להגדרה דרך `COPILOT_HOME`) וקיבוץ לפי שדה ה-`cwd`; פרויקטי Cursor Agent מתגלים על ידי סריקת מטא-נתונים לכל הפעלה תחת `~/.cursor/agent-sessions/<sessionId>/` (ניתן להגדרה דרך `CURSOR_HOME`, עם `conversations/` ו-`sessions/` כחלופות) לחיפוש סקלר `cwd` ב-`meta.json` / `session.json` / `workspace.yaml`; פרויקטי OpenCode מתגלים על ידי שאילתת בסיס הנתונים SQLite שלו ב-`~/.local/share/opencode/opencode.db` דרך `opencode db --format json` (אנו קוראים את טבלאות ה-`session` וה-`project` ומקבצים לפי `project_id`); פרויקטי Pi מתגלים על ידי סריקת תמלולי JSONL לכל הפעלה תחת `~/.pi/agent/sessions/<encoded-cwd>/<timestamp>_<uuid>.jsonl` (ניתן להגדרה דרך `PI_SESSIONS_DIR`) ומשיכת ה-`cwd` משיא הראשון של כל הפעלה; פרויקטי Gemini CLI מתגלים על ידי סריקת `~/.gemini/tmp/<basename>/chats/session-<timestamp>-<uuid-prefix>.jsonl` (ניתן להגדרה דרך `GEMINI_SESSIONS_DIR`) והשחזור של ה-cwd הקנוני מסמן טקסט `.project_root` השכן. פרויקט ששימש בו CLIs מרובים מופיע כשורה יחידה עם כל התגים התואמים. השתמש בתפריט הנפתח **CLI** מעל הטבלה כדי לסנן לפי CLI ספציפי של סוכן; כתובת ה-URL שומרת על הבחירה שלך כ-`?cli=claude|codex|copilot|cursor|opencode|pi|gemini`.

כל פרויקט מציג:
- שם הפרויקט (נגזר מנתיב התיקיה)
- תג CLI — `Claude Code` (כתום), `OpenAI Codex` (סגול), `GitHub Copilot` (כחול), `Cursor Agent` (אזמרגד), `OpenCode` (ענבר), `Pi` (ורוד), ו/או `Gemini CLI` (שמיים)
- תאריך של פעילות ההפעלה האחרונה

לחץ על פרויקט כדי לראות את ההפעלות שלו.

### Sessions

מציג את כל ההפעלות בתוך פרויקט. כל הפעלה מציגה:
- ID הפעלה
- ציוני זמן התחלה וסיום
- מספר קריאות כלים
- ספירת פעילות ווידאו (מדיניות שנכנסה לפעולה)

השתמש בסנן טווח התאריכים וחיפוש ID הפעלה כדי לצמצם את הרשימה. הפעלות עמודות בדפי פרסום.

לחץ על הפעלה כדי לפתוח את מציג ההפעלה.

### מציג הפעלה

מציג ההפעלה עונה על השאלה המרכזית לסוכנים אוטונומיים: מה עשה הסוכן, והאם הוא נשמר על הגדול? תג CLI ליד הכותרת מציין האם ההפעלה היא תמלול Claude Code, OpenAI Codex, GitHub Copilot CLI, Cursor Agent, OpenCode, Pi, או Gemini CLI. זה מציג קו זמן של כל מה שקרה בהפעלה:

- **Messages** - תגובות טקסט של Claude והנתות משתמש
- **Tool calls** - כל כלי שקראה Claude, עם קלט ופלט שלו
- **Policy activity** - עבור כל קריאת כלים, אילו מדיניות נכנסו לפעולה והחלטה שהחזירו

סרגל הנתונים בחלק העליון מציג משך ההפעלה, סך קריאות הכלים וסיכום החלטות ווידאו (ספירות allow / deny / instruct).

לחץ על כפתור **Download Logs** כדי לייצא את ההפעלה. עבור Claude Code, Codex, Copilot, Cursor, Pi, והפעלות Gemini אתה מקבל את תמלול JSONL המקורי על הדיסק בתור-לבתור; עבור OpenCode (שההפעלות שלו חיות ב-SQLite, לא על הדיסק) אתה מקבל מסמך JSON המשקף את טבלאות `session` / `messages` / `parts` הבסיסיות.

### Audit

דוח מונע אישיות של איך הסוכן שלך כן התנהג על פני הפעלות עבר. מריץ את אותה סריקה כמו CLI `failproofai audit` אבל משרטט אותה כלוח בקרה של שישה חלקים:

1. **Identity** — סיווג הסוכן שלך לאחד משמונה ארכיטיפים (`the optimist`, `the cowboy`, `the explorer`, `the goldfish`, `the paranoid architect`, `the precision builder`, `the hammer`, `the ghost`) בהתאם לתגלים וההדיניות שנכנסו לפעולה וכמה כבד. משרטט סיגיל של 8×8 פיקסל, קו הכרזה של הארכיטיפ, מסגור של "common in" / "primary risk", וההגנה החד-קווית.
2. **Show off your agent** — לוכדת את כרטיס הזהות כ-PNG של 1200×630 מתאים לפרסום ל-X / LinkedIn (לחץ על `make poster`).
3. **Strengths** — התנהגויות מסומנות בירוק שהסוכן שלך כבר עושה נכון, שנגזרו מנתוני הביקורת החיים (שיעור קריאות כלים נקי, אורך הפעלה ממוצע, אפס דליפות אישורים, אפס סערות ניסיון חוזר וכו').
4. **Score + leaderboard** — ניקוד 0–100 עם דרגת אות (S/A/B/C/D/F), היסטוגרמה הפצה המראה היכן אתה נופל בקוהורט, פרוזה (a B starts at 71. you're 13 points away.), וטבלת הדירוג עם השורה שלך מודגשת.
5. **Findings** — כרטיסי לכל ממצא מדורגים לפי השפעה. כל כרטיס משטח מה קרה, מה זה עולה, דוגמה הוכחה עם פקודות שנתפסו בפועל, והמדיניות failproofai שתתפסה את אותו דפוס (`$ failproofai policy add <slug>`, click-to-copy).
6. **Prescribed policies + return loop** — רשת של כל מדיניות מקוט-מוקצע לא מאופשרת שתסגור פער, עם הערכת ניקוד משוער, בתוספת קריאת פעולה של re-audit in 7 days.

מונע על ידי זמן ריצה `failproofai audit` — ראה [Audit CLI](/he/cli/audit) לתוך מנוע הסריקה, דגלים נתמכים, ובלתי-משתנים מטמון לכל תמלול. לוח הבקרה מטמן את התוצאה האחרונה ב-`~/.failproofai/audit-dashboard.json` (מצב `0600`, חריץ יחיד, ריצות חדשות דורסות) כך שביקורים חוזרים מיידיים; **גם מטמון התמלול בודד וגם כל התוצאה נדחים בקריאה ברגע שהם גדולים מ-7 ימים** כך שהלוח לא משמש בשקט תוצאה בת שבוע. סרגל בחלק העליון של הדף תמיד מציג את ציון הזמן של הריצה האחרונה (`audited 3d ago`), שבב ענבר `expires in 14h — re-audit to refresh` בשעות ה-24 האחרונות של החלון, ובנר `audit expired` אחרי ה-TTL. לחיצה על `[ re-audit ]` (למעלה) או `[ re-audit now ]` (למטה) POSTs `/api/audit/run` ולוח הבקרה סוקרים את `/api/audit/status` ב-1Hz עד שהריצה מסתיימת; רצועת התקדמות ורודה דביקה משתרעת לחלק העליון של התצוגה במהלך הריצה עם טיימר שחלף, והתוצאה מחליפה במקום בעת הצלחה (אין טעינת עמוד מלאה). בכשל הרצועה הופכת לאדום עם עותק שמפתח את `RerunError.kind` (`timeout` / `network` / `post_failed`). מדינה ריקה (אין מטמון) ומדינה של אפס-הפעלות (מטמון קיים אבל הסריקה לא מצאה תמלולים) משתפות לחוד.

## Auto-refresh

לוח הבקרה יש החלפת auto-refresh בניווט העליון. כשמופעל, הדף הנוכחי רענן בתדירות קבועה כדי להציג הפעלות חדשות ופעילות מדיניות בעת הופעתם. חיוני לניטור הפעלות סוכנים אוטונומיים זמן-ריצה ארוך.

---

## השבתת דפים

אם אתה צריך רק חלקים מסוימים מלוח הבקרה, קבע `FAILPROOFAI_DISABLE_PAGES` לרשימה מופרדת-בפסיקים של שמות דף:

```bash
FAILPROOFAI_DISABLE_PAGES=policies failproofai
```

ערכים תקפים: `policies`, `projects`, `audit`.

## הגדרת נתיב הפרויקטים

כברירת מחדל, לוח הבקרה קורא מתיקיית הפרויקטים הסטנדרטית של Claude Code. החלף אותו עבור הגדרות מותאמות אישית:

```bash
CLAUDE_PROJECTS_PATH=/custom/path/to/projects failproofai
```

## גישה מ-non-localhost host

כשמריצים את לוח הבקרה ב-**dev mode** (`npm run dev`) וגישה אליו מ-hostname שאינה `localhost` - למשל, תחום מותאם אישית, IP מרחוק, או כתובת URL עם מנהרה - ייתכן שתראה אזהרה כמו:

```text
⚠ Blocked cross-origin request to Next.js dev resource /_next/webpack-hmr from "dashboard.example.com".
```

זה Next.js חוסם גישה בחציית מקורות לווידאו HMR (hot module reload) שלו, שהיא תכונה של dev בלבד. כדי לאפשר את ה-host שלך, השתמש בדגל `--allowed-origins`:

```bash
npm run dev -- --allowed-origins dashboard.example.com
```

עבור מספר hosts או IPs, עבור רשימה מופרדת-בפסיקים:

```bash
npm run dev -- --allowed-origins dashboard.example.com,192.168.1.5
```

אתה יכול גם לקבוע את משתנה הסביבה `FAILPROOFAI_ALLOWED_DEV_ORIGINS` במקום זאת:

```bash
FAILPROOFAI_ALLOWED_DEV_ORIGINS=dashboard.example.com npm run dev
```

<Note>
זה חל רק ל-dev mode. כשמריצים `failproofai` (production mode), אין ווידאו HMR ואין בעיית dev resource בחציית מקורות.
</Note>
