מסמך זה מסביר כיצד failproofai עובד באופן פנימי: כיצד מערכת ה-hook מיירטת קריאות כלי סוכן, כיצד תצורה נטענת ומתמזגת, כיצד מדיניויות מוערכות, וכיצד לוח הבקרה מעקב אחר פעילות הסוכן.

---

## סקירה כללית

ל-failproofai יש שתי תת-מערכות עצמאיות:

1. **Hook handler** - תהליך CLI מהיר שـ Claude Code משדר בכל קריאת כלי סוכן. מערכת זו מעריכה מדיניויות ומחזירה החלטה.
2. **Agent Monitor (Dashboard)** - יישום Next.js לניטור מושבי סוכן וניהול מדיניויות.

שתי התת-מערכות חולקות קובצי תצורה בתיקיות `~/.failproofai/` ובתיקיית `.failproofai/` של הפרויקט, אך הן רצות כתהליכים נפרדים ותקשורו רק דרך מערכת הקבצים.

## Hook handler

### אינטגרציה עם Claude Code

כשאתה מריץ את `failproofai policies --install`, הוא כותב ערכים כאלו לתוך `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code משדר אז את `failproofai --hook PreToolUse` כתהליך משנה לפני כל קריאת כלי, ומעביר עומס JSON ב-stdin.

### פורמט העומס

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

עבור אירועי `PostToolUse`, העומס גם כולל את `tool_result` עם הפלט של הכלי.

המטפל אוכף הגבלת stdin של 1 MB. עומסים העולים על זה מושלכים וכל המדיניויות מאפשרות באופן מובהק.

### פורמט התגובה

**Deny (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Deny (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Instruct (כל אירוע חוץ מ-Stop):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**אירוע Stop instruct:**
- קוד יציאה: `2`
- סיבה כתובה ל-stderr (לא stdout)

**Allow:**
- קוד יציאה: `0`
- stdout ריק

**Allow עם הודעה:**

`allow(message)` מאפשר למדיניות לשלוח הקשר מידע בחזרה ל-Claude אפילו כשהפעולה מותרת. מטפל ה-hook כותב את ה-JSON הבא ל-**stdout** (לא קובץ תצורה — זו התגובה של מטפל ה-hook ל-Claude Code, בדיוק כמו תגובות deny ו-instruct למעלה):

```json
// Written to stdout by the hook handler process
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- קוד יציאה: `0` (הפעולה מותרת)
- כאשר מדיניויות מרובות מחזירות `allow` עם הודעה, הודעותיהן מצורפות עם קווים חדשים ל-`additionalContext` יחיד
- אם אף מדיניות לא מספקת הודעה, stdout ריק (כמו קודם)

### צינור עיבוד

`src/hooks/handler.ts` מיישם את הצינור המלא:

```text
stdin JSON
  → parse payload (max 1 MB)
  → extract session metadata (session_id, cwd, tool_name, tool_input, etc.)
  → readMergedHooksConfig(cwd)    ← merges project + local + global config
  → register enabled builtin policies with resolved params
  → load custom policies from customPoliciesPath (if set)
  → register custom policies into policy registry
  → evaluate all policies (builtins first, then custom)
      → first deny short-circuits
      → instruct decisions accumulate
      → allow messages accumulate
  → write JSON decision to stdout
  → persist event to ~/.failproofai/hook-activity.jsonl
  → exit
```

כל התהליך רץ בפחות מ-100ms עבור עומסים טיפוסיים ללא קריאות LLM.

## טעינת תצורה

`src/hooks/hooks-config.ts` מיישם טעינת תצורה בעלת שלוש תחומים.

```text
[1] {cwd}/.failproofai/policies-config.json        ← project  (highest priority)
[2] {cwd}/.failproofai/policies-config.local.json  ← local
[3] ~/.failproofai/policies-config.json             ← global   (lowest priority)
```

לוגיקת מיזוג:
- `enabledPolicies` - איחוד בחסר כפילויות בכל שלושת הקבצים
- `policyParams` - מפתח לכל מדיניות, הקובץ הראשון שמגדיר אותו מנצח לחלוטין
- `customPoliciesPath` - הקובץ הראשון שמגדיר אותו מנצח
- `llm` - הקובץ הראשון שמגדיר אותו מנצח

לוח הבקרה של האינטרנט משתמש ב-`readHooksConfig()` (גלובלי בלבד) לקריאה וכתיבה, מכיוון שהוא לא משדר עם cwd של פרויקט.

## הערכת מדיניות

`src/hooks/policy-evaluator.ts` מריץ מדיניויות בסדר.

לכל מדיניות:

1. חפש את סכימת `params` של המדיניות (אם יש לה כזו).
2. קרא את `policyParams[policy.name]` מהתצורה המתמזגת.
3. מזג ערכים שסופק משתמש על ערכי ברירת מחדל של סכימה כדי להייצר `ctx.params`.
4. קרא ל-`policy.fn(ctx)` עם ההקשר המוחזר.
5. אם התוצאה היא `deny`, עצור מיד וחזור להחלטה זו.
6. אם התוצאה היא `instruct`, צבור את ההודעה והמשך.
7. אם התוצאה היא `allow`, המשך למדיניות הבאה.

לאחר שכל המדיניויות רצות:
- אם כל `deny` הוחזר, פלוט את תגובת deny.
- אם כל הערכי `instruct` שהוחזרו נאספו, פלוט תגובת instruct יחידה עם כל ההודעות מצורפות.
- אחרת, פלוט תגובת allow (stdout ריק, exit 0).

## מדיניויות מובנות

`src/hooks/builtin-policies.ts` מגדיר את כל 26 המדיניויות המובנות בתור `BuiltinPolicyDefinition` אובייקטים:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

מדיניויות המקבלות `params` מצהירות `PolicyParamsSchema` עם סוגים וערכי ברירת מחדל לכל פרמטר. מעריך המדיניות מזריק ערכים מוחזרים ל-`ctx.params` לפני קריאה ל-`fn`. פונקציות מדיניות קוראות ל-`ctx.params` ללא הגנת null מכיוון שערכי ברירת מחדל תמיד מיושמים קודם.

התאמת דפוסים בתוך מדיניויות משתמשת בטוקנים פקודה מנולחים (argv), לא בהתאמה של מחרוזות גולמיות. זה מונע עקיפה דרך הזרקת אופרטור שקר (למשל דפוס עבור `sudo systemctl status *` לא ניתן להיות מעוקף על ידי הוספת `; rm -rf /` לפקודה).

## מדיניויות מותאמות

`src/hooks/custom-hooks-registry.ts` מיישם מרשם המתבסס על `globalThis`:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // used in tests
```

`src/hooks/custom-hooks-loader.ts` טוען את קובץ המדיניות של המשתמש:

1. קרא את `customPoliciesPath` מהתצורה; דלג אם חסר.
2. החזר לנתיב מוחלט; בדוק שהקובץ קיים.
3. כתוב מחדש את כל יבואי `from "failproofai"` לנתיב dist בפועל כדי ש-`customPolicies` יתחיל לאותו מרשם `globalThis`.
4. כתוב מחדש יבואים מקומיים חולפים להבטיח תאימות ESM.
5. כתוב קובצי `.mjs` זמניים ו-`import()` לקובץ הכניסה.
6. קרא ל-`getCustomHooks()` כדי להחזיר hooks רשומים.
7. נקה את כל הקבצים הזמניים בבלוק `finally`.

בכל שגיאה (קובץ לא נמצא, שגיאת תחביר, כישלון בייבוא), השגיאה נרשמת ל-`~/.failproofai/hook.log` והטוען מחזיר מערך ריק. מדיניויות מובנות אינן מושפעות.

מדיניויות מותאמות מוערכות לאחר כל המדיניויות המובנות. `deny` של מדיניות מותאמת עדיין עוצר מדיניויות מותאמות נוספות (אך כל המדיניויות המובנות כבר רצות בשלב זה).

## רישום פעילות

לאחר כל אירוע hook, המטפל מוסיף שורת JSONL ל-`~/.failproofai/hook-activity.jsonl`:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

שורה אחת לכל מדיניות שקיבלה החלטה שאינה allow. החלטות Allow אינן רשומות (כדי להשאיר את הקובץ קטן).

## ארכיטקטורת לוח הבקרה

לוח הבקרה הוא יישום **Next.js 16** המשתמש ב-App Router עם React Server Components ו-Server Actions.

```text
app/
  layout.tsx                  ← Root layout (theme, telemetry, nav)
  projects/page.tsx           ← Server component: list all Claude projects
  project/[name]/page.tsx     ← Server component: list sessions in a project
  project/[name]/session/
    [sessionId]/page.tsx      ← Server component: render session viewer
  policies/page.tsx           ← Client component: policy management + activity log
  actions/
    get-hooks-config.ts       ← Read config + policy list
    update-hooks-config.ts    ← Toggle policy on/off
    update-policy-params.ts   ← Update policy parameters
    get-hook-activity.ts      ← Paginate/search activity log
    install-hooks-web.ts      ← Install/remove hooks from the browser
  api/
    download/[project]/[session]/route.ts   ← Per-CLI session export (JSONL or JSON)
```

**זרימת נתונים:**

- רכיבי עמוד קוראים ל-`lib/projects.ts` ו-`lib/log-entries.ts` כדי לקרוא נתוני פרויקט/מושב ישירות ממערכת הקבצים (אין שכבת API לקריאות).
- עמוד Policies משתמש בServer Actions עבור כל המוטציות (החלפה, עדכון פרמטרים, התקנה/הסרה).
- צופה המושב מנתח את פורמט הטרנסקריפט JSONL של Claude ומשרטט ציר הזמן של הודעות וקריאות כלים.

**החלטות עיצוב עיקריות:**

- אין בסיס נתונים - כל המצב קבוע בקבצים רגילים (`~/.failproofai/`, `~/.claude/projects/`).
- Server Actions למוטציות - אין צורך ב-REST API עבור פעולות CRUD.
- React Server Components לעמודי קריאה - טעינה ראשונית מהירה יותר, ללא צרור לקוח עבור הביאת נתונים.
- רכיבים של לקוח רק כאשר יש צורך באינטראקטיביות (החלפות מדיניות, חיפוש פעילות, צופה יומן).
