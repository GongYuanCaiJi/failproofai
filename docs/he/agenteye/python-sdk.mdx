ראו בדיוק מה שהסוכנים שלכם עשו בייצור: כל הרצת סוכן, קריאת כלי, בקשת מודל, hook וגם התערבות אנושית. ה-SDK של FailproofAI Observability מתעד את השביל הזה מתוך קוד הסוכן שלכם כדי שתוכלו לבצע ניפוי שגיאות, ביקורת והערכה של מה שהתרחש. השתמשו בו בכל פעם שתרצו ש-FailproofAI Observability יעקוב אחר הסוכנים שלכם.

## התחלה מהירה

```python
import agenteye

agenteye.configure(environment="production")

agenteye.event.agent_start(session_id="run-001", agent_id="planner", goal="answer user query")

agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",
    input={"query": "latest AI research"},
)

agenteye.event.tool_result(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",
    output={"results": ["..."]},
)

agenteye.event.agent_end(session_id="run-001", agent_id="planner", outcome="success")
```

### ציוד של קריאה אמיתית

בפועל, אתם עוטפים את קוד הסוכן הקיים שלכם. יקפו קריאת מודל עם `model_request` לפני ו-`model_response` אחרי, כך ששני האירועים חופפים את הבקשה האמיתית וה-FailproofAI Observability יכול לשייך אותם:

```python
import anthropic
import agenteye

agenteye.configure(environment="production")
client = anthropic.Anthropic()

messages = [{"role": "user", "content": "Summarise today's incidents."}]

agenteye.event.model_request(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6",
    messages=messages,
)

reply = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    messages=messages,
)

agenteye.event.model_response(
    session_id="run-001",
    agent_id="planner",
    model=reply.model,
    stop_reason=reply.stop_reason,
    input_tokens=reply.usage.input_tokens,
    output_tokens=reply.usage.output_tokens,
    content=[block.model_dump() for block in reply.content],
)
```

עטפו קריאות כלי באותו אופן עם `tool_use` ו-`tool_result`, תוך שימוש חוזר ב-`tool_call_id` אחד בין השניים.

הנה איך נראים אירועים אלה כאשר הם מגיעים לדashboard, מקודדים בצבעים לפי סוג וניתנים לסינון לפי environment, סוכן וsession:

## configure()

```python
agenteye.configure(
    base_dir=None,           # Path | str | None. Default: $AGENTEYE_HOME or ~/.agenteye
    flush_interval=0.5,      # float, seconds between flush cycles
    environment=None,        # str | None. Deployment environment label
)
```

קראו פעם אחת לפני כל קריאה `event.*`. בטוח להשמיט; ברירות המחדל עובדות מעוקב. כל הארגומנטים הם keyword-only; העבירו אותם לפי שם כמו מוצג למעלה.

כאשר `base_dir` הוא `None` (ברירת המחדל), ה-SDK קורא את `$AGENTEYE_HOME` אם הוגדר,
אחרת חוזר ל-`~/.agenteye`. זה תואם את ההחלטה של הקולקטור עצמו,
כך שמשתנה `AGENTEYE_HOME` אחד מגדיר את ה-event spool המשותף עבור
ה-SDK והקולקטור.

## Environment

תייגו כל אירוע עם environment של פריסה (`production`, `staging`, `qa`, `canary`, וכו'). קבעו אותו פעם אחת; ה-SDK מצרף אותו לכל אירוע באופן אוטומטי.

**אפשרות 1: דרך `configure()`:**

```python
agenteye.configure(environment="production")
```

**אפשרות 2: דרך משתנה environment:**

```bash
export AGENTEYE_ENVIRONMENT=production
```

**עדיפות:** `configure(environment=...)` מנצח על משתנה ה-environment. אם לא קבעתם אף אחד מהם, ברירת המחדל היא `"dev"`.

ערך ה-environment מופיע כמסנן מדרגה ראשונה בדashboard ומאוחסן בשרת לעריכות מהירות.

> **אזהרה:** ערכי environment לא חייבים להכיל קומה `,` מילולית. מסנני הדashboard משתמשים בבחירה מרובה המופרדת בפסיקים על הציוד (`?environment=prod,staging`), כך שenvironment בשם `prod,blue` יהיה מפוצל לשני ערכים. אירועים עם environments המכילים פסיקים נדחים בזמן הנתינה.

## נתונים ופרטיות

ה-SDK מתעד רק את השדות שהעברתם במפורש. Prompts, הודעות, tool inputs ו-outputs, ותוכן מודל נתפסים אך ורק מכיוון שהעברתם אותם לקריאת `event.*`. שום דבר לא נקרא מתהליך שלכם ולא נתפס במרומז. כל שדה שלא קבעתם הוא מושמט מהאירוע לחלוטין; הוא לא נכתב לדיסק.

זה הופך את ההסתרה לבחירה שלכם ולאחריות שלכם. אם prompt או tool payload מכיל PII או סודות שלא תרצו לאחסן, הסירו או מסכו אותו לפני שהעברתם אותו לשיטת האירוע.

## Event Reference

רוב האירועים מגיעים בזוגות start/end החולקים ID מתאם: `tool_use` ו-`tool_result` חולקים `tool_call_id`, `hook_triggered` ו-`hook_completed` חולקים `hook_id`, ו-`human_wait` ו-`human_input` חולקים `input_id`. פעילו את אירוע ה-start, בצעו את העבודה, ואז פעילו את אירוע ה-end עם אותו ID. FailproofAI Observability משייך את הזוג ומחשב `duration_ms` עבורכם, כך שלעולם לא תעבירו `duration_ms` בעצמכם.

![גרף ביצוע בסגנון git של session לצד ציר הזמן של האירועים שלו, שנבנה מחדש מהאירועים המזווגים, עם פנל המפרק tool/model/hook](/agenteye/images/session-detail.png)

כל שיטות האירוע דורשות את שני השדות הללו:

| שדה | סוג | תיאור |
|---|---|---|
| `session_id` | `str` | מזהה את הרצת הסוכן ברמה העליונה |
| `agent_id` | `str` | מזהה איזה סוכן בתוך ה-session פעיל את האירוע |

כל השיטות גם מקבלות `**kwargs` שרירותיות עבור מטא-נתונים מותאמים (ראו [Custom Fields](#custom-fields)).

---

### `event.agent_start()`

פעיל כאשר סוכן מתחיל לעבוד.

```python
agenteye.event.agent_start(
    session_id="run-001",
    agent_id="planner",
    goal="answer user query",   # str | None
    parent_id=None,             # str | None - parent agent_id for nested agents
)
```

---

### `event.agent_end()`

פעיל כאשר סוכן מסיים לעבוד.

```python
agenteye.event.agent_end(
    session_id="run-001",
    agent_id="planner",
    outcome="success",          # str | None
    summary="Answered query",   # str | None
)
```

---

### `event.tool_use()`

פעיל כאשר סוכן משתמש בכלי. צווא עם `tool_result`; ה-SDK מחשב אוטומטית את `duration_ms`.

```python
agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",     # str, required
    tool_call_id="toolu_01",    # str, required - correlation key for the matching tool_result
    input={"query": "..."},     # dict | None
)
```

---

### `event.tool_result()`

פעיל כאשר כלי חוזר. משויך עם `tool_use` דרך `tool_call_id`.

```python
agenteye.event.tool_result(
    session_id="run-001",
    agent_id="planner",
    tool_name="web_search",
    tool_call_id="toolu_01",        # must match the prior tool_use
    output={"results": ["..."]},    # Any | None
    error=None,                     # str | None - set if the tool raised
    # duration_ms is computed automatically - do not pass it
)
```

---

### `event.model_request()`

פעיל רגע לפני שליחת prompt ל-LLM.

```python
agenteye.event.model_request(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6", # str | None - any provider/model string; not validated
    messages=[                   # list[dict] | None - conversation turns
        {"role": "user", "content": "..."},
    ],
    system="You are helpful.",   # Any | None - str or list of content blocks
    tools=[                      # list[dict] | None - tool schemas offered to the model
        {"name": "search", "input_schema": {"type": "object"}},
    ],
)
```

ערכי `messages` מקבלים או `content` מחרוזת רגילה או Anthropic-style list-of-blocks `content`. ניתן להעביר פרמטרי דגימה (`temperature`, `max_tokens`, וכו') כ-kwargs נוסף.

---

### `event.model_response()`

פעיל כאשר ה-LLM חוזר עם תגובה.

```python
agenteye.event.model_response(
    session_id="run-001",
    agent_id="planner",
    model="claude-sonnet-4-6", # str | None - any provider/model string; not validated
    stop_reason="end_turn",     # str | None
    input_tokens=1024,          # int | None
    output_tokens=256,          # int | None
    content=[                    # Any | None - str, or list of content blocks
        {"type": "text", "text": "..."},
    ],
    role="assistant",            # str | None
)
```

`content` מקבל או מחרוזת רגילה (ספקים גנריים) או רשימה של Anthropic-style content blocks. קריאות כלי מגרות בתוך `content` כ-`{"type": "tool_use", ...}` blocks, ללא שדה `tool_calls` נפרד.

---

### `event.hook_triggered()`

פעיל כאשר hook משתלח. צווא עם `hook_completed`; ה-SDK מחשב אוטומטית את `duration_ms`.

```python
agenteye.event.hook_triggered(
    session_id="run-001",
    agent_id="planner",
    hook_name="pre_tool_use",   # str, required
    hook_id="hook-abc",         # str, required - correlation key
    trigger_event="tool_use",   # str | None
    input={"tool": "search"},   # Any | None
)
```

---

### `event.hook_completed()`

פעיל כאשר hook מסיים. משויך עם `hook_triggered` דרך `hook_id`.

```python
agenteye.event.hook_completed(
    session_id="run-001",
    agent_id="planner",
    hook_name="pre_tool_use",
    hook_id="hook-abc",         # must match the prior hook_triggered
    outcome="allow",            # str | None
    output=None,                # Any | None
    error=None,                 # str | None
    # duration_ms is computed automatically - do not pass it
)
```

---

### `event.error()`

פעיל כאשר שגיאה לא מטופלת מתרחשת.

```python
agenteye.event.error(
    session_id="run-001",
    agent_id="planner",
    error_type="TimeoutError",  # str, required
    message="timed out",        # str, required
    traceback="Traceback...",   # str | None
)
```

## Human-in-the-Loop Events

אירועי human-in-the-loop מעניקים לכם שליטה על הרגעים בהם אדם צועד לביצוע הסוכן (מחכה לאישור, מספק קלט, מעצור או עוצר את הסוכן). הם מאפשרים לכם למדוד כמה זמן אנשים לוקחים להשיב (ה-SDK מחשב אוטומטית את `duration_ms` על האירועים המזווגים), ביקורת שעצר או הפריע לסוכן, ובנייה של workflows של אישור ופיקוח המופיעים בדashboard.

### `event.human_wait()`

פעיל כאשר הסוכן עוצר את הביצוע להמתנה לאדם להספקת קלט. צווא עם `human_input`; ה-SDK מחשב אוטומטית את `duration_ms` (כמה זמן אדם לקח להשיב).

```python
agenteye.event.human_wait(
    session_id="run-001",
    agent_id="planner",
    input_id="inp-abc",                          # str, required - correlation key for the matching human_input
    prompt="Do you approve this action?",        # str | None - the question shown to the human
    options=["approve", "reject", "defer"],      # list[str] | None - choices presented to the human
    reason="approval_required",                  # str | None - why the agent is waiting
)
```

### `event.human_input()`

פעיל כאשר אדם מספק קלט והסוכן חוזר. משויך עם `human_wait` דרך `input_id`. `duration_ms` מחושב אוטומטית ולא חייב להיות עבור הקורא.

```python
agenteye.event.human_input(
    session_id="run-001",
    agent_id="planner",
    input_id="inp-abc",    # str, required - must match the prior human_wait
    response="approve",    # str | None - the human's answer (free text or selected option)
    # duration_ms is computed automatically - do not pass it
)
```

### `event.human_pause()`

פעיל כאשר אדם בפעולה עוצר את הסוכן (לדוגמה דרך בקרת דashboard). הסוכן מושהה אך לא מוגבל.

```python
agenteye.event.human_pause(
    session_id="run-001",
    agent_id="planner",
    reason="user_requested",  # str | None
    user_id="usr_42",         # str | None - who paused the agent
)
```

### `event.human_interrupt()`

פעיל כאשר אדם בפעולה עוצר את הסוכן במהלך הביצוע. בניגוד ל-`human_pause`, עבודת הסוכן מסתיימת ולא מושהית.

```python
agenteye.event.human_interrupt(
    session_id="run-001",
    agent_id="planner",
    reason="output_incorrect",       # str | None
    user_id="usr_42",                # str | None - who interrupted the agent
    at_step="tool_use:web_search",   # str | None - what the agent was doing when stopped
)
```

## Custom Fields

כל ארגומנטים keyword נוסף מצורפים לאירוע לאחר השדות הסטנדרטיים:

```python
agenteye.event.tool_use(
    session_id="run-001",
    agent_id="planner",
    tool_name="db_query",
    tool_call_id="toolu_02",
    tenant_id="acme",           # custom field
    region="us-east-1",         # custom field
)
```

`timestamp`, `type` ו-`environment` שמורים והעלו `ValueError` (`Reserved field names cannot be used as custom fields: [...]`) אם הועברו כשדות מותאמים. `session_id` ו-`agent_id` הם פרמטרים דרושים בכל שיטת אירוע ולא ניתן לספק אותם פעם שנייה; Python העלו `TypeError` אם תעשו. קבעו את ה-environment עם `configure(environment=...)` (או משתנה `AGENTEYE_ENVIRONMENT`) במקום.

## איך אירועים נכתבים

אירועים מושהים בתוך התהליך ומדוללים לדיסק כל `flush_interval` שניות (ברירת מחדל 500 ms). כל flush כותב קובץ JSONL אחד:

```text
~/.agenteye/events/event-2026-04-01T12-00-00-000Z.jsonl
```

הקולקטור שומר בתיקייה זו ועלה קבצים באופן אוטומטי. אתם לא צריכים לנהל את הקבצים הללו ישירות.

כל קובץ נכתב בצורה אטומית: ה-SDK כותב לקובץ זמני ואז משם אותו במקום, כך שהקולקטור לעולם לא רואה קובץ למחצה כתוב. דדול סופי גם רץ כאשר התהליך שלכם יוצא, כך שאירועים במאגר בתוך המרווח האחרון לא הולכים לאיבוד. אם הקולקטור לא מקוון, אירועים פשוט מצטברים כקבצים על דיסק והם שולחים ברגע שהוא חוזר.
