Bu belge, failproofai'nin dahili olarak nasıl çalıştığını açıklar: hook sistemi aracı araç çağrılarını nasıl kesintiye uğratır, yapılandırma nasıl yüklenir ve birleştirilir, politikalar nasıl değerlendirilir ve dashboard aracı aktivitesini nasıl izler.

---

## Genel Bakış

failproofai'nin iki bağımsız alt sistemi vardır:

1. **Hook işleyici** - Claude Code'un her aracı araç çağrısında çağırdığı hızlı bir CLI alt işlemi. Politikaları değerlendirir ve bir karar döndürür.
2. **Agent Monitor (Dashboard)** - Aracı oturumlarını izlemek ve politikaları yönetmek için bir Next.js web uygulaması.

Her iki alt sistem de `~/.failproofai/` ve projenin `.failproofai/` dizininde yapılandırma dosyalarını paylaşır, ancak ayrı işlemler olarak çalışırlar ve yalnızca dosya sistemi aracılığıyla iletişim kurarlar.

## Hook işleyici

### Claude Code ile Entegrasyon

`failproofai policies --install` komutunu çalıştırdığınızda, `~/.claude/settings.json` içine şu şekilde girdiler yazılır:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "failproofai --hook PreToolUse"
          }
        ]
      }
    ],
    "PostToolUse": [ ... ]
  }
}
```

Claude Code daha sonra her araç çağrısından önce `failproofai --hook PreToolUse` komutunu bir alt işlem olarak çağırır ve stdin üzerinden bir JSON yükü iletir.

### Yük Biçimi

```json
{
  "session_id": "abc123",
  "transcript_path": "/home/user/.claude/projects/myproject/sessions/abc123.jsonl",
  "cwd": "/home/user/myproject",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": { "command": "sudo apt install nodejs" }
}
```

`PostToolUse` etkinlikleri için, yük ayrıca aracının çıktısını içeren `tool_result` değerini içerir.

İşleyici 1 MB stdin sınırını uygular. Bu limiti aşan yükler atılır ve tüm politikalar örtük olarak izin verir.

### Yanıt Biçimi

**Reddet (PreToolUse):**
```json
{
  "hookSpecificOutput": {
    "permissionDecision": "deny",
    "permissionDecisionReason": "Blocked by failproofai: sudo command blocked"
  }
}
```

**Reddet (PostToolUse):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Blocked by failproofai because: API key detected in output"
  }
}
```

**Talimat (Stop dışında herhangi bir etkinlik):**
```json
{
  "hookSpecificOutput": {
    "additionalContext": "Instruction from failproofai: Verify tests pass before committing."
  }
}
```

**Stop etkinliği talimatı:**
- Çıkış kodu: `2`
- Neden stderr'a yazılır (stdout değil)

**İzin Ver:**
- Çıkış kodu: `0`
- Boş stdout

**İletiyle İzin Ver:**

`allow(message)` bir politikanın işlem izin verildiğinde bile Claude'a bilgilendirici bağlam göndermesini sağlar. Hook işleyici aşağıdaki JSON'ı **stdout'a** yazar (yapılandırma dosyasına değil — bu, deny ve instruct yanıtları gibi hook işleyicisinin Claude Code'a verdiği yanıttır):

```json
// Hook işleyici işlemi tarafından stdout'a yazılır
{
  "hookSpecificOutput": {
    "additionalContext": "All CI checks passed on branch 'feat/my-feature'."
  }
}
```
- Çıkış kodu: `0` (işleme izin verilir)
- Birden fazla politika bir iletiyle `allow` döndürdüğünde, iletileri newline'larla birleştirilerek tek bir `additionalContext` dizesine katılır
- Hiçbir politika bir ileti sağlamadığında, stdout boştur (öncekiyle aynı)

### İşleme Hattı

`src/hooks/handler.ts` tam hattı uygular:

```text
stdin JSON
  → yükü ayrıştır (en fazla 1 MB)
  → oturum meta verilerini çıkar (session_id, cwd, tool_name, tool_input, vb.)
  → readMergedHooksConfig(cwd)    ← proje + yerel + genel yapılandırmayı birleştirir
  → etkinleştirilen yerleşik politikaları çözümlenen parametrelerle kayıt et
  → özel politikaları customPoliciesPath'ten yükle (ayarlanmışsa)
  → özel politikaları politika kaydına kayıt et
  → tüm politikaları değerlendir (önce yerleşikler, sonra özel olanlar)
      → ilk reddet kısa devre
      → talimat kararları birikir
      → izin mesajları birikir
  → JSON kararını stdout'a yaz
  → etkinliği ~/.failproofai/hook-activity.jsonl'ye kayıt et
  → çık
```

Tipik yükler için LLM çağrıları olmadan tüm işlem 100ms altında gerçekleşir.

## Yapılandırma Yükleme

`src/hooks/hooks-config.ts` üç kapsamlı yapılandırma yüklemesi uygular.

```text
[1] {cwd}/.failproofai/policies-config.json        ← proje  (en yüksek öncelik)
[2] {cwd}/.failproofai/policies-config.local.json  ← yerel
[3] ~/.failproofai/policies-config.json             ← genel   (en düşük öncelik)
```

Birleştirme mantığı:
- `enabledPolicies` - üç dosya arasında çoğaltılmamış birleşim
- `policyParams` - politika başına anahtar, ilk dosya tamamen kazanır
- `customPoliciesPath` - tanımlayan ilk dosya kazanır
- `llm` - tanımlayan ilk dosya kazanır

Web dashboard, proje cwd'si ile çağrılmadığından, yalnızca genel yapılandırma okuması ve yazması için `readHooksConfig()` (yalnızca genel) kullanır.

## Politika Değerlendirmesi

`src/hooks/policy-evaluator.ts` politikaları sırayla çalıştırır.

Her politika için:

1. Politikanın `params` şemasını arayın (varsa).
2. Birleştirilen yapılandırmadan `policyParams[policy.name]` okuyun.
3. Şema varsayılanları üzerine kullanıcı tarafından sağlanan değerleri birleştirerek `ctx.params` üretin.
4. `policy.fn(ctx)` çözümlenen bağlam ile çağırın.
5. Sonuç `deny` ise, hemen durdurun ve bu kararı döndürün.
6. Sonuç `instruct` ise, iletiyi biriktin ve devam edin.
7. Sonuç `allow` ise, sonraki politikaya geçin.

Tüm politikalar çalıştıktan sonra:
- Herhangi bir `deny` döndürüldüyse, deny yanıtını göster.
- Herhangi bir `instruct` dönüşü toplandıysa, tüm iletiler birleştirilmiş tek bir instruct yanıtı göster.
- Aksi takdirde, izin yanıtı göster (boş stdout, çıkış 0).

## Yerleşik Politikalar

`src/hooks/builtin-policies.ts` tüm 26 yerleşik politikaları `BuiltinPolicyDefinition` nesneleri olarak tanımlar:

```typescript
interface BuiltinPolicyDefinition {
  name: string;
  description: string;
  fn: (ctx: PolicyContext) => PolicyResult;
  match: {
    events: HookEventType[];
    tools?: string[];
  };
  defaultEnabled: boolean;
  category: string;
  beta?: boolean;
  params?: PolicyParamsSchema;
}
```

`params` kabul eden politikalar her parametre için türler ve varsayılanlar içeren bir `PolicyParamsSchema` bildirir. Politika değerlendiricisi, `fn`'i çağırmadan önce çözümlenen değerleri `ctx.params`'e enjekte eder. Politika fonksiyonları `ctx.params`'i null koruma olmadan okur çünkü varsayılanlar her zaman önce uygulanır.

Politikalar içindeki desen eşleştirmesi, ham dize eşleştirmesi değil, ayrıştırılmış komut jetonlarını (argv) kullanır. Bu, shell operatörü injeksiyonu aracılığıyla bypass'ı önler (örneğin `sudo systemctl status *` için bir desen, komuta `; rm -rf /` eklenerek bypass edilemez).

## Özel Politikalar

`src/hooks/custom-hooks-registry.ts` bir `globalThis` destekli kayıt uygulaması uygular:

```typescript
const REGISTRY_KEY = "__failproofai_custom_hooks__";

export const customPolicies = {
  add(hook: CustomHook): void { ... }
};

export function getCustomHooks(): CustomHook[] { ... }
export function clearCustomHooks(): void { ... }  // testlerde kullanılır
```

`src/hooks/custom-hooks-loader.ts` kullanıcının politika dosyasını yükler:

1. Yapılandırmadan `customPoliciesPath` okuyun; yoksa atla.
2. Mutlak yola çözün; dosyanın var olup olmadığını kontrol edin.
3. Tüm `from "failproofai"` importlarını gerçek dist yoluna yeniden yazın, böylece `customPolicies` aynı `globalThis` kaydına çözülür.
4. ESM uyumluluğunu sağlamak için geçişli yerel importları özyinelemeli olarak yeniden yazın.
5. Geçici `.mjs` dosyaları yazın ve giriş dosyasını `import()` edin.
6. Kaydedilmiş hook'ları almak için `getCustomHooks()` çağırın.
7. Bir `finally` bloğunda tüm geçici dosyaları temizle.

Herhangi bir hatada (dosya bulunamadı, sözdizimi hatası, import hatası), hata `~/.failproofai/hook.log` dosyasına kaydedilir ve yükleyici boş bir dizi döndürür. Yerleşik politikalar etkilenmez.

Özel politikalar tüm yerleşik politikaların sonrasında değerlendirilir. Özel bir politikanın `deny` değeri yine de ek özel politikaları kısa devre yapar (ancak bu noktada tüm yerleşikler zaten çalışmış durumdadır).

## Aktivite Günlüğü

Her hook etkinliğinden sonra, işleyici `~/.failproofai/hook-activity.jsonl` dosyasına bir JSONL satırı ekler:

```json
{
  "timestamp": "2026-04-06T12:34:56.789Z",
  "sessionId": "abc123",
  "eventType": "PreToolUse",
  "toolName": "Bash",
  "policyName": "block-sudo",
  "decision": "deny",
  "reason": "sudo command blocked by failproofai",
  "durationMs": 12
}
```

Bir izin verilmeyenden karar veren her politika için bir satır. İzin verilen kararlar kaydedilmez (dosyayı küçük tutmak için).

## Dashboard Mimarisi

Dashboard, React Sunucu Bileşenleri ve Sunucu Eylemleri ile App Router kullanan bir **Next.js 16** uygulamasıdır.

```text
app/
  layout.tsx                  ← Kök düzen (tema, telemetri, nav)
  projects/page.tsx           ← Sunucu bileşeni: tüm Claude projelerini listele
  project/[name]/page.tsx     ← Sunucu bileşeni: bir projedeki oturumları listele
  project/[name]/session/
    [sessionId]/page.tsx      ← Sunucu bileşeni: oturum görüntüleyiciyi renderla
  policies/page.tsx           ← İstemci bileşeni: politika yönetimi + aktivite günlüğü
  actions/
    get-hooks-config.ts       ← Yapılandırma + politika listesini oku
    update-hooks-config.ts    ← Politikayı aç/kapat
    update-policy-params.ts   ← Politika parametrelerini güncelle
    get-hook-activity.ts      ← Aktivite günlüğünü sayfalandır/ara
    install-hooks-web.ts      ← Tarayıcıdan hook'ları kur/kaldır
  api/
    download/[project]/[session]/route.ts   ← Proje başına CLI oturum dışa aktarma (JSONL veya JSON)
```

**Veri Akışı:**

- Sayfa bileşenleri proje/oturum verilerini doğrudan dosya sisteminden okumak için `lib/projects.ts` ve `lib/log-entries.ts` çağırır (okumalar için API katmanı yok).
- Policies sayfası tüm mutasyonlar (aç/kapat, parametreler güncellemesi, kur/kaldır) için Sunucu Eylemlerini kullanır.
- Oturum görüntüleyicisi Claude'un JSONL transkript biçimini ayrıştırır ve mesajların ve araç çağrılarının bir zaman çizelgesini renderlar.

**Temel Tasarım Kararları:**

- Veritabanı yok - tüm kalıcı durum düz dosyalardadır (`~/.failproofai/`, `~/.claude/projects/`).
- Mutasyonlar için Sunucu Eylemleri - CRUD işlemleri için REST API gerekli değil.
- Okuma sayfaları için React Sunucu Bileşenleri - daha hızlı ilk yükleme, veri getirme için istemci paket yok.
- İstemci bileşenleri yalnızca etkileşim gerekli olduğunda (politika açma/kapatma, aktivite araması, günlük görüntüleyici).
