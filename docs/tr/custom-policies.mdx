Özel ilkeler, herhangi bir aracı davranışı için kurallar yazmanızı sağlar: proje sözleşmelerini uygulayın, sapmaları engelleyin, yıkıcı işlemleri sınırlayın, takılı kalmış aracıları tespit edin veya Slack, onay iş akışları ve daha pek çok şeyle entegre olun. Yerleşik ilkelerle aynı kanca olay sistemini ve `allow`, `deny`, `instruct` kararlarını kullanırlar.

---

## Hızlı örnek

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

customPolicies.add({
  name: "no-production-writes",
  description: "Block writes to paths containing 'production'",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Write" && ctx.toolName !== "Edit") return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("production")) {
      return deny("Writes to production paths are blocked");
    }
    return allow();
  },
});
```

Yükleyin:

```bash
failproofai policies --install --custom ./my-policies.js
```

## Özel ilkeleri yüklemek için iki yol

### Seçenek 1: Sözleşmeye dayalı (önerilen)

`*policies.{js,mjs,ts}` dosyalarını `.failproofai/policies/` klasörüne bırakın ve otomatik olarak yüklenir — bayrak veya yapılandırma değişikliğine gerek yoktur. Bu, git kancaları gibi çalışır: dosya bırakın, işte oldu.

```
# Proje düzeyi — git'e kaydedilir, ekip ile paylaşılır
.failproofai/policies/security-policies.mjs
.failproofai/policies/workflow-policies.mjs

# Kullanıcı düzeyi — kişisel, tüm projeler için geçerli
~/.failproofai/policies/my-policies.mjs
```

**Nasıl çalışır:**
- Hem proje hem de kullanıcı dizinleri taranır (birleştirme — ilk kapsama kazananı değil)
- Dosyalar her dizin içinde alfabetik sırada yüklenir. Sırayı kontrol etmek için `01-`, `02-` ile öneki ekleyin
- Yalnızca `*policies.{js,mjs,ts}` ile eşleşen dosyalar yüklenir; diğer dosyalar yoksayılır
- Her dosya bağımsız olarak yüklenir (dosya başına açık başarısızlık)
- Açık `--custom` ve yerleşik ilkelerle birlikte çalışır

<Tip>
Sözleşmeye dayalı ilkeler, organizasyonunuz için bir kalite standardı oluşturmanın en kolay yoludur. `.failproofai/policies/` klasörünü git'e kaydedin ve her ekip üyesi otomatik olarak aynı kuralları alır — geliştirici başına kurulum gerekmez. Ekibiniz yeni başarısızlık modları keşfettikçe, bir ilke ekleyin ve gönderin. Zamanla bunlar, her katkı ile iyileşen, yaşayan bir kalite standardı haline gelir.
</Tip>

### Seçenek 2: Açık dosya yolu

```bash
# Özel ilkeler dosyası ile yükleyin
failproofai policies --install --custom ./my-policies.js

# İlkeler dosya yolunu değiştirin
failproofai policies --install --custom ./new-policies.js

# Yapılandırmadan özel ilkeler yolunu kaldırın
failproofai policies --uninstall --custom
```

Çözümlenen mutlak yol, `policies-config.json` içinde `customPoliciesPath` olarak depolanır. Dosya her kanca olayında taze yüklenir - olaylar arasında önbelleğe alma yoktur.

### Her ikisini birlikte kullanma

Sözleşmeye dayalı ilkeler ve açık `--custom` dosyası bir arada bulunabilir. Yükleme sırası:

1. Açık `customPoliciesPath` dosyası (yapılandırılmışsa)
2. Proje sözleşmesi dosyaları (`{cwd}/.failproofai/policies/`, alfabetik)
3. Kullanıcı sözleşmesi dosyaları (`~/.failproofai/policies/`, alfabetik)

## API

### İçe aktarma

```js
import { customPolicies, allow, deny, instruct } from "failproofai";
```

### `customPolicies.add(hook)`

Bir ilke kaydeder. Aynı dosyada birden fazla ilke için gerektiği kadar çağırın.

```ts
customPolicies.add({
  name: string;                         // required - unique identifier
  description?: string;                 // shown in `failproofai policies` output
  match?: { events?: HookEventType[] }; // filter by event type; omit to match all
  fn: (ctx: PolicyContext) => PolicyResult | Promise<PolicyResult>;
});
```

### Karar yardımcıları

| İşlev | Etki | Ne zaman kullanılır |
|-------|------|-------------------|
| `allow()` | İşlemi sessizce izin ver | İşlem güvenli, ileti gerekli değil |
| `deny(message)` | İşlemi engelle | Aracı bu işlemi yapmamalı |
| `instruct(message)` | Engellemeden bağlam ekle | Aracıya sırada kalmak için ek bağlam ver |

`deny(message)` - ileti Claude'a `"Blocked by failproofai:"` ön ekiyle görünür. Tek bir `deny` tüm sonraki değerlendirmeyi kısa devre yapar.

`instruct(message)` - ileti Claude'un mevcut araç çağrısı için bağlamına eklenir. Tüm `instruct` iletileri biriktirilir ve birlikte iletilir.

<Tip>
`policyParams` içine `hint` alanı ekleyerek herhangi bir `deny` veya `instruct` iletisine ek rehberlik ekleyebilirsiniz — kod değişikliğine gerek yoktur. Bu, özel (`custom/`), proje sözleşmesi (`.failproofai-project/`), ve kullanıcı sözleşmesi (`.failproofai-user/`) ilkeleri için de çalışır. Ayrıntılar için [Yapılandırma → hint](/tr/configuration#hint-cross-cutting) bölümüne bakın.
</Tip>

### Bilgilendirici izin iletileri

`allow(message)` işleme izin verir **ve** Claude'a geri bir bilgilendirici ileti gönderir. İleti, kanca işleyicisinin stdout yanıtında `additionalContext` olarak iletilir — `instruct` tarafından kullanılan aynı mekanizma, ancak anlamsal olarak farklı: bir uyarı değil, bir durum güncellemesidir.

| İşlev | Etki | Ne zaman kullanılır |
|-------|------|-------------------|
| `allow(message)` | İzin ver ve Claude'a bağlam gönder | Bir denetimin geçtiğini onaylayın veya bir denetimin neden atlanıldığını açıklayın |

Kullanım durumları:
- **Durum onayları:** `allow("All CI checks passed.")` — Claude'a her şeyin yeşil olduğunu söyler
- **Açık başarısızlık açıklamaları:** `allow("GitHub CLI not installed, skipping CI check.")` — Claude'a bir denetimin neden atlanıldığını söyler; tam bağlama sahip olması için
- **Birden fazla ileti birikiyor:** birkaç ilke her biri `allow(message)` döndürürse, tüm iletiler newline ile birleştirilir ve birlikte iletilir

```js
customPolicies.add({
  name: "confirm-branch-status",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    const cwd = ctx.session?.cwd;
    if (!cwd) return allow("No working directory, skipping branch check.");

    // ... check branch status ...
    if (allPushed) {
      return allow("Branch is up to date with remote.");
    }
    return deny("Unpushed changes detected.");
  },
});
```

### `PolicyContext` alanları

| Alan | Tür | Açıklama |
|------|-----|----------|
| `eventType` | `string` | `"PreToolUse"`, `"PostToolUse"`, `"Notification"`, `"Stop"` |
| `toolName` | `string \| undefined` | Çağrılan araç (örn. `"Bash"`, `"Write"`, `"Read"`) |
| `toolInput` | `Record<string, unknown> \| undefined` | Aracın giriş parametreleri |
| `payload` | `Record<string, unknown>` | Claude Code'dan tam ham olay yükü |
| `session` | `SessionMetadata \| undefined` | Oturum bağlamı (aşağıya bakın) |

### `SessionMetadata` alanları

| Alan | Tür | Açıklama |
|------|-----|----------|
| `sessionId` | `string` | Claude Code oturum tanımlayıcısı |
| `cwd` | `string` | Claude Code oturumunun çalışma dizini |
| `transcriptPath` | `string` | Oturumun JSONL transkripti dosya yolu |

### Olay türleri

| Olay | Ne zaman ateşlenir | `toolInput` içeriği |
|------|-------------------|-------------------|
| `PreToolUse` | Claude bir aracı çalıştırmadan önce | Aracın girdisi (örn. Bash için `{ command: "..." }`) |
| `PostToolUse` | Bir araç tamamlandıktan sonra | Aracın girdisi + `tool_result` (çıktı) |
| `Notification` | Claude bir bildirim gönderdiğinde | `{ message: "...", notification_type: "idle" \| "permission_prompt" \| ... }` - kancalar her zaman `allow()` döndürmelidir, bildirimleri engelleyemezler |
| `Stop` | Claude oturumu sona erdiğinde | Boş |

## Değerlendirme sırası

İlkeler şu sırada değerlendirilir:

1. Yerleşik ilkeler (tanım sırasında)
2. `customPoliciesPath` öğesinden açık özel ilkeler (`.add()` sırasında)
3. Proje `.failproofai/policies/` adresinden sözleşme ilkeleri (dosyalar alfabetik, içinde `.add()` sırası)
4. Kullanıcı `~/.failproofai/policies/` adresinden sözleşme ilkeleri (dosyalar alfabetik, içinde `.add()` sırası)

<Note>
İlk `deny` tüm sonraki ilkeleri kısa devre yapar. Tüm `instruct` iletileri biriktirilir ve birlikte iletilir.
</Note>

## Geçişli içe aktarmalar

Özel ilke dosyaları göreli yollar kullanarak yerel modülleri içe aktarabilir:

```js
// my-policies.js
import { isBlockedPath } from "./utils.js";
import { checkApproval } from "./approval-client.js";

customPolicies.add({
  name: "approval-gate",
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const approved = await checkApproval(ctx.toolInput?.command, ctx.session?.sessionId);
    return approved ? allow() : deny("Approval required for this command");
  },
});
```

Giriş dosyasından erişilebilir tüm göreli içe aktarmalar çözülür. Bu, `from "failproofai"` içe aktarmalarını gerçek dist yoluna yeniden yazarak ve ESM uyumluluğunu sağlamak için geçici `.mjs` dosyaları oluşturarak uygulanır.

## Olay türü filtreleme

Bir ilkenin ne zaman ateşleneceğini sınırlamak için `match.events` kullanın:

```js
customPolicies.add({
  name: "require-summary-on-stop",
  match: { events: ["Stop"] },
  fn: async (ctx) => {
    // Only fires when the session ends
    // ctx.session.transcriptPath contains the full session log
    return allow();
  },
});
```

Her olay türünde ateşlemek için `match` tamamen atlayın.

## Hata işleme ve başarısızlık modları

Özel ilkeler **açık başarısız**: hatalar hiçbir zaman yerleşik ilkeleri engellemez veya kanca işleyiciyi çökmez.

| Başarısızlık | Davranış |
|--------------|----------|
| `customPoliciesPath` ayarlanmamış | Açık özel ilkeler çalışmaz; sözleşme ilkeleri ve yerleşik ilkeler normal şekilde devam eder |
| Dosya bulunamadı | Uyarı `~/.failproofai/hook.log` dosyasına kaydedilir; yerleşik ilkeler devam eder |
| Söz dizimi/içe aktarma hatası (açık) | Hata `~/.failproofai/hook.log` dosyasına kaydedilir; açık özel ilkeler atlanır |
| Söz dizimi/içe aktarma hatası (sözleşme) | Hata kaydedilir; o dosya atlanır, diğer sözleşme dosyaları yine de yüklenir |
| `fn` çalışma zamanında hata fırlatır | Hata kaydedilir; o kanca `allow` olarak işlenir; diğer kancalar devam eder |
| `fn` 10 saniyeden uzun sürer | Zaman aşımı kaydedilir; `allow` olarak işlenir |
| Sözleşme dizini eksik | Sözleşme ilkeleri çalışmaz; hata yok |

<Tip>
Özel ilke hatalarını hata ayıklamak için günlük dosyasını izleyin:

```bash
tail -f ~/.failproofai/hook.log
```
</Tip>

## Tam örnek: birden fazla ilke

```js
// my-policies.js
import { customPolicies, allow, deny, instruct } from "failproofai";

// Prevent agent from writing to secrets/ directory
customPolicies.add({
  name: "block-secrets-dir",
  description: "Prevent agent from writing to secrets/ directory",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (!["Write", "Edit"].includes(ctx.toolName ?? "")) return allow();
    const path = ctx.toolInput?.file_path ?? "";
    if (path.includes("secrets/")) return deny("Writing to secrets/ is not permitted");
    return allow();
  },
});

// Keep the agent on track: verify tests before committing
customPolicies.add({
  name: "remind-test-before-commit",
  description: "Keep the agent on track: verify tests pass before committing",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    if (/git\s+commit/.test(cmd)) {
      return instruct("Verify all tests pass before committing. Run `bun test` if you haven't already.");
    }
    return allow();
  },
});

// Prevent unplanned dependency changes during freeze
customPolicies.add({
  name: "dependency-freeze",
  description: "Prevent unplanned dependency changes during freeze period",
  match: { events: ["PreToolUse"] },
  fn: async (ctx) => {
    if (ctx.toolName !== "Bash") return allow();
    const cmd = ctx.toolInput?.command ?? "";
    const isInstall = /^(npm install|yarn add|bun add|pnpm add)\s+\S/.test(cmd);
    if (isInstall && process.env.DEPENDENCY_FREEZE === "1") {
      return deny("Package installs are frozen. Unset DEPENDENCY_FREEZE to allow.");
    }
    return allow();
  },
});

export { customPolicies };
```

## Örnekler

`examples/` dizini çalışmaya hazır ilke dosyalarını içerir:

| Dosya | İçerik |
|-------|--------|
| `examples/policies-basic.js` | Yaygın aracı başarısızlık modlarını kapsayan beş başlangıç ilkesi |
| `examples/policies-advanced/index.js` | İleri düzey desenler: geçişli içe aktarmalar, eşzamansız çağrılar, çıktı temizleme ve oturum sonu kancaları |
| `examples/convention-policies/security-policies.mjs` | Sözleşmeye dayalı güvenlik ilkeleri (.env yazmaları engelle, git geçmişi yeniden yazmasını önle) |
| `examples/convention-policies/workflow-policies.mjs` | Sözleşmeye dayalı iş akışı ilkeleri (test hatırlatmaları, denetim dosyası yazmaları) |

### Açık dosya örneklerini kullanma

```bash
failproofai policies --install --custom ./examples/policies-basic.js
```

### Sözleşmeye dayalı örnekleri kullanma

```bash
# Proje düzeyine kopyalayın
mkdir -p .failproofai/policies
cp examples/convention-policies/*.mjs .failproofai/policies/

# Veya kullanıcı düzeyine kopyalayın
mkdir -p ~/.failproofai/policies
cp examples/convention-policies/*.mjs ~/.failproofai/policies/
```

Yükleme komutu gerekmez — dosyalar bir sonraki kanca olayında otomatik olarak alınır.
